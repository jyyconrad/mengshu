# Banto记忆工作体系思考

> **企业 agent 的记忆，不是为了保存更多信息，而是为了在正确的边界内，持续理解一个人如何工作、一个团队为何这样工作、一个组织允许怎样工作，以及当前工作推进到哪里。**

换句话说，它不是一个“超级备忘录”，也不是一个“企业搜索框”，而是一个 **工作连续性系统**。

---

# 一、对企业用户来说，记忆里最重要的不是“资料”，而是“上下文”

企业用户每天最痛苦的事情，往往不是缺少信息，而是反复重建上下文：

“这个项目为什么这样做？”
“上次谁决定的？”
“这个客户偏好什么？”
“这个流程现在卡在哪里？”
“这个人希望我怎么沟通？”
“这件事能不能做？要不要审批？”
“我上周已经解释过，为什么今天还要重说一遍？”

所以企业 agent 的记忆，最核心的价值不是“存储”，而是 **减少上下文重建成本**。

它应该记住的，是那些：

```text
重复解释成本高的东西
协作中容易丢失的东西
出错后代价很高的东西
人与组织之间需要保持连续性的东西
```

从企业用户的工作生活角度看，记忆最重要的内容可以分成六类。

---

## 1. 身份与角色：我是谁，我在什么位置上工作

企业用户不是一个抽象的“用户”，而是一个具体角色：

```text
我是销售负责人
我是产品经理
我是法务
我是财务审批人
我是某个项目的 owner
我是某个客户的接口人
我是某个团队的新人
```

同一个人在不同角色下，需要 agent 记住的东西不同。

比如一个产品经理问：

> “帮我整理一下这个需求。”

agent 需要知道这个人的角色、负责的产品线、当前版本节奏、团队沟通习惯。

一个财务人员问：

> “这个报销能不能过？”

agent 需要知道制度、审批权限、预算归属、历史异常情况。

所以第一类重要记忆是：

```text
用户当前承担什么角色
这个角色有哪些责任
他/她在哪些项目、团队、流程中拥有决策权或执行权
```

这类记忆不是为了“画像用户”，而是为了理解用户在企业中的工作位置。

---

## 2. 偏好与工作方式：我希望 agent 如何配合我

这是个人记忆里最有价值的一类。

例如：

```text
我喜欢先看结论，再看展开
我不喜欢太长的解释，除非是方案评审
我写周报希望偏业务语言，不要太技术化
我习惯上午处理深度工作，下午处理沟通
我发给老板的内容要更简洁
我给团队成员的反馈要更具体、更鼓励
我不希望 agent 自动替我发消息
```

这类记忆决定了 agent 是否像一个真正的工作助手。

它不是记住“用户隐私”，而是记住：

```text
这个人如何思考
如何表达
如何安排注意力
如何与他人协作
如何做决定
```

企业 agent 如果没有这类个人记忆，就会变成一个通用问答工具；有了这类记忆，它才开始像一个长期协作者。

---

## 3. 关系与协作：我和谁一起工作，我们之间有什么默契

企业工作不是个人单点任务，而是关系网络。

agent 需要理解：

```text
我经常和谁合作
谁是项目 owner
谁是审批人
谁需要被同步
谁只需要看结论
谁需要看细节
哪些人之间存在依赖关系
哪些团队有固定沟通节奏
```

例如：

> “帮我把这个方案发给项目组。”

如果 agent 只知道“项目组”三个字是不够的。它最好知道：

```text
项目组有哪些人
谁是决策人
谁是执行人
谁只需要抄送
以前类似内容通常发到哪个群
这个团队偏好飞书文档还是邮件
```

这不是八卦式的人际记忆，而是 **协作记忆**。

企业里的很多效率损失，来自“关系上下文”丢失。一个好的 agent 要能补上这部分。

---

## 4. 决策与承诺：我们之前为什么这样决定

这是工作记忆里非常关键的一类。

企业知识库常常记录“结果”，但很少完整记录“为什么”。

真正有价值的记忆是：

```text
某个方案为什么被选中
某个方案为什么被否决
某个客户为什么不能按标准流程处理
某个项目为什么延期
某个接口为什么暂时不重构
某个审批为什么破例通过
```

这些内容不一定都在正式文档里，但它们决定了后续工作能不能连续。

企业 agent 应该帮助组织保存这种“决策上下文”。

例如：

```text
2026-04-18 的评审会上，团队决定暂不重构支付模块，
原因是 Q2 重点是稳定性和合规整改，重构计划推迟到 Q3。
```

这比单纯记住“支付模块 Q3 重构”更有价值。

因为后续有人问：

> “为什么不现在重构？”

agent 能回答背后的约束，而不是只给一个孤立事实。

---

## 5. 规则与边界：什么能做，什么不能做

企业 agent 和个人 assistant 最大的区别，是它必须理解边界。

它要知道：

```text
哪些数据不能出企业内网
哪些动作必须审批
哪些内容不能自动发送
哪些系统只能读取不能写入
哪些角色不能查看某些资料
哪些流程必须走正式路径
哪些事情可以建议，但不能代替人决定
```

这类记忆不是普通记忆，而是 **治理记忆**。

它不是为了让 agent 更聪明，而是为了让 agent 更可靠。

在企业场景中，agent 的记忆必须始终回答一个问题：

```text
我现在知道这件事，但我是否有权使用它？
我能帮用户做这件事，但我是否有权执行它？
```

这就是企业记忆体系和普通个人记忆体系的根本差别。

---

## 6. 当前状态：事情推进到哪里了

企业用户非常关心“现在”。

```text
这个审批到谁那里了？
这个合同还差哪一步？
这个客户上次回复了什么？
这个会议有没有定时间？
这个 issue 是谁在处理？
这个任务是不是已经过期？
这个流程现在卡在哪个节点？
```

这类信息很重要，但它和长期记忆不同。

它更像 **工作状态记忆**，而不是永久记忆。

例如：

```text
明天下午 3 点有一个会议
某个审批当前在财务负责人处
某个工单当前状态是待测试
某封邮件还没有回复
```

这些东西高度动态，不应该被永久写进长期记忆。更合理的方式是：

```text
实时查询办公系统
短期缓存
必要时生成摘要
流程结束后沉淀为历史记录或经验
```

也就是说，agent 需要“知道现在”，但不应该把“现在”误当成永久事实。

---

# 二、从个人记忆与工作记忆两个维度来看

我建议把整个记忆体系抽象成两大类：

```text
个人记忆：agent 如何理解“我”
工作记忆：agent 如何理解“我所在的工作世界”
```

这两个维度非常重要，因为它们的所有权、生命周期、风险和价值都不同。

---

# 1. 个人记忆：围绕“我如何工作”

个人记忆的核心问题是：

> **这个 agent 如何更好地与我协作？**

它关注的是个体，而不是组织。

个人记忆应该记录：

```text
我的表达偏好
我的沟通风格
我的工作节奏
我的长期目标
我的注意力边界
我的常用模板
我的决策习惯
我的协作偏好
我对不同对象的沟通方式
```

例如：

```text
用户喜欢先看结论，再看理由。
用户给老板写汇报时偏好一页纸结构。
用户不喜欢 agent 在没有确认时自动发送消息。
用户每周五上午整理周报。
用户在客户沟通中偏好稳健、克制、留有余地的措辞。
```

这些记忆的价值在于：它们让 agent 不只是“回答问题”，而是逐渐适配这个人的工作方式。

---

## 个人记忆的设计原则

个人记忆应该遵循几个原则。

### 第一，个人记忆应该由用户拥有

哪怕这是企业桌面应用，个人记忆也不应该完全变成企业资产。

例如：

```text
我的写作偏好
我的沟通习惯
我的工作节奏
我的提醒方式
我的草稿风格
```

这些属于个人协作上下文。

企业可以定义安全策略，但不应该默认窥探用户的所有个人记忆。

否则这个系统会从“助手”变成“监控工具”。

---

### 第二，个人记忆应该可见、可改、可删除

用户应该能看到：

```text
agent 记住了我什么
什么时候记住的
为什么记住
是否还准确
能不能删除
能不能改写
能不能暂时禁用
```

一个好的设计不是偷偷记住用户，而是让用户愿意托付上下文。

---

### 第三，个人记忆不能凌驾于企业规则

比如用户偏好是：

```text
我希望你直接帮我发邮件，不要每次确认。
```

但企业策略规定：

```text
所有对外邮件发送前必须人工确认。
```

那么企业规则优先。

所以个人记忆是一个 **个性化层**，不是一个 **权限豁免层**。

---

# 2. 工作记忆：围绕“工作世界如何运转”

工作记忆的核心问题是：

> **agent 如何理解这个组织、这个团队、这个项目和当前流程？**

它不是围绕个人，而是围绕工作对象。

工作记忆应该记录：

```text
企业制度
组织结构
业务术语
项目背景
团队约定
关键决策
任务进展
流程状态
系统事实
历史经验
```

它的价值是让 agent 不再像一个外部顾问，而是像一个真正进入工作现场的协作者。

---

## 工作记忆的设计原则

### 第一，工作记忆必须有来源

工作记忆不能只存一句：

```text
张三负责 Alpha 项目。
```

它应该知道：

```text
这句话来自哪里？
是项目管理系统？
是会议纪要？
是用户口头说的？
是邮件里提到的？
更新时间是什么？
是否仍然有效？
```

企业里，来源比内容本身更重要。

因为同一句话的可信度完全不同。

```text
“张三负责 Alpha 项目”
来自 HR 系统：高可信
来自项目管理系统：高可信
来自会议纪要：中高可信
来自聊天记录：中可信
来自模型推断：低可信
```

---

### 第二，工作记忆必须有权限边界

同一个企业里，不同部门、不同项目、不同角色能看到的东西不同。

所以工作记忆必须带上：

```text
谁能看
谁能用
能不能被总结
能不能被引用
能不能跨项目使用
能不能用于生成外发内容
```

没有权限边界的企业记忆，越强大越危险。

---

### 第三，工作记忆必须区分“规则、知识、状态”

这点非常关键。

企业里的信息至少分三种：

```text
规则：应该怎么做，什么不能做
知识：为什么这样做，背景是什么
状态：现在推进到哪里了
```

它们不能混在一起。

例如：

```text
差旅报销制度       → 规则
某次客户特殊处理原因 → 知识
当前报销单审批节点   → 状态
```

规则相对稳定，应该来自权威来源。
知识需要沉淀和解释。
状态变化很快，应该实时查询。

---

# 三、工作记忆从上往下：企业、项目/团队、办公流程状态，是否合适？

我的判断是：**非常合适，但需要换一种表述。**

这三个层级本质上不是简单的信息分类，而是三个不同的工作世界层次：

```text
企业层：我们在什么规则和语言中工作
项目/团队层：我们为什么一起做这件事
流程状态层：这件事现在推进到哪里
```

我建议把它们命名为：

```text
企业记忆：制度与公共语义
项目/团队记忆：协作上下文与共同历史
办公流程状态：当前行动现场
```

这三个层级从上到下是非常自然的。

---

# 1. 企业层记忆：组织的“共同语言”和“边界”

企业层记忆回答的是：

> **在这个组织里，什么是被允许的？什么是标准做法？大家如何理解同一件事？**

它包括：

```text
企业制度
安全规范
审批规则
组织结构
岗位职责
业务术语
产品线说明
财务规则
法务规则
HR 规则
IT 服务规则
数据权限规则
```

企业层记忆是整个工作记忆的最高层。

它有点像组织的“宪法”和“词典”。

它告诉 agent：

```text
哪些事情可以做
哪些事情不能做
哪些事情必须走流程
哪些词在企业内部有特殊含义
哪些系统是权威来源
哪些信息不能跨边界传播
```

例如：

> 用户说：“帮我把这个客户数据整理一下发给外部合作方。”

agent 不能只看用户意图，还要查看企业层记忆：

```text
客户数据是否属于敏感信息？
外部合作方是否有数据授权？
是否需要脱敏？
是否需要法务审批？
是否允许通过当前渠道发送？
```

所以企业层记忆是安全边界，也是组织语境。

---

## 企业层记忆的特点

```text
稳定性高
权威性高
变更频率低
必须有来源
必须有版本
必须可审计
不能由普通对话随意改写
```

它不应该由 agent 自己“总结几次对话”后生成，而应该主要来自权威系统：

```text
制度文档
企业知识库
权限系统
组织架构系统
法务/财务/HR/IT 官方流程
```

---

# 2. 项目/团队层记忆：协作的“共同历史”

项目/团队层记忆回答的是：

> **我们这群人，围绕这件事，已经形成了什么共识？**

这是企业 agent 最有价值的一层。

因为企业制度通常会写得比较完整，但项目真实上下文常常散落在会议、聊天、邮件、文档、工单里。

项目/团队层记忆应该记录：

```text
项目目标
项目边界
阶段计划
关键负责人
团队分工
历史决策
重要争议
已知风险
客户偏好
技术约束
业务约束
会议结论
未完成事项
反复出现的问题
团队约定俗成的表达方式
```

例如：

```text
Alpha 项目当前目标是 Q3 前完成私有化部署能力。
团队已决定优先支持金融客户，不优先支持教育行业。
销售侧要求演示环境必须在每周三前稳定。
技术侧认为多租户权限是最大风险点。
客户 A 偏好邮件确认，客户 B 偏好会议同步。
```

这类记忆让 agent 理解“项目为什么是现在这个样子”。

---

## 项目/团队层记忆的特点

```text
稳定性中等
变化频率中等
上下文价值极高
通常跨越多个系统
需要持续整理
需要处理冲突
需要保留决策理由
```

项目/团队层记忆不是简单地把会议纪要堆进去，而是要沉淀出：

```text
共识
决策
承诺
分工
风险
约束
历史原因
```

它的核心价值是避免团队重复讨论同样的问题。

---

# 3. 办公流程状态：工作的“当前现场”

办公流程状态回答的是：

> **这件事现在卡在哪里？下一步该谁做？**

它包括：

```text
邮件状态
日历安排
审批节点
合同流转
报销进度
工单状态
任务状态
CRM 跟进状态
会议待办
未回复消息
待确认事项
```

这一层不是传统意义上的长期记忆，而是 **动态工作状态**。

它非常重要，但它不应该被永久化。

例如：

```text
报销单现在在财务审批中
合同现在等待法务确认
客户昨天回复了邮件
明天下午有一个评审会
某个 Jira issue 当前 assign 给李雷
```

这些信息今天正确，明天可能就过期。

所以它的设计原则是：

```text
实时查询优先
短期缓存辅助
流程结束后再沉淀
不要把瞬时状态当成长期事实
```

---

## 办公流程状态的特点

```text
变化频率高
时效性强
最接近行动
最容易过期
必须依赖源系统
不适合直接长期保存
```

这层记忆最适合和办公系统直接连接：

```text
邮件
日历
IM
OA
CRM
ERP
工单系统
项目管理系统
审批系统
文档系统
```

agent 在这里最应该做的不是“记住一切”，而是：

```text
帮用户感知当前状态
发现阻塞
提醒下一步
生成草稿
推动流程
但不擅自替用户做不可逆动作
```

---

# 四、这三个层级的关系：不是平铺，而是“从规则到协作再到行动”

我建议你在方案里用这个结构表达：

```text
企业层记忆：决定边界
项目/团队记忆：决定语境
办公流程状态：决定下一步行动
```

或者：

```text
企业层回答：能不能做？
项目/团队层回答：为什么这样做？
流程状态层回答：现在该做什么？
```

这是非常有力量的表达。

例如用户说：

> “帮我推进一下 Alpha 项目的合同。”

agent 的思考应该是：

```text
企业层：
合同推进是否有法务规则？
是否涉及金额阈值？
是否必须走审批？
是否可以对外发送？

项目/团队层：
Alpha 项目当前处于什么阶段？
客户是谁？
之前谈判卡点是什么？
谁是内部负责人？
哪些条款曾经有争议？

流程状态层：
合同现在在哪个系统？
当前审批到谁？
最近一封邮件是什么？
下一步是催法务、催客户，还是补材料？
```

这就是完整的工作记忆。

---

# 五、个人记忆与工作记忆不是上下级，而是两条并行轴

这里要特别注意。

个人记忆不应该被放在企业层、项目层、流程层下面。它应该是一条独立的轴。

因为个人记忆回答的是：

```text
agent 如何服务这个人？
```

工作记忆回答的是：

```text
agent 如何理解这份工作？
```

可以这样画：

```text
                    企业记忆
              制度 / 权限 / 规则 / 术语
                       │
                       ▼
                  项目/团队记忆
          目标 / 决策 / 分工 / 风险 / 共识
                       │
                       ▼
                  办公流程状态
          邮件 / 日历 / 审批 / 工单 / 待办
                       │
                       ▼
                    当前任务
                       ▲
                       │
              个人记忆作为协作镜片
      偏好 / 风格 / 节奏 / 习惯 / 目标 / 边界
```

也可以用一句话表达：

> **工作记忆告诉 agent “这个世界是什么样的”；个人记忆告诉 agent “应该怎样陪这个人穿过这个世界”。**

这句话很适合放在方案里。

---

# 六、从工作生活角度看，个人记忆和工作记忆的边界非常重要

企业用户不是只想提高效率。他们还会关心：

```text
这个 agent 会不会监控我？
它记住的是我授权的东西吗？
我离开项目后，这些记忆怎么办？
我的个人习惯会不会被企业拿去评价我？
企业资料会不会被写进我的个人记忆？
我能不能把个人偏好带到新的工作空间？
```

所以设计哲学上必须强调边界。

---

## 个人记忆强调“尊重”

个人记忆应该帮助用户，而不是评判用户。

它可以记住：

```text
用户喜欢怎样的输出格式
用户希望怎样被提醒
用户常用哪些工作模板
用户对沟通语气有什么偏好
用户希望保留哪些私人工作习惯
```

但它不应该偷偷推断：

```text
用户是不是效率低
用户是不是不适合管理
用户是否情绪不稳定
用户是否可能离职
```

个人记忆的底层伦理应该是：

> **帮助用户保持连续性，而不是把用户变成被分析对象。**

---

## 工作记忆强调“责任”

工作记忆不是用户私人所有，它通常涉及团队、客户、企业资产和合规要求。

所以它必须强调：

```text
来源
权限
版本
审计
有效期
责任归属
```

工作记忆的底层伦理应该是：

> **帮助组织保持连续性，而不是制造无法追责的黑箱。**

---

# 七、建议的顶层叙事框架

如果你要写成正式方案，我建议用下面这个叙事结构。

---

## 1. 先定义问题

企业工作中的主要损耗不是信息缺失，而是上下文断裂。

上下文断裂发生在：

```text
人和人之间
会议和执行之间
项目和项目之间
制度和实际操作之间
今天和昨天之间
个人工作习惯和组织流程之间
```

agent 的记忆体系，就是为了解决这些断裂。

---

## 2. 再定义目标

企业 agent 的记忆体系要实现三种连续性：

```text
个人连续性：
agent 记得我如何工作，不需要我每天重新训练它。

协作连续性：
agent 记得团队做过什么决定，不需要反复追问历史。

组织连续性：
agent 记得企业规则和权限边界，不会为了效率破坏治理。
```

这三种连续性非常关键。

可以作为方案的核心章节。

---

## 3. 然后定义两大记忆

```text
个人记忆：
围绕用户本人，服务于个性化协作。

工作记忆：
围绕组织工作，服务于业务执行、团队协作和企业治理。
```

---

## 4. 再定义工作记忆三层

```text
企业层：
组织的规则、语言、权限和制度。

项目/团队层：
具体协作单元的目标、历史、决策和共识。

办公流程状态层：
当前任务、流程、审批、邮件、日程、工单的实时状态。
```

---

## 5. 最后定义边界

```text
个人记忆不替代企业规则。
工作记忆不侵犯个人边界。
企业知识不等于个人偏好。
流程状态不等于长期事实。
agent 可以建议和草拟，但高风险动作必须确认。
```

---

# 八、可以直接放进方案里的设计哲学表述

你可以这样写：

> 企业 agent 的记忆体系，不应被理解为一个信息存储系统，而应被理解为一个上下文连续性系统。
>
> 对个人而言，它记住用户的工作方式、表达偏好、协作习惯和注意力边界，使 agent 能够长期适配一个具体的人。
>
> 对组织而言，它记住企业规则、项目历史、团队共识和流程状态，使 agent 能够在正确的权限、语境和时点上协助工作。
>
> 因此，记忆体系的核心不是“记得越多越好”，而是“在合适的范围内，记住那些反复解释成本高、协作价值高、出错代价高、并且有明确来源和边界的上下文”。

# 十、我对“企业、项目/团队、办公流程状态”这个层级的最终判断

这个结构是合适的，而且很有表达力。

但建议你在方案里不要把它说成简单的“三级知识库”，而是说成：

```text
企业层：边界系统
项目/团队层：语境系统
流程状态层：行动系统
```

即：

```text
企业层决定 agent 能不能做；
项目/团队层决定 agent 应该怎么理解；
流程状态层决定 agent 现在该怎么推进。
```

再加上个人记忆作为独立轴：

```text
个人记忆决定 agent 应该怎样与这个人协作。
```

最终可以浓缩成一句顶层设计哲学：

> **个人记忆让 agent 记得“我是谁、我如何工作”；工作记忆让 agent 记得“组织如何运转、团队为何协作、事情推进到哪里”。二者共同构成企业 agent 的上下文连续性。**
