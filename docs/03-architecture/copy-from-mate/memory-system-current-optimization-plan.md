# banto 当前版本记忆系统优化方案

> 日期：2026-05-31
> 状态：当前版本优化方案
> 范围：面向 banto/iFlyMate 产品记忆能力的近期优化，不是代码开发实施计划
> 上位设计：[记忆系统高层架构设计](./memory-system-upgrade-design.md)

## 1. 一句话结论

当前版本的优化重点不是实现完整工作图谱，也不是为代码开发场景补上下文，而是把 banto 的记忆能力收敛为一个清晰可用的产品闭环：

```text
用户发起短时任务对话或 Project 对话
  → banto 按规则召回全局记忆和 Project 记忆
  → 对话过程中识别学习信号
  → 会话结束生成候选记忆
  → 用户确认或后台整理后沉淀
  → 下一次对话更准确地继承上下文
```

本方案聚焦四类优化：

1. 规则与边界层产品化，不再写代码开发专属规则。
2. 明确全局记忆、Project 记忆、会话候选记忆的职责。
3. 补清楚不同记忆的召回逻辑。
4. 补清楚 banto 如何学习、更新、遗忘和防止错误回写。

## 2. 产品范围

### 2.1 覆盖范围

当前版本只围绕 banto 的正式主线设计：

- 短时任务对话
- Project
- 岗位角色
- 专家智能体
- 技能
- 连接器
- 知识库 / 外部文档指针

### 2.2 不单独设计自动化记忆

定时触发或自动化执行不需要独立记忆层。它本质上仍是一种对话执行：

- 有用户或 Project 上下文。
- 可以使用全局记忆。
- 可以使用 Project 记忆。
- 执行结束后按同一套候选记忆逻辑判断是否沉淀。

因此，当前版本删除“自动化记忆”作为独立容器的设计，只保留“定时触发执行复用对话记忆机制”这一产品原则。

## 3. 当前问题

### 3.1 规则与边界层语境不准

此前文档把代码开发中的仓库说明、开发约束写进了显性规则层。这不符合 banto 产品本身。

banto 的规则与边界应来自：

- 岗位角色配置。
- Project 目标和范围。
- 用户明确给出的任务边界。
- 专家、技能、连接器配置。
- 知识库和外部文档的权威指针。
- 用户确认过的外部动作边界，例如“只草拟，不发送”。

### 3.2 记忆作用域不够清晰

当前需要明确三种产品作用域：

| 作用域 | 作用 |
| --- | --- |
| 全局记忆 | 让 banto 长期理解这个用户如何工作 |
| Project 记忆 | 让 banto 持续理解这个 Project 为什么这样推进 |
| 会话候选记忆 | 让 banto 先暂存本次对话学到的内容，避免直接污染长期记忆 |

### 3.3 召回逻辑不足

当前设计只说“分层召回”，但没有讲清：

- 短时任务对话召回什么。
- Project 内对话召回什么。
- 专家执行召回什么。
- 技能执行召回什么。
- 连接器相关任务召回什么。
- 定时触发执行如何复用记忆。

### 3.4 学习逻辑不足

当前设计只说 chat-end 提取和周期整理，但没有讲清：

- 哪些用户表达是强学习信号。
- 哪些内容只属于当前会话，不该长期保存。
- 新记忆写到全局还是 Project。
- 同主题旧记忆如何更新。
- 用户删除后如何避免再次写回。

## 4. 目标方案

## 4.1 规则与边界层优化

### 4.1.1 内容定义

规则与边界层用于保存 banto 执行任务前必须尊重的高可信上下文。

| 类别 | 内容 | 示例 |
| --- | --- | --- |
| 岗位语境 | 用户当前角色和默认工作方式 | 用户以销售负责人角色使用 banto |
| Project 边界 | Project 目标、范围、阶段和优先级 | 本 Project 只做客户演示材料，不做内部培训文档 |
| 动作边界 | 是否允许外发、提交、修改外部系统 | 飞书消息只生成草稿，发送前必须确认 |
| 能力边界 | 专家、技能、连接器的适用范围 | 使用合同专家处理条款，但不替代法务审核 |
| 权威来源 | 知识库、飞书文档、外部系统指针 | 需求以飞书文档 X 为准 |

### 4.1.2 使用原则

- 规则与边界层优先于自动学习记忆。
- 自动学习不能静默覆盖规则与边界。
- 规则与边界不应包含代码仓库、开发命令、工程规范等非 banto 产品内容。
- reference 类型长期记忆可以指向权威来源，但权威事实本身仍由知识库或外部系统维护。

## 4.2 记忆作用域优化

### 4.2.1 全局记忆

全局记忆跟随用户账号，跨短时任务和 Project 生效。

适合保存：

- 用户长期表达偏好。
- 用户通用协作偏好。
- 用户对 banto 行为的通用纠正。
- 常用外部系统或资料入口。
- 与岗位角色相关但角色配置无法表达的细节偏好。

不适合保存：

- 某个 Project 的客户约束。
- 某次对话的临时任务状态。
- 外部系统当前动态数据。

### 4.2.2 Project 记忆

Project 记忆绑定单个 Project。

适合保存：

- Project 目标和范围。
- 关键决策和 Why。
- 客户、团队、交付物的偏好和约束。
- 当前阶段优先级。
- 重要产物或外部文档指针。

不适合保存：

- 用户跨场景通用偏好。
- 只在本轮对话中临时成立的推理。
- 可从外部系统实时查询的状态。

### 4.2.3 会话候选记忆

会话候选记忆是缓冲层，不是长期记忆。

适合暂存：

- 本轮对话中模型判断值得保存的偏好、决策、约束。
- 需要用户确认才能长期生效的信息。
- 置信度足够但作用域不确定的信息。

处理结果：

- 用户确认保存到全局。
- 用户确认保存到 Project。
- 后台整理后自动建议保存。
- 用户忽略或删除，进入防回写记录。

## 5. 召回逻辑

## 5.1 召回总流程

```text
读取当前任务上下文
  → 判断对话类型：短时任务 / Project / 专家 / 技能 / 连接器
  → 加载规则与边界层
  → 召回必读记忆
  → 计算任务相关 Top-K
  → 生成本轮 memory_context
  → 记录召回日志
```

### 5.1.1 输入信号

| 输入信号 | 用途 |
| --- | --- |
| 当前用户 | 确定全局记忆范围 |
| 当前岗位角色 | 补充默认工作语境 |
| 当前 Project | 确定 Project 记忆范围 |
| 用户本轮输入 | 提取主题、对象、动作和约束 |
| 当前专家 | 召回专家相关反馈 |
| 当前技能 | 召回技能使用偏好和约束 |
| 当前连接器 | 召回连接器读写边界和 reference |
| 知识库范围 | 召回知识库指针，而非知识库全文 |

## 5.2 必读层

必读层每次对话都注入，但必须小而稳定。

进入条件：

- 用户手动保存或明确确认。
- 高置信。
- 当前对话一定需要。
- 未归档、未失效。

按场景区分：

| 场景 | 必读内容 |
| --- | --- |
| 短时任务对话 | 用户长期表达偏好、动作确认边界、岗位默认语境 |
| Project 对话 | Project 目标、范围、关键约束、用户长期偏好 |
| 专家执行 | 专家能力边界、专家相关用户反馈、Project 关键约束 |
| 技能执行 | 技能适用范围、用户对该技能结果的偏好 |
| 连接器任务 | 连接器权限边界、外部动作确认要求 |
| 定时触发执行 | 按 Project 对话处理，注入全局记忆和 Project 记忆 |

## 5.3 任务相关层

任务相关层通过 Top-K 召回，不全量注入。

### 5.3.1 评分公式

```text
score =
  scope_score * 0.25
  + relevance_score * 0.30
  + type_score * 0.15
  + confidence_score * 0.15
  + recency_score * 0.10
  + user_signal_score * 0.05
```

权重后续可调，但当前版本必须保留这六个维度。

### 5.3.2 维度说明

| 维度 | 说明 |
| --- | --- |
| `scope_score` | 当前 Project 记忆最高；全局记忆次之；其他 Project 默认不参与 |
| `relevance_score` | 与用户输入、专家、技能、连接器、知识库范围的匹配程度 |
| `type_score` | 根据任务类型调整 user / feedback / project / reference 权重 |
| `confidence_score` | 用户确认和多次验证过的记忆优先 |
| `recency_score` | 最近更新或最近使用的记忆优先 |
| `user_signal_score` | 用户置顶、编辑、保存、确认过的记忆优先 |

### 5.3.3 类型权重示例

| 场景 | user | feedback | project | reference |
| --- | --- | --- | --- | --- |
| 普通短时任务 | 高 | 高 | 低 | 中 |
| Project 继续推进 | 中 | 高 | 高 | 高 |
| 专家任务 | 中 | 高 | 高 | 中 |
| 技能任务 | 中 | 高 | 中 | 中 |
| 连接器任务 | 低 | 高 | 中 | 高 |

## 5.4 按需检索层

按需检索层不默认进入上下文，适用于：

- 用户问“之前我们怎么定的？”
- 用户问“这个 Project 上次推进到哪里？”
- 用户需要查看某个 reference 的来源。
- banto 判断当前任务需要更多历史依据。

按需检索返回应包含：

- 记忆标题。
- 记忆摘要。
- 来源。
- 置信度/状态。
- 是否需要用户确认。

## 5.5 召回后的输出约束

memory_context 应明确告诉模型：

- 哪些是规则与边界。
- 哪些是全局记忆。
- 哪些是 Project 记忆。
- 哪些只是候选或低置信记忆。
- 哪些 reference 需要实时查询外部系统或知识库。

模型不得把低置信记忆说成确定事实。

## 6. 学习逻辑

## 6.1 学习触发

| 信号 | 示例 | 处理 |
| --- | --- | --- |
| 显式记忆请求 | “以后都这样”“记住这个” | 高置信候选 |
| 明确纠正 | “不是这样”“以后不要直接发” | feedback 候选，并查找旧记忆 |
| 明确决策 | “这个 Project 先不做英文版” | Project 候选，必须记录 Why |
| 明确引用 | “需求以这份飞书文档为准” | reference 候选 |
| 重复模式 | 多次选择同一表达风格 | 提升旧记忆置信度或生成候选 |
| 删除/忽略 | 用户删除候选或长期记忆 | 加入防回写，降低同主题学习优先级 |

## 6.2 写入判断

```text
如果是用户跨场景偏好
  → 全局 user / feedback

如果是 banto 行为纠正
  → 全局或 Project feedback

如果是当前 Project 的目标、决策、客户要求、阶段结论
  → Project project

如果是外部系统、知识库、文档或素材入口
  → reference

如果只是当前任务步骤、临时状态、外部系统动态数据
  → 不写长期记忆
```

## 6.3 作用域判断

| 用户表达 | 作用域 |
| --- | --- |
| “以后都...”且没有 Project 限定 | 全局 |
| “这个 Project / 这次项目 / 这个客户...” | Project |
| “这次先...” | 会话，不进长期记忆，除非后续确认 |
| “这个专家以后...” | 全局 feedback + relatedExpertIds |
| “这个技能在本 Project 中...” | Project feedback + relatedSkillIds |
| “这个连接器只读...” | 全局或 Project reference / feedback，取决于是否限定 Project |

## 6.4 更新旧记忆

学习不是只新增，还要更新。

| 情况 | 处理 |
| --- | --- |
| 新记忆与旧记忆同主题且冲突 | 新记忆 supersede 旧记忆，旧记忆归档 |
| 新记忆是旧记忆的补充 | 更新旧记忆正文，补充适用条件 |
| 新记忆只在 Project 内成立 | 不更新全局，写入 Project |
| 用户删除旧记忆 | 记录删除意图，阻止自动回写 |
| 用户编辑旧记忆 | 以用户编辑内容为准，提升可信度 |

## 6.5 置信度逻辑

| 来源 / 事件 | 初始或变化 |
| --- | --- |
| 用户手动保存 | 最高 |
| 用户明确纠正 | 高 |
| chat-end 自动提取 | 中 |
| 后台整理生成 | 中 |
| 多次召回且无负反馈 | 小幅提升 |
| 用户删除或否定 | 明显降低或归档 |
| reference 校验失败 | 标记 stale |
| 长期未使用 | 进入修剪候选 |

置信度只用于系统排序和整理，不应被包装成用户评价。

## 6.6 防回写逻辑

用户删除或忽略记忆后，系统应保存一条防回写记录：

```text
被删除主题
  → 删除时间
  → 删除作用域
  → 原记忆摘要
  → 禁止自动生成的时间窗口
```

后续 extractor 命中相似主题时：

- 如果用户没有重新明确表达，直接丢弃。
- 如果用户重新明确表达，生成候选，但标记“曾被删除，需要确认”。

## 7. 当前版本功能方案

### 7.1 记忆展示

记忆管理界面应至少区分：

- 全局记忆。
- Project 记忆。
- 本次会话候选记忆。
- 已归档 / 已失效记忆。

每条记忆展示：

- 标题。
- 类型。
- 作用域。
- 来源。
- 状态。
- 最近更新时间。
- 为什么会被使用的简短说明。

### 7.2 候选记忆处理

会话结束后，候选记忆可提供三个动作：

- 保存到全局。
- 保存到当前 Project。
- 忽略。

如果系统不确定作用域，应默认让用户选择，而不是自动保存。

### 7.3 记忆召回说明

在调试或内部日志中，应能看到：

- 本轮召回了哪些规则与边界。
- 本轮召回了哪些全局记忆。
- 本轮召回了哪些 Project 记忆。
- 每条记忆被召回的主要原因。
- 哪些候选被裁剪。

这不是用户主界面必须展示的内容，但必须成为评估召回质量的依据。

### 7.4 后台整理

当前版本后台整理只做四件事：

1. 合并重复记忆。
2. 处理冲突和 supersedes。
3. 把相对时间改成绝对时间。
4. 标记长期未使用或低置信记忆为归档候选。

不做：

- 自动生成 Skill。
- 主动给用户推流程优化建议。
- 建立复杂关系图谱。
- 独立自动化记忆。

## 8. 验收标准

### 8.1 产品验收

- 用户发起短时任务时，banto 能继承全局偏好，但不会带入无关 Project 历史。
- 用户进入 Project 继续对话时，banto 能继承 Project 目标、决策和关键约束。
- 用户使用专家时，banto 能继承专家相关反馈。
- 用户使用连接器时，banto 能遵守外部动作边界。
- 定时触发执行按 Project 对话处理，不产生独立记忆容器。

### 8.2 学习验收

- 用户明确说“以后”或“不要”时，系统能生成候选记忆。
- 用户纠正旧偏好时，系统能更新旧记忆，而不是新增冲突记忆。
- 用户删除候选后，同主题内容不会自动反复出现。
- Project 决策类记忆必须包含 Why 或来源。
- reference 类记忆只保存指针和边界，不复制外部文档全文。

### 8.3 召回验收

- 必读层内容可解释且数量受控。
- 任务相关层能根据 Project、专家、技能、连接器变化而变化。
- 其他 Project 记忆默认不进入当前 Project。
- stale 或低置信记忆不会被当成确定事实。
- 内部日志可解释每条记忆为什么被召回。

## 9. 后续暂缓内容

以下能力不进入当前版本：

- 完整工作图谱。
- 多层 L0/L1 摘要体系。
- 向量检索。
- Skill 自动生成。
- 主动工作预测。
- 跨用户共享记忆。
- 自动化独立记忆。

这些可以作为后续增强，但前提是当前版本的召回和学习闭环已经稳定。

## 10. 推荐推进顺序

```text
先修正文档口径
  → 明确作用域和规则与边界层
  → 落地召回逻辑
  → 落地学习逻辑
  → 增加候选记忆处理
  → 增加后台整理
```

当前最重要的不是“记忆系统看起来更高级”，而是让用户在 banto 的短时任务对话和 Project 中真实感到：系统记得该记的，忘掉该忘的，并且知道什么时候需要重新确认。
