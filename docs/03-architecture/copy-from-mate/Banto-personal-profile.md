# **一、产品定位**

|  |
| --- |
| **Banto 的定位** Banto 是面向企业内部知识工作者的个人 Claw 类型知识工作台，连接企业知识、个人任务、会议、沟通与工作流程，帮助用户完成检索、理解、组织、生成和执行。 |

| **定位维度** | **说明** |
| --- | --- |
| 服务对象 | 企业内部知识工作者，包括管理者、产品、研发、销售、客户成功、运营、HR、财务、法务等角色。 |
| 核心场景 | 企业搜索、文档总结、会议工作台、沟通辅助、任务跟进、个人知识整理、跨系统执行。 |
| 关键差异 | Banto 不只是连接企业信息，而是基于用户的工作上下文，判断哪些信息重要、应如何处理、何时需要确认。 |
| 产品目标 | 将通用 AI 助手升级为懂用户角色、知识、协作关系和授权边界的个人工作台。 |

# **二、核心概念：个人工作上下文层**

|  |
| --- |
| **定义** 个人工作上下文层，是 Banto 为每位知识工作者建立的私有工作理解层。它记录用户在组织角色、知识领域、任务习惯、协作关系、沟通风格和 AI 授权边界上的稳定倾向，并用于指导 Banto 的检索、总结、写作、计划和执行。 |

**核心目标：不判断用户是什么人，而是明确在具体工作场景下，Banto 应该如何帮助用户完成工作。**

| **不做** | **要做** |
| --- | --- |
| 不做员工性格分析 | 做个人工作偏好与操作方式适配 |
| 不做绩效画像 | 做用户可见、可编辑的工作上下文 |
| 不做管理者监控 | 做个人工作台的私有辅助能力 |
| 不做心理诊断 | 做场景化、证据化、可撤销的 Agent 行动假设 |

# **三、理论依据：从组织社会心理学到工作上下文**

Banto 的个人工作上下文层不是泛化的用户标签，而是将组织社会心理学中的角色、身份、动机、协作、呈现和信任转化为可执行的产品变量。

| **理论视角** | **核心问题** | **产品转化** |
| --- | --- | --- |
| 角色理论 | 用户在组织中被期待承担什么责任、遵循什么规则、拥有何种决策边界？ | 形成角色画像：职能角色、项目角色、责任范围、决策权限、当前重点。 |
| 社会身份理论 | 用户会以团队、专业、项目 owner、管理者、新人或专家等身份行动。 | 影响信息粒度、输出视角、专业表达和默认工作方式。 |
| 自我决定理论 | 用户需要自主感、胜任感和关系感。 | 保证用户掌控 AI 授权；让 AI 帮用户提升工作能力；在协作场景中考虑关系与责任。 |
| 分布式认知与跨人记忆 | 企业知识分布在文档、会议、IM、邮件、系统和同事之间。 | 形成知识画像：关注领域、常用来源、证据要求、术语偏好、输出格式。 |
| 目标调节与任务推进 | 知识工作需要将目标拆解为任务、行动项、优先级和完成标准。 | 形成任务画像：拆解粒度、推进节奏、风险偏好、提醒方式、完成标准。 |
| 社会交换与相互依赖 | 企业工作由承诺、依赖、协作关系和责任网络构成。 | 形成协作画像：stakeholder、依赖关系、承诺记录、协作节奏、敏感关系。 |
| 印象管理与沟通适配 | 企业沟通既传递信息，也呈现专业形象并维护关系。 | 形成沟通画像：不同对象、层级、场景下的语气、结构、正式度和承诺边界。 |
| 信任、心理安全与代理边界 | AI Agent 需要明确哪些可自动、哪些需确认、哪些禁止执行。 | 形成授权画像：检索、草拟、创建、发送、修改、执行等动作的确认边界。 |

# **四、六层画像推导：六层画像是怎么来的**

|  |
| --- |
| **推导逻辑** 企业知识工作可以抽象为一个闭环：明确组织角色 → 调用相关知识 → 推进具体任务 → 协调协作关系 → 完成沟通呈现 → 决定 AI 代理边界。Banto 的六层画像即由这一工作闭环推导而来。 |

| **工作闭环问题** | **对应理论来源** | **沉淀为画像层** | **进入 Agent 的方式** |
| --- | --- | --- | --- |
| 我在组织中是谁、负责什么？ | 角色理论、社会身份理论 | 角色画像 | 决定检索视角、摘要重点、会议责任、汇报口径。 |
| 我需要处理哪些知识？ | 分布式认知、跨人记忆 | 知识画像 | 决定信息来源、证据要求、摘要深度、输出格式。 |
| 我如何推进工作？ | 目标调节、任务推进、风险偏好 | 任务画像 | 决定任务拆解、优先级、提醒频率、完成标准。 |
| 我和谁一起工作？ | 社会交换、相互依赖、责任网络 | 协作画像 | 决定 stakeholder 识别、承诺跟进、跨团队协作策略。 |
| 我如何表达和呈现？ | 印象管理、沟通适配 | 沟通画像 | 决定邮件、IM、纪要、周报、汇报材料的语气与结构。 |
| AI 可以替我做到哪一步？ | 信任、心理安全、代理边界 | 授权画像 | 决定只建议、可草拟、可执行、需确认或禁止执行。 |

因此，六层画像不是对用户进行分类，而是覆盖知识工作从理解、处理到执行的完整操作链路。

# **五、六层工作画像框架**

|  |
| --- |
| **框架总览** Banto 的用户画像能力收束为六层工作上下文：角色画像、知识画像、任务画像、协作画像、沟通画像、授权画像。 |

| **层级** | **回答的问题** | **主要用途** |
| --- | --- | --- |
| 1. 角色画像 | 用户在组织里是谁，负责什么？ | 决定搜索视角、摘要重点、会议责任和汇报口径。 |
| 2. 知识画像 | 用户需要处理哪些知识？ | 决定信息来源、证据要求、摘要深度和输出格式。 |
| 3. 任务画像 | 用户如何推进工作？ | 决定任务拆解、优先级、提醒方式和完成标准。 |
| 4. 协作画像 | 用户和谁一起工作？ | 决定 stakeholder 识别、承诺跟进和跨团队沟通策略。 |
| 5. 沟通画像 | 用户如何表达和呈现？ | 决定邮件、IM、会议纪要、周报和汇报材料的语气与结构。 |
| 6. 授权画像 | AI 可以替用户做到哪一步？ | 决定检索、草拟、创建、发送、修改、执行等动作的确认边界。 |

# **六、六层画像内容设计**

| **画像层** | **关键字段** | **Banto 的适配方式** |
| --- | --- | --- |
| 角色画像 | 职能角色、项目角色、管理角色、决策权限、责任范围、当前重点 | 按用户职责筛选信息；会议总结突出用户相关事项；汇报自动站在用户角色视角。 |
| 知识画像 | 关注领域、常用知识源、术语偏好、证据要求、摘要粒度、输出格式 | 搜索结果个性化排序；总结时附出处；按用户常用格式输出。 |
| 任务画像 | 任务拆解、推进方式、反馈周期、决策方式、风险偏好、完成标准 | 将目标拆成合适颗粒度；识别阻塞项；生成行动清单和节奏提醒。 |
| 协作画像 | 高频协作对象、关系类型、协作节奏、依赖关系、承诺记录、敏感关系 | 会议前生成关系上下文；提醒历史承诺；识别谁在等用户。 |
| 沟通画像 | 上级、同级、下属、客户、全员公告、冲突场景的表达风格 | 按对象生成不同语气版本；自动控制正式度、边界感和承诺强度。 |
| 授权画像 | 搜索、总结、草拟、发送、改日历、建任务、删文件、外部动作、高风险动作权限 | 将个人授权边界转化为 Agent 执行策略。 |

# **七、从理论到落地：Banto 的工作上下文闭环**

落地时，Banto 不直接从一次行为判断用户，而是通过“事件 → 观察 → 画像候选 → 画像项 → Agent 策略 → 用户反馈”的闭环逐步建立工作上下文。

| **步骤** | **输入** | **处理** | **输出** |
| --- | --- | --- | --- |
| 1. 场景识别 | 用户请求、当前应用、文档类型、会议、沟通对象、工具动作 | 判断当前是搜索、总结、会议、沟通、任务还是执行场景。 | context：domain、subdomain、risk、audience、action\_type |
| 2. 信息抽取 | 显性设置、用户纠错、企业元数据、重复行为、任务与文档交互 | 提取工作相关观察，不直接生成永久结论。 | observation：可验证、带来源、带时间、带场景 |
| 3. 内容整合 | 新观察 + 历史画像 + 企业权限 + 当前上下文 | 去重、归并、冲突处理、置信度更新、场景拆分。 | profile\_item：有证据、有置信度、有适用范围 |
| 4. 策略适配 | 六层画像 + 当前场景 + 风险等级 | 转换为检索、总结、写作、提醒、执行策略。 | agent\_policy：输出结构、语气、证据、确认边界 |
| 5. 反馈闭环 | 用户修改、确认、拒绝、撤销、评价 | 将反馈作为最高价值信号更新画像。 | 更新、降级、遗忘或锁定相关画像项 |

## **七点一：落地架构**

| **模块** | **职责** |
| --- | --- |
| 企业数据层 | 连接文档、邮件、IM、日历、会议、项目管理、知识库等企业信息。 |
| 权限与合规层 | 处理 SSO、RBAC、审计、数据边界、管理员策略和企业安全要求。 |
| 个人工作上下文层 | 沉淀角色、知识、任务、协作、沟通、授权六层画像。 |
| Agent 策略层 | 将画像转化为检索策略、总结策略、沟通策略、任务策略和执行策略。 |
| Banto 工作台 | 承载搜索、总结、写作、会议、任务、知识整理和自动化执行。 |

# **八、信息抽取机制：如何从工作行为中形成画像**

## **八点一：可抽取的信息来源**

| **来源** | **可抽取内容** | **进入画像方式** |
| --- | --- | --- |
| 用户显性设置 | 角色、输出风格、知识来源、授权边界、沟通偏好 | 直接写入画像，置信度最高。 |
| 用户纠错 | “下次更简洁”“先问我”“不要自动发送”等反馈 | 更新已有画像或新建场景化规则，权重高。 |
| 企业元数据 | 部门、岗位、项目、权限、会议参与、知识库访问范围 | 作为基础上下文，不做心理判断。 |
| 工作内容交互 | 搜索、打开、收藏、引用、总结、转发、生成、修改 | 形成知识偏好、输出格式和证据要求的观察。 |
| 任务与会议行为 | 行动项、deadline、owner、阻塞项、承诺、后续跟进 | 形成任务画像和协作画像。 |
| 工具执行反馈 | 确认、拒绝、撤销、预览后修改、自动执行接受度 | 形成授权画像和风险边界。 |

## **八点二：抽取单元**

| **抽取层级** | **定义** | **示例** |
| --- | --- | --- |
| Event 事件 | 用户或系统中发生的一次原始行为。 | 用户将客户邮件草稿改得更正式。 |
| Observation 观察 | 从事件中抽出的可验证信号。 | 在客户沟通场景中，用户偏好更正式、审慎的表达。 |
| Profile Candidate 画像候选 | 尚未成为稳定画像的临时假设。 | customer\_email\_style 可能为 formal\_careful。 |
| Profile Item 画像项 | 通过显性确认、纠错或重复行为沉淀后的画像记录。 | 客户邮件默认正式审慎，发送前确认。 |
| Agent Policy 策略 | 画像项在当前场景下转成的执行规则。 | 生成客户邮件草稿，不自动发送，附更正式版本。 |

## **八点三：按六层画像抽取**

| **画像层** | **主要抽取信号** | **抽取结果** |
| --- | --- | --- |
| 角色画像 | 岗位、项目 owner、会议角色、审批权限、责任范围 | 用户在特定项目和流程中的职责与决策边界。 |
| 知识画像 | 常搜索来源、常引用材料、摘要长度、是否要求出处、术语处理 | 用户关心哪些知识，以及希望如何处理知识。 |
| 任务画像 | 用户如何拆任务、如何设置 deadline、是否要求 checklist、是否需要提醒 | 用户推进工作的节奏、粒度和完成标准。 |
| 协作画像 | 高频联系人、会议共现、承诺对象、阻塞关系、转发对象 | 用户与团队、同事、客户之间的责任和依赖关系。 |
| 沟通画像 | 用户对草稿的修改、对象层级、正式度、是否保留边界、是否避免承诺 | 不同对象和场景下的表达风格。 |
| 授权画像 | 用户确认、拒绝、撤销、允许自动执行的行为 | AI 在不同动作和风险等级下的执行边界。 |

## **八点四：抽取原则**

| **原则** | **说明** |
| --- | --- |
| 只抽取工作相关变量 | 不抽取性格、心理状态、健康状态、政治倾向、家庭关系等非工作必要信息。 |
| 从行为到观察，不从行为到人格 | 用户拒绝一次自动发送，只表示该场景需确认，不表示用户“保守”。 |
| 每条画像保留证据 | 保留来源、时间、上下文、触发动作和用户反馈。 |
| 高风险策略不靠推断 | 涉及外部发送、客户承诺、财务、合同、人事、公开发布等动作，必须由显性设置或确认行为驱动。 |

# **九、内容整合机制：如何把零散信息沉淀为可用画像**

## **九点一：画像项标准结构**

| **字段** | **含义** |
| --- | --- |
| id | 画像项唯一标识，例如 communication.customer\_email.style。 |
| domain | 所属层级：角色、知识、任务、协作、沟通、授权。 |
| context | 适用场景，例如客户邮件、同级协作、项目周会、外部会议。 |
| value | 具体偏好或规则，例如 formal\_careful、source\_required、confirm\_every\_time。 |
| confidence | 置信度，来自来源权重、重复次数、时间衰减和用户确认。 |
| evidence | 证据链：显性设置、纠错、重复行为、企业元数据、模型观察。 |
| scope | 适用范围与不适用范围，避免全局误判。 |
| agent\_policy | 该画像项如何影响搜索、总结、写作、提醒或执行。 |
| sensitivity | 敏感等级，用于控制可见性、保留时间和使用边界。 |
| lifecycle\_state | active、candidate、archived、expired、deleted 等生命周期状态。 |

## **九点二：整合规则**

| **规则** | **处理方式** |
| --- | --- |
| 显性设置优先 | 用户主动设置高于系统推断和重复行为。 |
| 纠错优先 | 用户最近纠错高于历史习惯。 |
| 重复行为增强置信度 | 同类行为多次出现后提升置信度，但仍保留场景范围。 |
| 场景拆分优先于全局覆盖 | 客户沟通、同级协作、上级汇报、内部草稿分别建模。 |
| 冲突不强行合并 | 若用户在不同场景表现不同，保留多个上下文画像。 |
| 高风险动作保守处理 | 缺少明确授权时，默认预览确认或禁止执行。 |
| 画像可解释 | 每次个性化建议都能说明使用了哪些上下文。 |

## **九点三：置信度来源**

| **来源** | **默认权重** | **使用方式** |
| --- | --- | --- |
| 用户显性设置 | 最高 | 直接进入 active 画像，可被用户随时修改。 |
| 用户明确纠错 | 高 | 更新或覆盖相关场景规则。 |
| 重复行为 | 中 | 多次出现后增强置信度，单次行为只作为候选。 |
| 企业元数据 | 基础 | 作为角色、权限和项目上下文，不推断心理属性。 |
| 模型语义观察 | 低 | 仅进入 candidate，不直接驱动高风险执行。 |

## **九点四：策略转化**

| **画像输入** | **转化为 Agent 策略** |
| --- | --- |
| 角色画像 | 决定回答视角、优先信息、会议责任和默认输出结构。 |
| 知识画像 | 决定是否附出处、摘要深度、搜索源优先级和术语处理方式。 |
| 任务画像 | 决定任务拆解颗粒度、提醒频率、优先级规则和完成标准。 |
| 协作画像 | 决定是否提醒 stakeholder、是否标记承诺、是否提高敏感关系确认级别。 |
| 沟通画像 | 决定语气、正式度、结构、版本数量和承诺边界。 |
| 授权画像 | 决定是否只建议、只草拟、可执行、需确认或禁止执行。 |

# **十、信息降级与遗忘机制**

Banto 的个人工作上下文需要保持“有用、不过度、可控、可撤销”。因此，画像不是永久累积，而是按生命周期进行降级、归档、过期和删除。

## **十点一：画像生命周期**

| **状态** | **含义** | **是否影响 Agent 行为** |
| --- | --- | --- |
| candidate 候选 | 来自低置信度观察或单次行为，等待确认或重复验证。 | 不直接影响高风险动作，只可轻微影响低风险输出。 |
| active 生效 | 有明确设置、纠错或稳定证据支持。 | 可以影响搜索、总结、写作、提醒和授权策略。 |
| contextual 场景有效 | 只在特定项目、会议、对象或时间窗口内有效。 | 仅在匹配场景下生效。 |
| archived 归档 | 历史证据保留，但不再主动影响策略。 | 不影响默认行为，可用于用户查看或审计。 |
| expired 过期 | 超过有效期或与当前角色、项目不再相关。 | 不再调用。 |
| deleted 删除 | 用户删除或策略要求删除。 | 完全移除或按企业策略做不可逆匿名化。 |

## **十点二：降级触发条件**

| **触发条件** | **降级方式** |
| --- | --- |
| 时间衰减 | 长期未使用、未确认的画像从 active 降为 archived 或 expired。 |
| 项目结束 | 项目相关角色、任务、协作关系从 active 降为 archived。 |
| 角色变化 | 岗位、团队、项目职责变化后，旧角色画像自动降级。 |
| 用户纠错 | 与纠错冲突的旧画像立即降级或失效。 |
| 低置信度 | 候选画像长期未被重复验证，自动过期。 |
| 敏感等级高 | 涉及外部沟通、客户承诺、授权边界的信息保留更谨慎，默认需要明确确认。 |
| 企业策略 | 按照企业安全、合规、数据保留策略进行删除或匿名化。 |

## **十点三：保留周期建议**

| **信息类型** | **建议生命周期** | **处理方式** |
| --- | --- | --- |
| 当前工作状态 | 短周期 | 用于近期计划和提醒，过期后自动删除或降级。 |
| 会议行动项 | 直到完成或失效 | 完成后归档，保留为项目上下文。 |
| 项目上下文 | 项目活跃期内 | 项目结束后降级为历史项目资料。 |
| 沟通偏好 | 中长期 | 持续生效，但可被用户纠错覆盖。 |
| 知识偏好 | 中长期 | 根据使用频率和反馈更新。 |
| 授权边界 | 长期且显性 | 只有用户明确设置或修改后才变化。 |
| 模型推断项 | 短周期 | 未被确认或重复验证时自动过期。 |

## **十点四：遗忘原则**

| **原则** | **说明** |
| --- | --- |
| 用户可手动遗忘 | 用户可以删除单条画像、某个场景画像或全部工作上下文。 |
| 默认最小必要 | 只保留对当前工作有用的信息，不永久保留低价值行为痕迹。 |
| 低置信度自动遗忘 | 候选画像未被确认或重复使用时自动失效。 |
| 上下文失效自动遗忘 | 项目、角色、任务结束后，对应画像不再主动生效。 |
| 敏感信息优先遗忘 | 不必要的敏感信息不进入画像；如误入，应优先删除。 |
| 遗忘可解释 | 系统应说明某条画像为何过期、降级或不再使用。 |

# **十一、核心应用场景**

| **场景** | **Banto 的工作方式** | **输出结果** |
| --- | --- | --- |
| 企业搜索 | 结合角色、项目、知识源、协作对象和时间上下文，对搜索结果进行个性化排序。 | 返回与用户当前工作最相关的信息，并附来源、影响和下一步建议。 |
| 文档总结 | 根据用户角色提取重点，不做通用摘要。 | 生成与用户相关的结论、行动项、风险点、需同步对象和原文证据。 |
| 会议工作台 | 会前整理背景和承诺，会中提取决策，会后生成个人 follow-up。 | 输出用户行动项、owner、deadline、待同步对象和后续任务。 |
| 企业沟通辅助 | 识别组织关系、对象身份、沟通风险和用户表达风格。 | 生成适合上级、同级、下属、客户或全员场景的邮件、IM、纪要和汇报。 |
| 个人知识整理 | 按用户的项目、客户、主题、任务、汇报需求组织企业与个人知识。 | 形成可检索、可复用、可汇报的个人知识工作区。 |
| 跨系统执行 | 根据授权画像判断执行边界。 | 低风险自动处理；中高风险预览确认；高风险禁止或强确认。 |

# **十二、Agent 行为策略**

|  |
| --- |
| **策略转换** Banto 将个人工作上下文转化为 Agent 策略，直接影响检索、总结、写作、计划和执行。 |

| **当前场景** | **调用画像** | **默认策略** |
| --- | --- | --- |
| 搜索客户反馈 | 角色画像、知识画像、当前项目 | 优先返回与用户职责和项目相关的反馈，并附来源。 |
| 总结会议 | 角色画像、协作画像、任务画像 | 突出用户行动项、相关 stakeholder、阻塞项和 deadline。 |
| 撰写客户邮件 | 沟通画像、授权画像、协作画像 | 生成正式审慎草稿，不自动发送，避免过度承诺。 |
| 生成周报 | 任务画像、知识画像、沟通画像 | 输出进展、结果、风险、下周计划和需要支持事项。 |
| 跨系统执行 | 授权画像、风险等级、企业权限 | 低风险执行；中风险确认；高风险禁止或强确认。 |

# **十三、授权分级设计**

| **级别** | **含义** | **示例** |
| --- | --- | --- |
| L0 只检索 | 只查找和展示信息 | 搜索文档、找邮件、查会议记录 |
| L1 只建议 | 给出分析和建议，不生成正式内容 | 分析优先级、提出处理方案 |
| L2 可草拟 | 生成草稿，但不发送、不提交 | 邮件草稿、会议纪要、汇报提纲 |
| L3 可本地执行 | 可整理本地内容，需可撤销 | 文件归档、任务创建、笔记整理 |
| L4 企业系统操作 | 可改日历、建 ticket、更新状态，默认需确认 | 创建项目任务、调整内部会议 |
| L5 高风险操作 | 涉及客户承诺、财务、合同、人事、公开发布 | 强确认或禁止自动执行 |

# **十四、核心产品页面**

| **页面** | **内容** |
| --- | --- |
| 我的工作上下文 | 展示 Banto 对用户角色、当前重点、常用知识源、输出偏好和默认工作方式的理解。 |
| AI 授权边界 | 展示和管理搜索、总结、草拟、发送、日历、任务、文件、外部动作和高风险动作权限。 |
| 沟通风格设置 | 按上级、同级、下属、客户、冲突场景、全员公告设置默认表达方式。 |
| 知识偏好设置 | 设置摘要长度、出处要求、优先来源、输出格式、术语解释和时间偏好。 |
| 画像证据与来源 | 展示某条画像来自显性设置、用户纠错、重复行为还是企业元数据。 |
| 降级与遗忘管理 | 允许用户查看、删除、关闭或重置某类工作上下文。 |
| 为什么这样建议 | 解释当前建议或执行策略使用了哪些工作上下文。 |

# **十五、企业治理边界**

|  |
| --- |
| **治理原则** 个人工作上下文只用于让 Banto 更好地协助用户完成工作，不用于绩效评估、员工评级或管理者监控。 |

| **不用于** | **用于** |
| --- | --- |
| HR 绩效画像 | 个人工作台的上下文适配 |
| 员工心理测评 | 输出偏好、协作规则和授权边界 |
| 管理者监控 | 用户自己可见、可改、可删的私有设置 |
| 离职风险预测 | 任务、知识、沟通和执行策略优化 |
| 团队政治分析 | 工作关系中的责任、承诺和协作上下文 |

| **治理要求** | **说明** |
| --- | --- |
| 可见 | 用户能看到 Banto 记住了什么。 |
| 可改 | 用户能修改错误画像和场景适用范围。 |
| 可删 | 用户能删除单条画像或清空上下文。 |
| 可解释 | 关键建议和执行动作能说明使用了哪些上下文。 |
| 可审计 | 高风险动作保留必要的操作记录和确认记录。 |
| 可配置 | 企业可配置保留周期、敏感边界、权限策略和合规要求。 |

# **十六、MVP 路线**

| **阶段** | **建设重点** | **交付能力** |
| --- | --- | --- |
| V1 工作上下文设置 | 显性收集角色、目标、输出偏好、知识来源和授权边界。 | 形成基础个人工作上下文。 |
| V2 抽取与证据链 | 将用户纠错、重复行为、企业元数据转为可验证观察。 | 形成带来源、置信度和场景范围的画像项。 |
| V3 内容整合 | 建立画像项合并、冲突处理、场景拆分和置信度更新机制。 | Banto 能稳定适配不同工作场景。 |
| V4 策略适配 | 将画像应用到企业搜索、总结、会议、沟通和周报。 | 不同工作场景输出不同结构、重点和语气。 |
| V5 授权执行 | 将授权画像接入 Agent 工具调用。 | 低风险自动处理，中高风险预览确认，高风险禁止或强确认。 |
| V6 降级与遗忘 | 接入画像生命周期、过期、归档、删除和用户管理。 | 保证上下文持续有用、不过度沉淀、可控可撤销。 |

# **十七、典型使用示例**

| **用户请求** | **Banto 处理方式** |
| --- | --- |
| “帮我总结这份客户反馈。” | 按用户角色提取需求、风险、影响当前项目的点，并附来源。 |
| “帮我给研发同事催一下进度。” | 识别同级协作场景，生成直接但礼貌的提醒版本，并保留责任边界。 |
| “帮我生成这周周报。” | 汇总会议、文档、任务和项目更新，输出进展、结果、风险、下周计划和需要支持事项。 |
| “帮我把会后事项都安排一下。” | 提取行动项，创建任务前展示 owner、deadline 和涉及对象，按授权边界确认执行。 |
| “以后客户邮件都先让我确认。” | 更新授权画像，将客户邮件发送策略设为每次确认。 |
| “这个项目已经结束了，不要再优先推荐相关资料。” | 将该项目上下文从 active 降为 archived，不再主动影响搜索排序。 |
