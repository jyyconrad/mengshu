# banto 记忆体系设计 v0.6

> **定位**：基于 banto 产品形态（短时任务对话 / Project）的记忆体系，围绕 agent 启动任务时回答的 5 个本质问题展开
> **版本**：v0.6-rev8
> **日期**：2026-06-02
> **状态**：设计中

---

## 零、核心主线：5 个本质问题

> **v0.6-rev8 核心**：整套记忆系统围绕 agent 启动任务时回答的 5 个本质问题展开。自动化/定时任务不再作为独立记忆容器，而是复用普通对话与 Project 记忆链路。

| # | 问题 | 召回什么 | 对应 type |
|---|------|---------|----------|
| Q1 | **我为谁工作？** | 用户画像与偏好 | `profile` |
| Q2 | **我在做什么？** | 任务上下文 | `task-context` |
| Q3 | **什么不能做？** | 行为约束边界 | `rules` |
| Q4 | **之前怎么做过？** | 历史经验 | `experience` |
| Q5 | **有什么可用资源？** | 资源指针（专家/技能/连接器/知识库） | `resource` |

任何不能回答这 5 个问题之一的设计都是噪音。

## 一、设计目的：banto 记忆体系是工作进化系统

记忆体系不是被动的信息存储，而是 banto 持续进化的核心引擎。它承载三重职能：

### 三重职能

| 职能 | 含义 | 对应的能力 |
|---|---|---|
| **做好当下** | 让 banto 更好地为客户工作，提升当下任务的执行质量 | 记忆写入、读取召回、上下文注入 |
| **沉淀能力** | 通过持续的自我分析升级，从重复模式中迭代出 Skill，降低工作难度 | 模式识别、Skill 候选生成、升格审核 |
| **优化未来** | 从日常工作中发现可优化点，主动诊断、预测、建议 | 工作模式分析、流程优化建议、能力缺口提示、工作预测、后续工作建议 |

### 三重职能的关系

```
                  banto 记忆体系（工作进化系统）
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   【做好当下】          【沉淀能力】          【优化未来】
        │                   │                   │
   服务"现在的工作"      把重复变能力         把工作变更聪明
        │                   │                   │
   写入 / 召回 / 注入    模式识别 / 升格        诊断 / 预测 / 建议
        │                   │                   │
        └───────────────────┼───────────────────┘
                            ▼
                    用户的工作越来越顺
```

**关键洞察**：三重职能并非独立运行，而是**相互喂养**——做好当下产生记忆原料，沉淀能力把原料变成可复用资产，优化未来基于已沉淀的资产主动出招。

### 输出形式（来自三重职能）

| 输出形式 | 来源职能 | 时机 |
|---|---|---|
| Skill 候选 | 沉淀能力 | 用户主动询问时输出 |
| 流程优化建议 | 优化未来 | 用户主动询问时输出 |
| 能力缺口提示 | 优化未来 | 用户主动询问时输出 |
| 工作预测 | 优化未来 | 用户进入空闲态时主动推送 |
| 后续工作建议 | 优化未来 | 用户进入空闲态时主动推送 |

**克制度原则**：banto 主动推送的是"低风险建议"（工作预测、后续建议），高风险建议（Skill 候选、流程优化）等用户主动询问。避免变成唠叨工具。

## 三、核心设计概览

### 3.1 容器记忆模型

| 记忆层 | 所有权 | 映射容器 | 典型节点类型 | 生命周期 |
|--------|--------|----------|--------------|----------|
| **全局记忆** | 用户账号 | 所有会话的基础上下文 | `profile`, `rules`, `resource` | 跨会话、跨项目持久化 |
| **项目记忆** | 单个 Project | Project 容器 | `task-context`, `rules`, `experience`, `resource` | Project 生命周期内持久化 |
| **会话候选区** | 单次 Session | 短时任务对话 / Project 内会话 | 待晋升的 5 type 候选 | 会话期 + 7 天归档 |
| **团队/企业记忆** | 团队/企业 | 远程下发的公共约束和资源 | `rules`, `experience`, `resource` | 团队/企业存续期 |

**核心原则**：
- **向上晋升，向下继承** — 会话 → 项目 → 全局（晋升），全局 → 项目 → 会话（继承）
- **容器即边界** — 记忆分层与短时任务对话 / Project 主线对应
- **自动化不建独立记忆容器** — 自动化是系统触发的一次对话执行，复用全局 + Project 记忆；执行过程归自动化日志
- **全局记忆所有权归用户账号** — Identity 不作为产品对象或记忆 scope；岗位/团队/角色信息作为 profile/task-context 的输入维度
- **二维记忆模型**（v0.6-rev2）— 每个容器内部按 L0（原始）/ L1（摘要）两层组织

### 3.1.1 二维记忆模型：容器层 × 折叠层（v0.6-rev2 升级）

借鉴 OpenHuman 记忆树的金字塔压缩思想，banto 在容器层之上引入**第二维度——折叠层**：

```
┌──────────────────────────────────────────────────────────────┐
│                   容器层（横向）                               │
├──────────────┬──────────────┬──────────────┬─────────────────┤
│  全局记忆     │  项目记忆     │  会话候选区   │  团队/企业缓存    │
├──────────────┼──────────────┼──────────────┼─────────────────┤
│ L1 摘要      │ L1 摘要      │ 候选索引      │ L1/L2 摘要       │ ← 折叠
│ L0 原始      │ L0 原始      │ L0 候选       │ L0/L1 原始       │ ← 事实
└──────────────┴──────────────┴──────────────┴─────────────────┘
                  折叠层（纵向，2 层）
```

| 层 | 含义 | 谁生成 |
|---|---|---|
| **L0 原始** | AI 观察到的事实 / 用户提供的偏好 / 外部数据提炼的决策 | 写入触发时立即生成 |
| **L1 摘要** | 按时间/主题/容器三视角对 L0 群的 LLM 摘要 | chat-end 阶段 1.5 + TTL Flush |

**关键约束**：L0 节点不会因系统摘要折叠、TTL 或后台治理被主动删除；用户显式删除、撤回或清空记忆不受该约束限制。L1 节点的 `sourceNodeIds` 永远反向指向仍存在的具体 L0。详见 [01 文档](./01-memory-layers-and-containers.md)。

### 3.2 md+json 工作图谱

**双存机制**：
- **JSON**：结构化存储，支持多维索引（类型 / 置信度 / 引用次数 / 时间 / **热度**）
- **Markdown**：人类可读，frontmatter + 正文 + 关系链接 + 历史版本

**Markdown 主文件规则**：
- `experience` / `resource` 的正文内容直接写在各自 Markdown 主文件的 frontmatter 后面，不额外使用 `fileRef` 指向同一个正文文件。
- 递进式编写只在内容非常复杂、详细、篇幅过长时启用：主文件保留概述、必要正文和附录索引，详细决策依据、复杂参数、历史版本等写入 appendix 附录文件。
- 必读层仍只注入概要索引；需要详情时先读取主文件正文，再按主文件中的附录索引读取附录。

**图谱结构**：
- **节点**：4 类基础类型（user/feedback/project/reference）+ 扩展类型（decision/artifact/task/expert/skill 等）+ **summary 类型 L1 节点**（v0.6-rev2 新增）
- **关系**：5 类关系（derives_from / conflicts_with / depends_on / supersedes / references）
- **容器**：节点归属于全局 / 项目 / 会话候选区 / 团队 / 企业等容器；自动化触发复用项目和会话容器

**目录布局**：
```
~/.iflymate/memory/
├── nodes/                    # 节点存储（按容器分层）
│   ├── global/              # 全局记忆节点
│   ├── projects/<projectId>/ # 项目记忆节点
│   ├── sessions/<sessionId>/ # 会话记忆节点
│   └── _sessions/<sessionId>/ # 会话候选节点
├── edges/                    # 关系存储（独立管理）
├── insights/                 # 沉淀能力 + 优化未来的产物
│   ├── skill-candidates/    # Skill 候选
│   ├── optimization-suggestions/  # 流程优化建议
│   ├── capability-gaps/     # 能力缺口提示
│   ├── predictions/         # 工作预测
│   └── follow-ups/          # 后续工作建议
└── indexes/                  # 多维索引
    ├── by-type.json
    ├── by-confidence.json
    ├── by-reference-count.json
    └── by-time.json
```

### 3.3 chat-end 触发的工作图谱整理 + 主动观察

**核心流程**（v0.6-rev2 扩展为四阶段）：

```
阶段 1（做好当下）：
会话结束 → 10s debounce → 提取最近 20 条消息
→ LLM 分析（节点/关系/决策/摘要）
→ 去重合并 → 写入 L0/

阶段 1.5（折叠层维护，v0.6-rev2 新增）：
基于 L0 变更 → 检查 bucket 满 / TTL 到期
→ 按时间/主题/容器三视角生成 L1 摘要
→ 写入 L1/ 子目录（旧版归档）
→ L0 不因系统折叠或 TTL 主动删除

阶段 2（沉淀能力）：
基于 L1 摘要 → 模式识别（高频引用 / 跨项目复用）
→ 生成 Skill 候选 → 写入 insights/skill-candidates/
→ 等待用户主动审核

阶段 3（优化未来）：
基于 L1 摘要 + 当前任务 → 工作模式分析
→ 生成工作预测 / 后续建议 / 能力缺口提示
→ 写入 insights/ 对应目录
→ 低风险建议（工作预测 / 后续建议）在用户进入空闲态时推送
→ 高风险建议（Skill 候选 / 流程优化）等用户主动询问
```

**性能优化**：
- 四阶段都异步执行，不阻塞用户
- 阶段 2 和 3 基于 L1 摘要而非 L0，效率显著提升
- 阶段 2 和 3 可并行
- 增量更新热点节点

**召回升级**：复合排序公式 `score = α × hotness + β × confidence + γ × task_relevance + δ × user_signal`，从单一 confidence 升级为多维度复合分。详见 [03 文档第 5.5 节](./03-runtime-mechanisms.md)。

### 3.4 与支撑体系的耦合

| 支撑体系 | 与记忆的联动方式 |
|---------|----------------|
| **专家智能体** | 记忆可标记 `relatedExpertIds`，专家调用时优先注入相关记忆；用户反馈生成带专家标记的 feedback 记忆 |
| **技能** | 高频记忆（≥5 次引用 + 跨项目复用）可晋升为候选 Skill，需用户审核；Skill 评估分数影响相关记忆置信度 |
| **连接器（MCP）** | 外部数据不直接进入记忆，只提炼基于外部数据的**决策、约束、偏好**；记忆可标记 `sourceConnector` |
| **后端记忆服务**（v0.6-rev5 重构） | 双端架构：APP 本地（用户+AI 维护）+ 后端（管理员+系统维护）；agent 只调本地 `memory-recall`，对双端无感知；远程→本地单向同步（登录拉全量+定时拉增量）；用户主动贡献走 UI 通道，agent 不能上传；详见 [07 文档](./07-dual-end-memory-architecture.md) |

> **注 1**：产品设计上已经移除 Identity，不再作为一级产品对象或独立记忆 scope；但岗位、团队归属、默认工作角色等身份信息仍是 profile/task-context 召回时的重要输入维度，用于重现用户画像的信息模型。全局记忆的所有权归"用户账号"层。
>
> **注 2**：banto 不引入 Obsidian Vault 等第三方文件镜像。OpenHuman 中"用户可编辑权威事实"的需求由 banto 已有的**知识库能力**承接（事实层归知识库，工作上下文归 banto 记忆）。详见 [04 文档第 7.5 节](./04-coupling-with-capability-system.md)。

### 3.5 双端记忆架构（v0.6-rev5 引入）

记忆体系采用 **APP 本地 + 后端远程** 双端架构：

- **APP 本地**：用户 + AI 实时维护，纯文件元数据召回，agent 唯一入口 `memory-recall`
- **后端远程**：管理员 + 系统维护，向量化检索 + 记忆树 + 决策链，远程→本地单向同步

6 条核心原则：agent 对双端无感知 / 远程是只读权威源 / 同步单向 / 贡献走非 agent 通道（SKILL 晋升机制）/ 数据结构统一 / 记忆→SKILL→团队。

详见 [07 文档](./07-dual-end-memory-architecture.md)。

| 7. 升格需要用户审核 | ✅ 需要审核 |

### 5.2 设计哲学补充决策

| 决策 | 选择 | 含义 |

## 六、与现状的差异

| 维度 | 当前 v0.3-v0.4 | v0.6 设计 | 升级方向 |
|------|----------------|-----------|---------|
| **设计哲学** | 上下文连续性 | 工作进化系统（三重职能） | 从被动记录升级为主动进化 |
| **记忆分层** | 全局 + 项目两级 | 全局 + 项目 + 会话候选区 + 团队/企业远程缓存 | 新增候选区与双端缓存 |
| **数据模型** | 纯 Markdown + frontmatter | md+json 双存 + 工作图谱（节点 + 关系） | 引入结构化图谱 |
| **写入触发** | 仅 chat-end 后被动提取 | chat-end 三阶段（整理 + 模式识别 + 主动建议） | 多触发源 + 主动观察 |
| **读取方式** | 全量截断注入（8K 上限） | P0 固定 5 槽位系统自动注入；后续扩展任务相关层 / 按需检索 | 先稳住自动注入 |
| **chat-end 机制** | 10s debounce + 提取 5 条记忆 | 三阶段异步流程（图谱整理 + Skill 候选 + 工作建议） | 从条目提取升级为工作进化 |
| **与产品容器映射** | 未显式映射 | 围绕短时任务对话 / Project 主线，自动化触发复用同一链路 | 显式映射 |
| **与专家关系** | 解耦 | 记忆可标记 `relatedExpertIds`，专家调用时优先注入 | 新增联动 |
| **与技能关系** | 解耦 | 高频记忆可晋升为候选 Skill | 新增晋升通道 |
| **与连接器关系** | 未定义 | 外部数据不直接进入，只提炼规则 | 新增边界 |
