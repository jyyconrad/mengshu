## 二、5 个本质问题与 5 type 的映射

agent 启动任务时需要回答 5 个本质问题，每个问题对应一个核心记忆 type：

| 问题 | Type | 内容 | 决定 agent 的什么 |
|------|------|------|-----------------|
| **Q1 我为谁工作？** | `profile` | 用户身份/岗位/偏好/表达习惯 | 输出风格、详细度、表达方式 |
| **Q2 我在做什么？** | `task-context` | 项目目标/阶段/客户约束/里程碑 | 目标、优先级、阶段感 |
| **Q3 什么不能做？** | `rules` | 个人偏好约束/团队规范/合规/资源边界 | 行动边界、合规底线 |
| **Q4 之前怎么做过？** | `experience` | 决策依据/方法论种子/领域知识 | 执行模式、决策依据 |
| **Q5 有什么可用资源？** | `resource` | 知识库/外部系统/专家/技能/连接器指针 | 调用哪些工具/文档 |

**核心约束**：每条记忆必须明确归属于 5 个 type 之一。任何无法归类的内容要么是会话过程数据（移出记忆系统），要么是设计中没考虑到的盲区（需要单独讨论）。

## 二·一、社会心理学抽取框架：5 type 不是“怎么看”，而是“怎么抽”

> **核心修正（v0.6-rev9）**：社会心理学在这里不是解释层，不是帮助我们事后说明"为什么这样设计"；它应该直接成为**信息抽取的分类依据**。也就是说，我们不是先抽一条记忆，再从社会心理学角度解释它，而是先定义：一个人/一个任务/一条规则/一段经验/一个资源，应该按哪些稳定的心理与社会行为维度被识别、归并和组织。

### 2.1.1 5 type 的抽取依据总表

| Type | 要抽的不是 | 真正要抽的是 | 社会心理学依据 |
|------|-----------|-------------|--------------|
| `profile` | 零散偏好 | **稳定的人格化工作倾向**：驱动方式、成就取向、控制感、表达风格、风险偏好 | 自我决定理论、成就目标理论、控制点理论、印象管理 |
| `task-context` | 事件流水账 | **任务所处的社会情境**：目标张力、角色关系、时间压力、责任结构、协作场域 | 情境认知、角色理论、目标设定理论 |
| `rules` | 单句命令 | **被明确外化的规范与边界**：禁区、义务、授权范围、规训来源 | 心理契约、规范内化、权威与服从 |
| `experience` | 历史事实 | **可迁移的决策模式**：在什么情境下、因为什么、采用了什么策略、带来什么结果 | 自传体记忆、归因理论、经验学习循环 |
| `resource` | 工具名单 | **可借用的社会/组织能力**：外脑入口、资源依赖、专家分工、制度化工具 | 分布式认知、社会资本、组织支持系统 |

**关键结论**：5 type 的抽取单位不是"句子"，而是"稳定心理-行为模式"。一句话能不能入库，不看它说得响不响，而看它能不能映射到一个稳定维度。

### 2.1.2 `profile` 的维度拆分

**理论依据**：自我决定理论（动机来源）、成就目标理论（目标取向）、控制点理论（控制偏好）、调节焦点理论（风险态度）

**粗粒度维度（v1，后续迭代）**：

| 维度 | 理论来源 | 两端 | 对 agent 的影响 |
|------|---------|------|----------------|
| **动机来源** | 自我决定理论 | 内驱（兴趣/掌握）↔ 外部规训（合规/交付/评价） | 决定优先给"更优方案"还是"更安全方案" |
| **成就取向** | 成就目标理论 | 掌握导向（学习/质量）↔ 表现导向（结果/效率） | 决定输出侧重"打磨"还是"落地" |
| **控制偏好** | 控制点理论 | 高自主（自己掌控）↔ 可授权（委托代办） | 决定 agent 能自动推进到什么程度 |
| **风险态度** | 调节焦点理论 | 促进焦点（追求收益）↔ 预防焦点（避免损失） | 决定是否主动给实验性方案 |

**前期粗抽规则**：只在用户明确表达 + 多次重复时才归维度，不强行归类。允许输出"待观察"。

### 2.1.3 `task-context` 的维度拆分

**理论依据**：情境认知理论（认知镶嵌在情境中）、角色理论（角色身份与期望）、目标设定理论（目标属性）

**粗粒度维度**：

| 维度 | 理论来源 | 要抽什么 | 对 agent 的影响 |
|------|---------|---------|----------------|
| **目标结构** | 目标设定理论 | 目标内容 + 来源（自设/指派）+ 承诺度 | 决定优先级和推进方向 |
| **角色关系** | 角色理论 | 我的角色 + 协作方角色 + 责任归属 | 决定沟通对象和汇报方向 |
| **约束情境** | 情境认知 | 时间约束 + 资源约束 + 外部要求 | 决定方案的可行边界 |

**前期粗抽规则**：只抽影响"项目整体方向"的信息，不抽单条任务执行细节。

### 2.1.4 `rules` 的维度拆分

**理论依据**：心理契约理论（契约类型与违约感知）、规范激活理论（规范来源与内化程度）

**粗粒度维度**：

| 维度 | 理论来源 | 要抽什么 | 对 agent 的影响 |
|------|---------|---------|----------------|
| **规范来源** | 规范激活理论 | 个人偏好 / 团队约定 / 组织制度 / 法规合规 | 决定优先级（合规 > 组织 > 团队 > 个人） |
| **约束强度** | 心理契约理论 | 硬约束（禁止/必须）↔ 软约束（建议/期望） | 决定 agent 是"绝对不做"还是"尽量不做" |
| **后果意识** | 规范激活理论 | 违反后果是什么（高代价/低代价/未知） | 决定在资源紧张时是否可以突破 |

**前期粗抽规则**：只抽含明确否定/必须语气的持续约束，不抽一次性纠正。

### 2.1.5 `experience` 的维度拆分

**理论依据**：自传体记忆理论（情节记忆 vs 语义记忆）、归因理论（因果解释）、经验学习循环（Kolb）

**粗粒度维度**：

| 维度 | 理论来源 | 要抽什么 | 对 agent 的影响 |
|------|---------|---------|----------------|
| **因果链** | 归因理论 | 决策 + 原因 + 结果（缺 why 不入库；outcome 可后补） | 让 agent 能解释"为什么建议这样做" |
| **迁移边界** | 经验学习循环 | 这个经验适用于什么场景、不适用于什么 | 避免 agent 错误泛化 |
| **记忆类型** | 自传体记忆 | 情节型（具体事件）↔ 语义型（通用方法论） | 情节型直接参考，语义型晋升 SKILL |

**前期粗抽规则**：缺 Why 不入库——只有"决定了什么"没有"为什么"的记忆不入 experience。

### 2.1.6 `resource` 的维度拆分

**理论依据**：分布式认知理论（认知主体分布）、交互记忆系统理论（元知识）、社会资本理论（信任度）

**粗粒度维度**：

| 维度 | 理论来源 | 要抽什么 | 对 agent 的影响 |
|------|---------|---------|----------------|
| **认知主体** | 分布式认知 | 人（专家）/ 工具（连接器/SKILL）/ 制品（文档/知识库） | 决定调用什么类型的外脑 |
| **元知识** | 交互记忆系统 | "谁知道什么" / "什么能做什么" / "什么记录了什么" | 让 agent 知道去哪问 |
| **信任度** | 社会资本理论 | 权威来源 / 团队验证 / 未验证 | 影响资源召回优先级 |

**前期粗抽规则**：只存"指针 + 适用场景 + 信任度"，不存资源的具体内容。

### 2.1.7 抽取的统一原则：抽"稳定模式"，不抽"表面句式"

| 原则 | 含义 |
|------|------|
| **抽稳定性** | 至少能跨任务复用，不能只服务当前一句话 |
| **抽生成机制** | 尽量识别"为什么他会这样要求"，而不只记"他要求了什么" |
| **抽行为导向** | 抽出来之后必须能改变 agent 的行为 |
| **抽社会位置** | 判断这条信息来自个人、项目、团队还是企业规训 |
| **抽可解释性** | 每一条都要能回答：为什么归到这个维度？ |

### 2.1.8 用户感受导向（仍然保留，但降为约束层）

社会心理学抽取框架是**正向分类依据**，用户感受是**负向约束条件**：

| 约束 | 含义 |
|------|------|
| **被理解，不被审视** | 可以抽工作风格，不可以抽人格评价 |
| **被尊重，不被监控** | 可以抽稳定模式，不可以抽私密生活 |
| **省心，不啰嗦** | 可以抽驱动维度，不可以把每句话都记成 profile |
| **可控，不黑箱** | 可以形成画像，但必须能回溯原话 |

### 2.1.9 通用提取边界（5 type 共享）

| 必须满足 | 必须避开 |
|---------|---------|
| 用户**主动表达**或**多次重复**的稳定模式 | 一次性情绪反应（"今天好累"）、临时上下文（"刚才那个文件"） |
| 对**未来**任务有指导价值 | 仅当下相关的事实（"这周三下午 3 点开会"） |
| 能映射到某个**稳定维度** | 模糊感受、含糊态度（"感觉还行吧"） |
| **不暴露隐私**敏感信息 | 身份证号、银行卡号、密码、家庭关系细节 |
| 用户**会希望 agent 记住** | 用户会觉得"你在分析我"的内容 |

> **底线红线**：宁可漏记，不可错记。尤其是 `profile`，一旦把用户误判为"内驱型/保守型/强控制型"，后续行为就会整体偏航，比漏掉一条偏好危险得多。

### 2.1.10 统一生命周期字段（`lifecycleStatus`）

所有 5 type 共享同一个生命周期状态字段，避免散落多个布尔标记：

| 值 | 含义 | 对召回的影响 | 适用场景 |
|---|---|---|---|
| `active` | 当前有效（默认值） | 正常参与召回 | 所有新写入的记忆 |
| `archived` | 历史归档（已被新版本取代但保留用于回溯） | 不参与必读层/任务相关层召回；按需检索可命中 | task-context 阶段切换、profile 新证据覆盖旧值 |
| `revoked` | 被用户或系统显式废止 | 完全不参与召回 | rules 被撤回、resource 被卸载 |
| `superseded` | 被更高优先级或更新版本的同类节点取代 | 不参与必读层召回；按需检索可命中 | rules 冲突解决、profile 同维度覆盖 |
| `promoted` | 已晋升为 SKILL 或更高抽象 | 降低召回优先级（避免与 SKILL 重复）；按需检索可命中 | experience → SKILL |

**附加字段**（当 lifecycleStatus != active 时可选填写）：
- `supersededBy: string` — 取代本节点的新节点 id
- `promotedToSkillId: string` — 晋升到的 SKILL id
- `revokedAt: string` — 废止时间
- `revokedReason: string` — 废止原因

### 2.1.11 5 type 统一入库阈值表

| Type | 入库下限 | 候选区 → 共享区 | 前置过滤（直接丢弃） | 备注 |
|------|---------|----------------|---------------------|------|
| `profile` | confidence ≥ 0.7 | 0.9 + 多次重复 | value=unknown 不入库 | 同维度唯一 |
| `task-context` | confidence ≥ 0.7 | 0.9 / 用户确认 | 缺项目级触发词降级为 experience 候选 | 纯本地 |
| `rules` | confidence ≥ 0.8 | 0.9 / 用户配置 | strength=soft 直接丢弃；推断的约束丢弃 | compliance 永不淘汰 |
| `experience` | confidence ≥ 0.75 | 0.9 + 用户确认 | 缺 Why 直接丢弃 | what+why 必须齐备 |
| `resource` | confidence ≥ 0.7 | 0.95 / 用户主动挂载 | trustLevel=low 直接丢弃 | 只存指针 |

**通用规则**：
- confidence < 入库下限 → 直接丢弃
- 入库下限 ≤ confidence < 候选区阈值 → 写入 `_sessions/<sid>/candidates/`
- confidence ≥ 候选区阈值 → 直接写入目标容器共享区
- 必读层固定注入不影响 hotness（不计入 referenceCount）

## 三、5 type 详解（含抽取维度 / 提示词 / 聚合规则）

每个 type 的详解按以下统一结构组织：

1. **定位**：这个 type 服务的本质问题
2. **抽取维度**：按什么社会心理学维度或社会行为维度去识别
3. **典型内容**：举例
4. **社会心理学抽取依据**：为什么这些维度能直接指导 agent 行为
5. **提取提示词模板**：给 LLM 的指令
6. **聚合去重规则**：写入时如何合并
7. **反例**（不该提取的内容）
8. **写入触发 / 召回方式 / 禁止内容**

> **统一原则**：一切提取动作都不是为了把用户“描述清楚”，而是为了把用户在工作中的稳定模式“抽出来”。提取偏保守、归类偏中性、合并偏激进。

### 3.1 `profile` — 我为谁工作

#### 3.1.1 定位

`profile` 不是简单的"用户偏好记录"，而是用户在工作中的**稳定协作人格模型**。它回答的不是"用户喜欢什么语气"这么表层的问题，而是：

- 这个人做事主要受什么驱动？
- 这个人更在意过程质量还是结果交付？
- 这个人希望自己掌控到什么程度？
- 这个人面对风险时偏保守还是偏探索？
- 这个人如何处理组织关系、客户关系与对外表达？

agent 只有知道这些，才能真的像长期合作同事，而不是像一个只会记格式要求的工具。

#### 3.1.2 抽取维度（前期粗粒度 v1）

> 与第二·一节 2.1.2 一致。前期只抽 4 个核心维度，每个维度用理论的两极做粗分类。后续根据数据反馈再扩展（v2 候选见末尾）。

**核心 4 维（v1，前期落地）**：

| 维度 | 理论来源 | 两端 | 对 agent 的直接影响 |
|------|---------|------|-------------------|
| **动机来源**（motivation） | 自我决定理论 | 内驱（intrinsic）↔ 外部规训（external_regulated） | 决定优先给"更优方案"还是"更稳妥/合规方案" |
| **成就取向**（achievement_orientation） | 成就目标理论 | 掌握导向（mastery）↔ 表现导向（performance） | 决定输出侧重"打磨质量"还是"落地交付" |
| **控制偏好**（control_preference） | 控制点理论 | 高自主（high_control）↔ 可授权（delegatable） | 决定 agent 能自动推进到什么程度 |
| **风险态度**（risk_attitude） | 调节焦点理论 | 促进焦点（promotion）↔ 预防焦点（prevention） | 决定是否主动给实验性方案 |

**v2 候选维度**（数据稳定后再开启）：表达组织方式 / 关系协作方式 / 时间压力阈值。前期不抽，避免维度过多导致归类混乱。

#### 3.1.3 典型内容

| 维度 | 示例 |
|------|------|
| 动机来源（intrinsic） | "只要逻辑自洽，我愿意多花时间打磨" |
| 动机来源（external_regulated） | "先按客户要求做对，别自我发挥" |
| 成就取向（mastery） | "我希望方案看起来完整、漂亮" |
| 成就取向（performance） | "先把结果交出去，再慢慢优化" |
| 控制偏好（high_control） | "重要决定先生成草稿，确认后再发送" |
| 控制偏好（delegatable） | "常规整理和汇总你可以直接做" |
| 风险态度（prevention） | "不要冒进，先给最稳的方案" |
| 风险态度（promotion） | "可以给我两套激进方案比较" |
| 表达组织方式 | "先结论后细节，最好有 1/2/3" |
| 关系协作方式 | "对外措辞要保守，先内部统一口径" |

#### 3.1.4 抽取原则

`profile` 只抽**稳定的协作人格维度**，不抽零碎喜好，不抽人格评价。

| 可以抽 | 不可以抽 |
|-------|-----------|
| "重要内容先确认" → 高控制感 | "用户控制欲强" → 人格评价 |
| "先满足客户要求" → 外部规训导向 | "用户比较怕担责" → 主观揣测 |
| "先交付再优化" → 结果导向 | "用户做事比较糙" → 贬义评价 |
| "希望多给备选方案" → 探索型风险偏好 | "用户性格纠结" → 人格判断 |

#### 3.1.5 提取提示词模板（LLM）

```
[任务] 从对话中提取用户稳定的协作人格维度，生成 profile 记忆。

[抽取目标]（前期粗粒度 v1，仅抽 4 维）
判断用户在以下 4 个维度上的稳定倾向，**每个维度独立判断、独立输出一条记忆**（同一对话可同时输出多条不同 dimension 的记忆）：

1. motivation（动机来源）：intrinsic | external_regulated | balanced | context_switching | unknown
2. achievement_orientation（成就取向）：mastery | performance | balanced | context_switching | unknown
3. control_preference（控制偏好）：high_control | delegatable | balanced | context_switching | unknown
4. risk_attitude（风险态度）：promotion | prevention | balanced | context_switching | unknown

取值说明：
- **两端值**（如 intrinsic / external_regulated）：用户在该维度上有明显倾向
- **balanced**：同一维度两端兼有，用户两端表现稳定且强度相近
- **context_switching**：按场景切换（reason 中必须说明触发场景）
- **unknown**：证据不足，不入库（仅作占位说明）

[判断标准]
- 只在用户明确表达或多次重复时归类，否则输出 unknown
- 优先抽"稳定工作倾向"，不要抽一次性语气
- 不强行二选一，含糊就输出 unknown
- 输出 balanced / context_switching 时 reason 必须给两端各一条原话佐证

[严禁提取]
- 人格评价（"控制欲强"、"不专业"、"玻璃心"）
- 情绪推断（"焦虑"、"烦躁"、"缺安全感"）
- 私生活与隐私
- 仅本轮有效的一次性要求

[输出格式]
{
  "memories": [
    {
      "type": "profile",
      "dimension": "motivation | achievement_orientation | control_preference | risk_attitude",
      "value": "对应取值（含 balanced / context_switching / unknown）",
      "title": "≤20 字，直接描述稳定倾向",
      "body": "中性、可执行的话。例：'更偏结果导向，优先完成交付，再逐步优化细节'",
      "confidence": 0.0-1.0,
      "reason": "归类依据 + （若 context_switching）触发场景说明",
      "evidenceQuotes": ["原话1", "原话2"],
      "tags": [...]
    }
  ]
}

[置信度评分]
- 用户明确自述 + 重复出现 → 0.9+
- 用户明确自述一次，语义稳定 → 0.8
- 仅一次模糊表达 → 输出 unknown 或 0.6（候选区）
- 需较强推断 → 不要输出
- value=unknown 的条目不入库，仅 reason 记录"该维度证据不足"

[v2 候选维度（前期不开启）]
expression_style / collaboration_style / time_pressure_threshold —— 数据稳定后再启用
```

#### 3.1.6 聚合去重规则

| 场景 | 规则 |
|------|------|
| **同维度唯一** | 同一用户同一 `dimension` 仅保留 1 条 active 结论 |
| **unknown 不入库** | value=unknown 的条目不写入仓库，仅在提取日志中记录"该维度证据不足"，等下次提取再判 |
| **balanced 处理** | value=balanced 视为稳定结论占用该维度的 active 槽位，不再强行归为某一端；evidenceQuotes 必须给两端各一条原话 |
| **context_switching 处理** | value=context_switching 占用 active 槽位，body 必须列出"场景 A → 取值 X / 场景 B → 取值 Y"；reason 必须含触发场景说明 |
| **新证据覆盖** | 当新证据更强且与旧值冲突时，标记 supersedes；旧值保留历史，不参与召回 |
| **L1 合并策略** | L1 不是简单偏好拼接，而是按"驱动方式 → 成就取向 → 控制感 → 风险偏好"顺序生成人格摘要；balanced/context_switching 在摘要中明示 |
| **总字符数控制** | L1 ≤600 字符；优先保留驱动方式、成就取向、控制感偏好 |
| **弱证据淘汰** | 仅出现一次且无后续印证的维度，30 天后自动降权 |

#### 3.1.7 反例（不该写入 profile）

```
不要写：用户今天有点急（一次性状态）
不要写：用户不懂技术（能力评价）
不要写：用户性格强势（人格判断）
不要写：这次请短一点（一次性要求）
不要写：用户老板比较严（这不是用户自身稳定画像）
```

#### 3.1.8 操作元数据

- **所有权**：用户账号（personal scope 主导，远程 identity 只提供岗位/团队归属，不提供人格判断）
- **双端分工**：远程 `identity/user.json`（岗位/团队，硬信息）+ 本地 `local/global/profile/`（协作人格维度，软画像）
- **写入触发**：用户在设置页主动填写 / chat-end 从多轮原话中归纳稳定维度 / 用户确认 agent 总结结果
- **召回方式**：必读层槽 1 注入 L1 摘要（按维度组织，约 600 字符）
- **禁止内容**：具体决策、任务上下文、资源指针、人格评价、情绪推断、能力贬抑

### 3.2 `task-context` — 我在做什么

#### 3.2.1 定位

让 agent 知道当前在哪个项目、目标是什么、走到了哪一步。

#### 3.2.2 抽取维度（基于情境认知 + 角色理论 + 目标设定理论）

| 维度 | 要回答的问题 | 两端/选项 | 对 agent 的影响 |
|------|-------------|---------|----------------|
| **目标结构** | 项目目标是什么、从哪来、多确定？ | 自设目标 ↔ 外部指派目标；明确 ↔ 模糊 | 决定推进方向与优先级 |
| **角色关系** | 用户在这个项目里是什么角色？和谁协作？ | owner / 执行者 / 参与者；汇报对象 / 协作方 | 决定沟通对象和建议口吻 |
| **约束情境** | 时间、资源、外部要求的边界在哪？ | 时间紧 ↔ 从容；资源充足 ↔ 受限；客户要求明确 ↔ 开放 | 决定方案的可行边界 |

#### 3.2.3 典型内容

| 子类 | 示例 |
|------|------|
| 项目目标 | "金融客户合规整改 Project，Q2 必须完成 ISO 27001 审计" |
| 当前阶段 | "审计准备期，剩余 3 周；上周完成数据分类，本周写控制清单" |
| 客户/合作方约束 | "客户 A 要求周五前发邮件，正文简短附件详细" |
| 关键里程碑 | "M1 风险评估（已完成）、M2 控制设计（进行中）、M3 审计预演（5-15 前）" |
| 阶段性结论 | "Q2 优先合规整改，支付模块重构推迟到 Q3"（注：含 Why 时归 experience，此处仅表示纯状态描述） |
| 范围边界 | "本次范围只覆盖客户 A，客户 B 的合规整改另立 Project" |

#### 3.2.4 提取提示词模板（LLM）

```
[任务] 从 Project 内对话中提取项目级稳定状态，按 3 个维度分类。**每个维度独立判断、独立输出一条记忆**（同一对话可同时输出多个 dimension 的记忆）。

[按维度抽取]
1. 目标结构（goal_structure）：
   - 项目目标内容（"目标是..."、"要完成..."）
   - 目标来源：自设 / 指派 / 客户要求
   - 目标确定度：明确 / 待确认 / 模糊

2. 角色关系（role_relation）：
   - 用户角色："作为..."、"我负责..."
   - 协作方："向 X 汇报"、"与 Y 协作"、"Z 是审批人"
   - 责任归属："这部分我来"、"X 负责那块"

3. 约束情境（constraint_context）：
   - 时间约束："截止..."、"剩余 N 周"
   - 资源约束："预算..."、"人力..."
   - 外部要求："客户要求..."、"合规要求..."

[严禁提取]
- 单条任务执行细节（"现在写第三段"）
- 工具操作步骤
- 临时疑问和一次性沟通
- 推断性状态（"看来应该..."）

[输出格式]
{
  "memories": [
    {
      "type": "task-context",
      "dimension": "goal_structure | role_relation | constraint_context",
      "title": "≤20 字",
      "body": "客观陈述",
      "confidence": 0.0-1.0,
      "reason": "归类逻辑解释",
      "evidenceQuotes": ["原话1", "原话2"],
      "projectId": "必填",
      "tags": [...]
    }
  ]
}
```

#### 3.2.5 聚合去重规则

| 场景 | 规则 |
|------|------|
| **subType 唯一性** | 同一 Project 同一 subType 只能有 1 条最新 active 状态（如 "phase" 只能有 1 个当前阶段） |
| **阶段切换** | 新阶段写入时，旧阶段标记 `lifecycleStatus: archived` 不删除（用于回溯） |
| **目标稳定性** | goal 类条目修改需 confidence ≥ 0.9 或用户在 Project 详情页确认 |
| **L1 合并策略** | LLM 把当前 active 的 L0 拼成一段项目状态描述，**默认非递进**（一段直白描述）；当总字符 > 1000 时改递进式：核心目标 + 当前阶段 + 关键约束在 prompt 内，里程碑细节按需下钻 |
| **总字符数控制** | L1 ≤1000 字符；超出按"目标 > 阶段 > 约束 > 里程碑 > 范围 > 历史结论"优先级裁剪 |
| **跑偏防护** | 阶段 1 提取出的 task-context 必须有明确"项目级触发词"（目标/阶段/范围/客户/里程碑），否则降级为 experience 候选 |

#### 3.2.6 反例（不该写入 task-context）

```
不要写：刚才那段代码有 bug（执行细节，归会话日志）
不要写：用户对项目进度很焦虑（情绪推断）
不要写：李总今天提到一个想法（除非确认成为项目方向）
不要写：上周三开了个会（事件日志，归会话日志；除非会议产出影响项目方向）
不要写：这个项目挺难的（主观评价）
```

#### 3.2.7 操作元数据

- **所有权**：单个 Project（project scope 主导）；短时任务对话（无 Project）不存 task-context
- **写入触发**：Project 创建时初始化 / chat-end 提取阶段感变化 / 用户在 Project 详情页编辑
- **召回方式**：必读层槽 2 注入当前 Project 的 task-context L1 摘要（约 1000 字符）
- **禁止内容**：个人偏好（归 profile）、团队规范（归 rules）、决策的"为什么"（归 experience）、执行细节、情绪推断

### 3.3 `rules` — 什么不能做

#### 3.3.1 定位

让 agent 知道哪些事不能做、哪些底线不能碰。

#### 3.3.2 抽取维度（基于心理契约 + 规范激活）

| 维度 | 要回答的问题 | 取值 | 对 agent 的影响 |
|------|-------------|------|----------------|
| **规范来源** | 这条规则从哪来？ | personal / team / project / enterprise（合规） | 决定优先级（合规 > 团队 > 项目 > 个人） |
| **约束强度** | 这条规则多硬？ | 硬约束（禁止/必须）↔ 软约束（建议/期望） | 决定是"绝对不做"还是"尽量不做" |
| **后果意识** | 违反会怎样？ | 高代价（合规/安全）/ 中（信任）/ 低（习惯偏好） | 资源紧张时决定能否突破 |

#### 3.3.3 典型内容

| 子类 | scope | 示例 |
|------|-------|------|
| 个人偏好约束 | personal | "不要自动发送，先生成草稿" |
| 项目特定约束 | project | "本 Project 客户数据禁止发外部" |
| 团队规范 | team | "周报必须含'本周完成 / 下周计划 / 风险提示'三段" |
| 企业合规 | enterprise | "客户 PII 数据禁止存入第三方云盘" |
| 资源使用边界 | personal/team | "飞书连接器只读不写"、"代码审查专家只用于核心模块" |

**优先级（合并冲突时）**：enterprise > team > project > personal

#### 3.3.4 提取提示词模板（LLM）

```
[任务] 从对话中提取持续生效的行为约束，按 3 个维度分类。**每条候选规则独立输出**（一段对话可同时输出多条 rules）。

[抽取目标]
按以下三个维度判断每条候选规则：
1. 规范来源（source）：personal / team / project / enterprise
   - personal：用户个人偏好（"我不喜欢..."）
   - team：团队约定（"我们团队..."）
   - project：项目特定（"这个 Project..."）
   - enterprise：企业合规/法规（"公司规定..."、"合规要求..."）

2. 约束强度（strength）：hard / soft
   - hard：含"必须/禁止/不能/只能/不得"等强制性词
   - soft：含"建议/最好/尽量/通常"等期望性词
   - **前期粗抽阶段：strength=soft 不入库，直接丢弃**（避免软建议污染必读层）

3. 后果意识（consequence）：high / medium / low
   - high：违反会引起合规风险、信任崩塌、安全事故
   - medium：违反会引起返工、关系紧张
   - low：违反只是不符合个人习惯

[严禁提取]
- 一次性纠错（"这次改一下" 不补充"以后..."）
- 推断的约束（用户没明说）
- 软性建议（前期 strength=soft 一律丢弃）

[输出格式]
{
  "memories": [
    {
      "type": "rules",
      "source": "personal | team | project | enterprise",
      "strength": "hard",
      "consequence": "high | medium | low",
      "title": "≤20 字，动词开头（禁止/必须/只读）",
      "body": "一句话约束",
      "confidence": 0.0-1.0,
      "reason": "归类逻辑解释",
      "evidenceQuotes": ["原话1", "原话2"],
      "relatedResourceId": "若约束某资源",
      "tags": [...]
    }
  ]
}

[置信度评分]
- 强否定/必须语气 + 明确来源 → 0.9+
- 用户在规范页主动配置 → 0.95+
- 软性建议 → 前期不输出
```

#### 3.3.5 聚合去重规则（最严格）

| 规则 | 说明 |
|------|------|
| **强去重** | 新条目 vs 已有条目 bigram Jaccard > 0.7 → 视为同条；保留更高优先级 + 更晚版本 |
| **同义合并** | "禁止发外部" 与 "客户数据不能外传"语义相近 → LLM 判断后合并为一条 |
| **总数硬上限** | 任意时刻召回的 rules ≤20 条；超限按"compliance > team > project > personal"淘汰 |
| **冲突解决** | 高优先级覆盖低优先级；同优先级用最新版本 |
| **永不淘汰** | compliance 类 rules 不参与体量淘汰（即使 >20，先淘汰其他三类） |
| **L1 合并策略** | 元数据聚合（不需 LLM）——按优先级排序后输出条目清单，每条 ≤30 字 |
| **失效检测** | 用户出现"以后改成 X" / "这条不用守了" → 标记 `lifecycleStatus: revoked`，从召回中剔除但保留历史 |

#### 3.3.6 反例（不该写入 rules）

```
不要写：用户希望文档写得好一点（软性，归 profile）
不要写：这次任务先简短点（一次性，不是持续约束）
不要写：项目挺紧的别拖（情绪化表达，不是规则）
不要写：用户提到 "可能不太好" （含糊态度）
不要写：基于法规推断的约束（必须由用户/管理员明确表达）
```

#### 3.3.7 操作元数据

- **所有权**：跨 scope，按来源分别归属
- **双端分工**：远程 enterprise/rules（合规，强制覆盖本地）/ 远程 teams/<id>/rules（团队规范，远程优先）/ 本地 projects/<id>/rules（项目，仅本地）/ 本地 global/rules（个人，仅本地）
- **写入触发**：chat-end 提取约束语句 / 用户纠正 agent 后系统提议 / 远程同步下发
- **召回方式**：必读层槽 3 全量注入（按优先级排序，约 1000 字符）
- **与 resource 的关联**：rules 通过 `relatedResourceId` 关联到具体 resource
- **禁止内容**：执行步骤（归 SKILL）、决策方法论（归 experience）、软性建议

### 3.4 `experience` — 之前怎么做过

#### 3.4.1 定位

让 agent 知道类似情况之前怎么处理过、为什么这样决定。

#### 3.4.2 抽取维度（基于自传体记忆 + 归因理论 + Kolb 学习循环）

| 维度 | 要回答的问题 | 取值 | 对 agent 的影响 |
|------|-------------|------|----------------|
| **因果链** | What + Why 齐备吗？Outcome 有更好 | 必须含 What（决定了什么）+ Why（为什么）；Outcome（结果如何）可后补 | 让 agent 能解释"为什么这样建议" |
| **迁移边界** | 这个经验适用于什么场景？ | 单次特例 / 同类项目可用 / 通用方法论 | 决定召回范围，避免错误泛化 |
| **记忆类型** | 是具体事件还是抽象方法？ | 情节型（具体一次决策）↔ 语义型（抽象的"通常这样做"） | 情节型供参考，语义型可晋升 SKILL |

> **核心约束**：缺 Why 不入库——只有"决定了什么"没有"为什么"的内容不能进 experience。Outcome 可在后续对话中补充（置信度从 0.75 升至 0.85+）。

#### 3.4.3 典型内容

| 子类 | 示例 |
|------|------|
| 关键决策依据 | "选 JWT 而非 Session，原因：支持分布式部署 + 移动端友好" |
| 决策推翻原因 | "推迟支付模块重构（4-18），原因：Q2 优先合规整改" |
| 客户/合作方处理记录 | "客户 A 偏好邮件简短附件详细（4-20 沟通确认）" |
| 跨项目方法论种子 | "周报先写业务结论再附技术细节"（≥3 项目重复时晋升 SKILL） |
| 失败教训 | "上次直接发邮件给客户没确认，引发问题；以后必须先生成草稿" |
| 领域知识沉淀 | "公司 SSO 用 OAuth 2.0，回调地址需在白名单" |

#### 3.4.4 提取提示词模板（LLM）

```
[任务] 从对话中提取可迁移的决策模式，按 3 个维度抽取。**每条候选经验独立输出**。

[抽取目标]
按以下三个维度判断每条候选经验：

1. 因果链（causality）：**what + why 必须齐备，outcome 可后补**
   - what：决定了什么（"决定/选择..."）
   - why：为什么这样决定（"因为/考虑到..."）——**缺 why 直接丢弃，不入库**
   - outcome：结果或预期影响（可后续补充，填"待观察"即可）

2. 迁移边界（transferability）：
   - case_specific：单次特例（仅当下场景有效）
   - project_pattern：本类项目可复用
   - general_methodology：通用方法论（候选 SKILL）

3. 记忆类型（memory_type）：
   - episodic（情节型）：具体一次决策，有时间/项目/角色
   - semantic（语义型）：抽象的"通常这样做"

[严禁提取]
- 缺 Why 的纯陈述（"用了 X 方法"）
- 工具调用过程
- 模糊判断（"感觉这样行"）
- 用户思考过程（"我在想..."）

[输出格式]
{
  "memories": [
    {
      "type": "experience",
      "causality": {
        "what": "决定了什么",
        "why": "为什么（必填）",
        "outcome": "结果（可空，填'待观察'）"
      },
      "transferability": "case_specific | project_pattern | general_methodology",
      "memoryType": "episodic | semantic",
      "title": "≤20 字，决策动作",
      "summary": "≤80 字：决策 + 核心依据",
      "body": "完整经验（注入时只给 summary，下钻时给 body）",
      "decisionContext": "做决定时的项目阶段/客户/约束",
      "tags": [...],
      "confidence": 0.0-1.0,
      "reason": "归类逻辑解释",
      "evidenceQuotes": ["原话1", "原话2"],
      "relatedResourceId": "若涉及具体资源",
      "knowledgeBaseRef": "若引用知识库"
    }
  ]
}

[置信度评分]
- what+why+outcome 三件套齐备 + 用户确认 → 0.9+
- what+why+outcome 三件套齐备 → 0.85
- what+why 明确但缺 outcome → 0.75（outcome 标"待观察"）
- 缺 why → 不入库
```

#### 3.4.5 聚合去重规则（递进式 + 反复用导向）

| 规则 | 说明 |
|------|------|
| **元数据聚合，不全文 LLM** | experience 数量大，L1 不做全文摘要，只维护"概要索引"（title + summary + tags + fileRef + 复用次数） |
| **概要索引格式** | `id / title / summary / subType / tags / fileRef / referenceCount / confidence / lastUsedAt` |
| **去重策略** | 同 transferability + memoryType + bigram Jaccard > 0.75 + decisionContext 相似 → 合并；保留 referenceCount 累加 |
| **跨项目识别** | 当同一 experience 在 ≥3 个项目中出现 → 标 `crossProject: true` → SKILL 候选触发 |
| **召回 top-K** | 必读层槽 5 只放 top-3~5 概要（按复合分），其余不出现在 prompt |
| **下钻路径** | agent 通过 `memory_query({fileRef})` 读取完整 body |
| **方法论种子识别** | 含"我们一般 / 通常 / 习惯先做 X 再做 Y" → 标记 `methodologySeed: true`，优先做 SKILL 候选 |
| **晋升 SKILL 后** | 原 experience 不删，标 `lifecycleStatus: promoted`（附 `promotedToSkillId: <skillId>`），召回优先级降低（避免与 SKILL 重复） |

#### 3.4.6 反例（不该写入 experience）

```
不要写：用户问了 X 问题（提问不是经验）
不要写：执行了 X 步骤完成了任务（缺 Why，是过程数据）
不要写：上次推理时考虑了 A B C（推理过程归会话日志）
不要写：用户对 X 不满意（情绪化，不是决策依据）
不要写：今天天气不好（无关）
```

#### 3.4.7 操作元数据

- **所有权**：跨 scope（personal/project/team/enterprise）
- **晋升路径**：会话候选区 → 项目 experience → 全局 experience；项目 experience → memory-review → SKILL 候选
- **写入触发**：chat-end 提取决策语句 / 用户在对话中明确解释原因后系统标记 / 关键节点（专家会话结束、技能执行完成）
- **召回方式**：必读层槽 5 注入概要索引（top-3~5，约 800 字符）；任务相关层关键词召回详情 + 决策链下钻
- **关系图谱**：experience 之间通过 `derives_from`、`supersedes`、`conflicts_with` 形成决策链
- **禁止内容**：会话内推理过程、执行日志、外部数据快照、纯陈述无 Why

### 3.5 `resource` — 有什么可用资源

#### 3.5.1 定位

让 agent 知道当前可用哪些工具、文档、专家、连接器。

#### 3.5.2 抽取维度（基于分布式认知 + 交互记忆系统 + 社会资本）

| 维度 | 要回答的问题 | 取值 | 对 agent 的影响 |
|------|-------------|------|----------------|
| **认知主体** | 这个资源是什么类型？ | 人（expert）/ 工具（skill/connector）/ 制品（kb_doc/external_system） | 决定调用什么类型的外脑 |
| **元知识** | 这个资源能回答什么？ | "谁知道什么" / "什么能做什么" / "什么记录了什么" | 让 agent 知道去哪问 |
| **信任度** | 这个资源多可信？ | high（权威/官方）/ medium（团队验证）/ low（待验证） | 影响召回优先级与是否需要交叉确认 |

#### 3.5.3 典型内容（subType 区分）

| subType | 示例 |
|---------|------|
| `kb_doc` | "周报模板：飞书文档 X（链接 + 适用场景）" |
| `external_system` | "公司 GitLab：https://git.company.com，只读权限" |
| `expert` | "代码审查专家 code-reviewer（适用：核心模块、安全敏感代码）" |
| `skill` | "周报生成 SKILL（输入：本周事项；输出：三段式周报）" |
| `connector` | "飞书 MCP 连接器（只读，可访问文档/日历，不可发消息）" |

#### 3.5.4 提取提示词模板（LLM）

```
[任务] 从对话中提取可调用资源的指针，按 3 个维度归类。**每条候选资源独立输出**。

[抽取目标]
按以下三个维度判断每个候选资源：

1. 认知主体（cognition_agent）：
   - person：人（专家/同事担当）
   - tool：工具（SKILL / connector / external_system）
   - artifact：制品（kb_doc / 文档 / 代码库）

2. 元知识（meta_knowledge）：必须三选一明确表达
   - "who_knows_what"：谁知道什么（人型）
   - "what_can_do"：什么能做什么（工具型）
   - "what_records"：什么记录了什么（制品型）

3. 信任度（trust_level）：
   - high：官方/权威来源（企业 KB、合规文档、组织指派的专家）
   - medium：团队验证/同事推荐
   - low：未验证/外部链接
   - **前期粗抽阶段：trustLevel=low 直接丢弃不入库**

[严禁提取]
- 资源的实时数据快照（"那文档现在写着 X"）
- 资源的具体使用步骤（归 SKILL）
- 资源的内容摘要（resource 只存指针）
- 临时使用一次的资源
- 信任度 low 的资源（前期粗抽直接丢弃）

[输出格式]
{
  "memories": [
    {
      "type": "resource",
      "cognitionAgent": "person | tool | artifact",
      "subType": "kb_doc | external_system | expert | skill | connector",
      "metaKnowledge": "who_knows_what | what_can_do | what_records",
      "trustLevel": "high | medium",
      "title": "≤20 字",
      "summary": "≤60 字适用场景",
      "entryPoint": "调用入口（链接/ID/路径）",
      "relatedSkillIds": ["若 tool 类，关联对应 SKILL"],
      "scope": "personal | project | team | enterprise",
      "confidence": 0.0-1.0,
      "reason": "归类逻辑解释",
      "evidenceQuotes": ["原话1", "原话2"],
      "tags": [...]
    }
  ]
}

[置信度评分]
- 用户在资源管理页主动挂载 + 描述适用场景 → 0.95+
- 用户对话中明确引用 + 适用场景 + 信任度可判 → 0.85+
- 缺适用场景或信任度未明 → 0.6（写候选区）
- 推断的资源 → 不要输出
```

#### 3.5.5 聚合去重规则

| 规则 | 说明 |
|------|------|
| **元数据聚合，不需 LLM** | resource L1 不做语义摘要，按 subType 分组直接列出"概要索引" |
| **概要索引格式** | `id / title / summary / subType / entryPoint / relatedSkillIds / scope` |
| **去重策略** | 同 entryPoint 视为同条；title bigram Jaccard > 0.8 + scope 相同视为同条 |
| **subType 分组** | L1 注入时按"专家 / 技能 / 连接器 / 知识库 / 外部系统"分组排列 |
| **失效检测** | 资源被卸载/删除时自动标记 `lifecycleStatus: revoked`；连接器停用时同步 |
| **tool 类双向关联** | resource (subType=skill) 创建/删除时同步更新对应 SKILL 的 `linkedResourceId` |
| **总字符控制** | L1 ≤600 字符；超出按"connector > expert > skill > kb_doc > external_system"优先级裁剪（高频调用类优先） |

#### 3.5.6 反例（不该写入 resource）

```
不要写：飞书文档第 3 段说 X（资源内容，不是指针）
不要写：今天发现 GitLab 上有个项目叫 Y（一次性观察）
不要写：用户也许会用到 X 工具（推断）
不要写：飞书最新版本支持 Z 功能（资源状态，归资源系统）
不要写：用户登录了 GitHub（操作日志）
```

#### 3.5.7 操作元数据

- **所有权**：跨 scope
- **双端分工**：远程 enterprise/resource（企业 KB / 工具）/ 远程 teams/<id>/resource（团队公共资源）/ 本地 projects/<id>/resource（项目）/ 本地 global/resource（个人）
- **写入触发**：用户挂载资源 / 启用专家 / 安装连接器 / chat-end 提取资源引用
- **召回方式**：必读层槽 4 注入概要索引（约 600 字符）；任务相关层按关键词召回具体使用方式
- **与 rules 的协同**：resource 描述"有什么"，rules 描述"怎么用"（通过 `relatedResourceId` 关联）
- **禁止内容**：资源实时状态、资源具体内容、使用步骤（归 SKILL）

### 3.6 5 type 之间的关系与"用户感受"全局视角

#### 3.6.1 type 间的相互引用

```
profile  ──(影响表达风格)──>  agent
task-context  ──(限定情境)──>  agent
rules  ──(划定边界)──>  agent
experience  ──(指导决策)──>  agent
resource  ──(指明工具)──>  agent

experience  ──relatedResourceId──>  resource
rules       ──relatedResourceId──>  resource
experience  ──knowledgeBaseRef──>  resource (subType=kb_doc)
experience  ──derives_from──>  experience（决策链）
rules       ──supersedes──>  rules（规则更新）
```

#### 3.6.2 提取触发的优先级（避免重复入库）

当一段对话同时命中多个 type 时，按以下优先级提取（前者优先，避免重复）：

```
rules（强否定/必须语气）
  > profile（稳定协作人格倾向）
  > experience（决策 + 原因）
  > resource（资源指针 + 适用场景）
  > task-context（项目级状态变化）
```

**排序逻辑**：刚性约束 > 稳定画像 > 决策模式 > 资源指针 > 项目状态。越刚性的内容越先"吃掉"该段对话的提取额度，避免软概念把硬约束稀释。

**消歧规则（常见冲突场景）**：

| 冲突场景 | 判定方式 | 归属 |
|---------|---------|------|
| "以后请..." + **风格描述**（口吻/长度/详略/先结论后细节） | 描述的是 agent 输出风格 | → `profile` |
| "以后请..." + **禁止/必须语气**（不能/必须/只能/不得） | 描述的是行为边界 | → `rules` |
| "以后请..." + **模糊**（两种都沾） | 优先 rules（宁当约束不当画像） | → `rules` |
| 状态变更 + **无 Why**（"Q2 优先合规，支付推迟"） | 只描述当前状态/范围 | → `task-context` |
| 状态变更 + **有 Why**（"推迟支付，因为 Q2 优先合规"） | 含明确决策原因 | → `experience` |
| 单纯"我们用 X 工具"（无决策原因） | 纯资源指针 | → `resource` |
| "我们用 X 工具，因为 Y"（含原因） | 同时出 resource + experience（不冲突，各归各） | → 两条 |

#### 3.6.3 用户感受的全局校验清单

每条记忆写入前，先过一遍"用户感受四问"：

| 问题 | 不通过的处理 |
|------|------------|
| 1. 用户看到这条会觉得"被理解"还是"被审视"？ | 被审视 → 丢弃或脱敏 |
| 2. 这条对未来 agent 任务有用吗？ | 否 → 丢弃 |
| 3. 这条用一句话能说清吗？ | 否 → 拆条或丢弃 |
| 4. 这条会让用户觉得"被监控"吗？ | 是 → 丢弃 |

任一问题不通过，记忆不入库。这是写入前的最后一道闸门。

#### 3.6.4 "宁可漏，不可错"的工程化体现

| 措施 | 体现 |
|------|------|
| confidence < 0.7 直接丢弃 | 宁可不记，不冒险 |
| confidence 0.7-0.9 写候选区 | 给一次"被验证"的机会 |
| confidence ≥ 0.9 才直写共享区 | 高确定性才进 agent 视野 |
| L1 摘要总字符严格上限 | 防止"求全责备"式过度记忆 |
| rules ≤20 条硬上限 | 防止"凡事都立规矩"式越界 |
| 用户可一键清空任意 type | 给"反悔权" |
| 任何记忆可下钻到原始会话 | 给"可解释" |

## 四、type 与 scope 的二维矩阵

每个 type 在不同 scope 下都有内容。下表说明双端如何协作：

| Type | personal | project | team | enterprise |
|------|----------|---------|------|-----------|
| **profile** | ◉ 偏好/习惯（本地） | — | — | ◉ 岗位/团队（远程 identity） |
| **task-context** | — | ◉ 项目目标/阶段（本地） | — | — |
| **rules** | ◉ 个人约束（本地） | ◉ 项目约束（本地） | ◉ 团队规范（远程） | ◉ 合规约束（远程） |
| **experience** | ◉ 个人方法论种子（本地） | ◉ 项目决策（本地） | ◉ 团队 SKILL 来源（远程） | ◉ 企业最佳实践（远程） |
| **resource** | ◉ 个人工具（本地） | ◉ 项目资源（本地） | ◉ 团队资源（远程） | ◉ 企业知识库（远程） |

**关键约束**：
- profile：personal 来源本地，岗位归属来源远程
- task-context：纯本地，无远程对应
- rules：本地 + 远程都有，远程优先级更高
- experience：本地原始 + 远程沉淀，agent 看本地 SKILL 不看远程 experience 原文
- resource：本地 + 远程都有，按 scope 分层

## 五、容器分层（与产品形态对应）

容器层与 type 是正交两个维度。容器决定记忆"放在哪个空间"，type 决定记忆"回答哪个问题"。

### 5.1 六个容器

| 容器 | 主要承载的 type | 生命周期 | 双端归属 | 路径 |
|------|---------------|---------|---------|------|
| **全局** | profile + 个人 rules + 个人 resource | 跨会话、跨项目持久化 | APP 本地 | `local/global/` |
| **项目** | task-context + 项目 rules + 项目 experience + 项目 resource | Project 生命周期 | APP 本地 | `local/projects/<id>/` |
| **会话候选区** | 待晋升的 5 type 候选 | 会话期 + 7 天归档 | APP 本地 | `local/_sessions/<sid>/` |
| **自动化** | task-context + 关联 experience | 任务生命周期 | APP 本地 | `local/automation/<taskId>/` |
| **团队** | 团队 rules + 团队 experience + 团队 resource | 团队存续期 | 后端同步 | `remote/teams/<teamId>/` |
| **企业** | 企业 rules + 企业 resource | 企业存续期 | 后端同步 | `remote/enterprise/` |

### 5.2 容器内目录结构（按 type 分组）

每个容器内部按 5 type 分目录，便于按问题召回：

```
local/global/
  ├── profile/
  │   ├── nodes/
  │   │   ├── L0/   (单条偏好/习惯)
  │   │   └── L1/   (画像摘要)
  │   └── _index.json
  ├── rules/
  │   ├── nodes/
  │   └── _index.json
  └── resource/
      ├── nodes/
      └── _index.json

local/projects/<id>/
  ├── task-context/
  │   ├── nodes/
  │   │   ├── L0/   (项目目标/阶段/客户约束)
  │   │   └── L1/   (项目状态摘要)
  │   └── _index.json
  ├── rules/
  ├── experience/
  │   ├── nodes/
  │   ├── edges/    (决策链关系)
  │   └── _index.json
  └── resource/

local/_sessions/<sid>/
  ├── candidates/   (待晋升的 5 type 候选)
  └── _archived/

remote/enterprise/
  ├── rules/    (合规约束，只读)
  └── resource/    (企业知识库指针)

remote/teams/<teamId>/
  ├── rules/    (团队规范)
  ├── experience/  (团队最佳实践 - 已晋升 SKILL 来源)
  └── resource/    (团队公共资源)

remote/identity/
  └── user.json   (岗位、团队归属)
```

### 5.3 流动规则

```
本地纵向晋升（用户审核）：
  会话候选区 -> 项目容器 -> 全局容器
  （会话 rules 候选 -> 项目 rules -> 全局 rules）

横向贡献（非 agent 通道）：
  本地 experience -> memory-review -> SKILL 候选 -> 用户分享 -> 团队 SKILL

远程下发（单向同步）：
  企业 rules/resource -> 拉到本地 remote/ 缓存
  团队 rules/experience/resource -> 拉到本地 remote/ 缓存

向下继承（自动注入）：
  全局 profile/rules -> 项目内会话注入
  项目 task-context/rules/experience/resource -> 项目内会话注入
  远程 rules/resource -> 必读层注入
```

## 六、二维记忆模型：容器 × 折叠层

容器与折叠层正交。每个容器内部都有 L0/L1（部分远程容器还有 L2）：

```
┌──────────────────────────────────────────────────────────────────────┐
│                        容器层（6 个空间）                              │
├──────┬──────────┬──────────┬──────────┬──────────┬──────────────────┤
│ 全局  │ 项目      │ 会话      │ 自动化     │ 团队      │ 企业              │
├──────┴──────────┴──────────┴──────────┴──────────┴──────────────────┤
│                        type 层（5 type 体系）                          │
│        profile / task-context / rules / experience / resource     │
├──────────────────────────────────────────────────────────────────────┤
│                        折叠层（粒度控制）                                │
│  L2 综合（仅远程，月度）                                                  │
│  L1 摘要（按 type 分组聚合）   ← agent 必读层注入                          │
│  L0 原始（事实/决策/偏好）     ← 任务相关层下钻                            │
└──────────────────────────────────────────────────────────────────────┘
```

### 6.1 L0 / L1 / L2 职责

| 层 | 含义 | 数量级 | 谁生成 | 何时生成 |
|---|---|---|---|---|
| **L0 原始** | 单条事实/偏好/决策/约束/资源指针 | 海量（不限） | AI 实时观察 + chat-end 提取 + 用户手动 | 写入触发时立即生成 |
| **L1 摘要** | 按 type 分组聚合的摘要 | 受控（每容器每 type 约 1-3 个） | APP 本地（元数据聚合）/ 后端（LLM 语义摘要） | bucket 满 / TTL 到期 |
| **L2 综合** | 跨 type 的月度综合 | 极少（仅团队/企业） | 后端定时任务 | 每月 |

### 6.2 关键约束（可逆性）

1. L0 永久保留：摘要生成后 L0 不删除、不归档
2. L1 反向链接：`sourceNodeIds` 永远指向具体 L0
3. 下钻可达：用户/AI 看到 L1 时可下钻到任意 L0
4. L1 可重生：L0 变化后 L1 重新生成（version+1），旧 L1 归档

### 6.3 L1 摘要的生成方式（按 type 差异化）

> **重要变更（v0.6-rev7）**：原"时间/主题/容器三视角"摘要被废弃。L1 摘要按 **type 分组**生成，每个 type 一份 L1，对应必读层一个槽位。每个 type 有不同的总结策略，详见第 3 节各 type 的"聚合去重规则"。

| Type | 生成方式 | 总结策略 | 字符上限 |
|------|---------|---------|---------|
| `profile` | LLM 合并 | 软+硬画像合并为一段画像 | 600 |
| `task-context` | LLM 合并 | 默认非递进；超长时递进式 | 1000 |
| `rules` | 元数据聚合（不需 LLM） | 优先级排序 + 强去重 ≤20 条 | 1000 |
| `experience` | 元数据聚合（不需 LLM） | top-K 概要索引 + 文件下钻 | 800 |
| `resource` | 元数据聚合（不需 LLM） | subType 分组概要索引 + 文件下钻 | 600 |

每个 type 的 L1 直接对应必读层一个槽位，注入逻辑非常简单。

### 6.4 TTL Flush 机制

| 触发条件 | 含义 | 阈值方向 |
|---|---|---|
| **Bucket 满** | 容器内某 type 的 L0 节点数达到阈值 | profile 5 / task-context 10 / rules 15 / experience 20 / resource 10 |
| **TTL 到期** | 距上次摘要生成超过时间窗口 | profile 每月 / task-context 每周 / rules 实时（变更触发）/ experience 每周 / resource 实时 |
| **变更触发** | 关键 L0 变更（如新增合规） | rules / resource 立即重生 |

具体阈值在 P0 实施阶段确定。

## 七、会话过程数据移出记忆系统

> **重要变更（v0.6-rev7）**：reasoning / tool-chain / todo / execution-log 等"过程数据"不再属于记忆系统。

### 7.1 移出的数据类型

| 类型 | 旧归属 | 新归属 | 理由 |
|------|-------|-------|------|
| `task` 节点（会话任务目标） | 会话记忆 | gateway 会话日志 | 会话内有效，结束后归 task-context（如晋升） |
| `reasoning` 节点（中间推理） | 会话记忆 | gateway 会话日志 | 推理过程不是长期记忆 |
| `tool-chain` 节点（工具调用链） | 会话记忆 | gateway 会话日志 + execution log | 执行痕迹不应永久存储 |
| `todo` 节点（待办列表） | 会话记忆 | `todoManager.ts` 模块级 reactive state | 会话内 UI 状态 |
| `execution-log` 节点 | 自动化记忆 | 自动化模块独立日志 | 与记忆系统隔离 |
| `anomaly` 节点 | 自动化记忆 | 自动化模块独立日志 + 可选晋升 rules | 异常约束晋升后归 rules |
| `tuning` 节点 | 自动化记忆 | 自动化模块配置（不是记忆） | 配置数据 |
| `trigger` 节点 | 自动化记忆 | 自动化模块配置 | 配置数据 |

### 7.2 边界判断标准

判断一条数据是否进记忆系统：

```
对未来 agent 任务有用？ -> 是 -> 进记忆系统（按 5 type 分类）
                       -> 否 -> 不进记忆系统

是会话内临时状态？ -> 是 -> 不进记忆系统
                  -> 否 -> 继续判断

是过程痕迹（推理/执行/日志）？ -> 是 -> 不进记忆系统
                            -> 否 -> 继续判断

可以归类到 5 type 之一？ -> 是 -> 进记忆系统
                       -> 否 -> 设计盲区，重新审视
```

### 7.3 移出后如何关联

会话日志和记忆系统通过 `sessionId` 关联：

- 记忆 frontmatter 含 `sourceSessionId`
- agent 通过工具可从记忆下钻到原始会话日志（用于复盘）
- 会话日志不参与 system prompt 注入

## 八、与现状的差异

### 8.1 当前 v0.3-v0.4 与新 5 type 体系的对应

| 当前 type | 含义 | 新 type | 拆分规则 |
|-----------|------|---------|---------|
| `user` | 用户身份偏好 | `profile` | 全部归 profile |
| `feedback` | 反馈约束/偏好 | `profile` + `rules` | 偏好语句归 profile，约束语句归 rules |
| `project` | 项目目标/决策 | `task-context` + `experience` | 目标/阶段归 task-context，决策依据归 experience |
| `reference` | 外部系统指针 | `resource` | 全部归 resource，扩展 subType 字段 |

### 8.2 v0.6 早期设计中过度细分的 type（已废弃）

`decision` / `artifact` / `domain-knowledge` / `task` / `reasoning` / `tool-chain` / `todo` / `execution-log` / `anomaly` / `tuning` / `trigger` / `expert` / `skill` / `connector` / `preference` / `constraint` 全部废弃。

### 8.3 迁移方案（P0 阶段）

```
迁移规则：
1. 现有 user 类型 -> profile（直接迁移）
2. 现有 feedback 类型：
   - 含"喜欢/偏好/习惯"关键词 -> profile
   - 含"不要/禁止/必须/约束"关键词 -> rules
   - 模糊条目 -> 用户审核后分配
3. 现有 project 类型：
   - 含"目标/阶段/范围/客户约束"关键词 -> task-context
   - 含"决定/选择/原因/因为"关键词 -> experience
   - 模糊条目 -> 用户审核后分配
4. 现有 reference 类型 -> resource（扩展 subType）

迁移过程：
- 提供迁移脚本，支持 dry-run 预览
- 保留原文件作为 _archived_legacy/，可回滚
- 用户可对模糊条目逐条审核
```

## 九、关键设计决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | 记忆类型组织方式 | 5 type 体系（按 5 个本质问题） | 直接服务 agent 需要回答的问题 |
| 2 | 节点细分粒度 | 仅 5 个核心 type，不再扩展 | 过度细分导致召回时无法组装 |
| 3 | 会话过程数据 | 移出记忆系统 | reasoning/tool-chain 不是长期记忆 |
| 4 | L1 摘要生成方式 | 按 type 分组，不按"时间/主题/容器"三视角 | 直接对应必读层 5 槽位 |
| 5 | 自动化记忆隔离 | 完全隔离 | 不污染短时任务和 Project |
| 6 | Identity 处理 | 完全移除（profile 归用户账号 + 远程 identity） | Identity 已下架 |
| 7 | 容器与 type 关系 | 正交两维度 | 容器决定空间，type 决定问题 |
| 8 | 提取逻辑（v0.6-rev8） | 社会心理学导向：宁可漏不可错 | 错记的伤害 >> 漏记的损失 |
| 9 | 用户感受校验（v0.6-rev8） | 写入前过"四问校验" | 把"被理解不被审视"工程化 |
