## 一、什么是远程记忆

### 1.1 定义

**远程记忆**：banto 记忆体系中的部分或全部数据，存储在 banto 桌面应用之外的位置（云端 / 团队服务器 / 跨工具共享存储），并能被多个 agent 或用户读取/写入。

与之对应的是 **本地记忆**：v0.6 设计的主体——存储在 `~/.iflymate/memory/` 的 md+json 工作图谱。

### 1.2 远程记忆的三种典型形态

| 形态 | 含义 | 参考方案 |
|---|---|---|
| **跨智能体共享** | 同一用户的多个 AI 工具（banto / Claude Code / Cursor 等）共享同一份记忆 | OpenHuman `agentmemory` 后端 |
| **跨用户协作** | 一个团队的多个用户共享某个 Project 的记忆 | banto 企业版协作场景 |
| **企业级中央存储** | 整个组织的记忆汇总到企业服务器，统一治理 | banto 企业部署场景 |

### 1.3 与企业知识库的区别

远程记忆 ≠ 企业知识库（详见 [04 文档 7.5 节](./04-coupling-with-capability-system.md#75-知识库与记忆体系的联动关系v06-rev2-新增)）：

| 维度 | 企业知识库 | 远程记忆 |
|---|---|---|
| 内容性质 | 用户/组织维护的**事实** | AI 工作中沉淀的**判断与上下文** |
| 由谁主导维护 | 用户主动 | AI 自动观察 + 系统沉淀 |
| 在 banto 中的角色 | 通过 `reference` 类型节点引用 | 是 banto 记忆本身的"远程位置" |
| 跨工具/跨用户 | 已有方案（讯飞星火 KB） | 需要新设计 |

**关键区分**：知识库是 banto **引用**的外部资源，远程记忆是 banto 记忆**自己的远程存储**。两者职责不冲突。

## 二、为什么 v0.6 暂不实施

### 2.1 优先级判断

v0.6 的核心交付目标是**完成 banto 单机记忆体系的工作进化系统**——把记忆从被动记录升级为主动进化（三重职能）。这本身已是大规模升级。

远程记忆需要解决的问题（数据同步、冲突解决、权限边界、隐私保护）与 v0.6 的核心问题正交，**强行同时推进会让两件事都做不好**。

### 2.2 v0.6 不实施远程记忆的具体原因

| 原因 | 含义 |
|---|---|
| **设计哲学还在演化** | v0.6 的"三重职能 + 二维记忆模型"本身需要在实践中迭代，远程化前应先稳定本地形态 |
| **数据所有权边界未定** | 跨工具共享时谁拥有数据？跨用户协作时谁能看？需要先在企业场景验证 |
| **冲突解决机制缺失** | 多端写入时如何合并？OpenHuman 的 `agentmemory` 也未公开这部分 |
| **隐私与合规风险** | 把工作上下文上传到远程，可能违反企业内网部署的合规要求 |
| **性能与离线** | 远程访问引入延迟，离线场景下记忆体系不能停摆 |

### 2.3 但必须为未来留好扩展点

v0.6 的设计**必须**考虑未来远程化的可能性。具体在以下几点保持兼容：

- **md+json 双存格式**：本身就是可移植的标准格式，未来可上传到任何远程存储
- **节点 ID 全局唯一**：`mem_{date}_{hash}` 格式即使跨实例也不冲突
- **关系 ID 全局唯一**：`edge_{date}_{hash}` 同样
- **frontmatter 携带溯源**：`createdBy`、`source`、`sessionId` 等元数据让"哪个实例写的"可识别
- **置信度与热度透明**：未来合并多端记忆时，可以用复合分排序去重

## 三、未来扩展的可能形态

### 3.1 形态 A：跨智能体共享（OpenHuman `agentmemory` 模式）

**含义**：用户在 banto 中沉淀的记忆，可被同一台机器上的其他 AI 工具（Claude Code、Cursor、Codex 等）读写。

**参考方案**：OpenHuman 通过 `agentmemory` 后端实现共享存储（参见 [GitHub: rohitg00/agentmemory](https://github.com/rohitg00/agentmemory)）。

**对 banto 的可能落点**：

```
~/.iflymate/memory/        ← banto 主存储
       ↓ 双向同步
~/.agentmemory/            ← 跨工具共享存储（v0.7+）
       ↑ 读写
其他 AI 工具（Claude Code 等）
```

**关键挑战**：

- 数据格式标准化（banto 的工作图谱 vs 其他工具的格式）
- 写入冲突处理
- 记忆类型语义对齐（banto 的 `feedback` 在 Claude Code 中是什么？）

### 3.2 形态 B：跨用户协作（banto 企业版）

**含义**：一个团队的多个用户在同一个 Project 上工作，Project 记忆共享。

**典型场景**：

- 销售团队共同维护"客户 A 的历史决策与偏好"
- 研发团队共同维护"项目架构演化与技术债"

**可能的设计方向**：

```
团队服务器
└── shared-memory/
    └── projects/
        └── proj_xyz/
            ├── L0/             ← 团队共享的原始记忆
            ├── L1/             ← 团队共享的摘要
            └── permissions/    ← 谁能读写哪些
```

**关键挑战**：

- 私人偏好 vs 团队共识的边界（用户偏好不应共享）
- 团队成员各自的 `feedback` 是否合并？（建议保持各自所有权）
- 实时同步还是周期同步？

### 3.3 形态 C：企业级中央存储

**含义**：整个组织的所有 banto 实例的记忆汇总到企业服务器，用于治理、审计、知识沉淀。

**典型场景**：

- 企业合规审计："过去一年所有 AI 决策的依据"
- 组织知识沉淀："新员工继承前任的工作上下文"
- 治理与脱敏："敏感记忆不落本地，全部上云"

**关键挑战**：

- 大规模数据存储与检索性能
- 与现有企业知识库的边界（避免重复）
- 用户隐私与合规边界（哪些可上传，哪些必须本地）

## 四、远程记忆与现有知识库能力的关系

> **核心定位**：远程记忆能力 **由后续知识库能力扩展提供**，不另起炉灶。

### 4.1 思路

banto 当前已有的知识库能力（`KnowledgeScope` + `kb_get_context` MCP 工具）已经对接了**外部知识库**的协议。未来远程记忆能力可以借助同一套知识库基础设施扩展：

```
banto 现有能力：
- kb_get_context  → 拉取外部知识库内容（用户维护的事实）

未来扩展（v0.7+）：
- kb_save_memory  → 把 banto 记忆推送到远程存储
- kb_query_memory → 从远程拉取属于本用户/团队的记忆
- kb_sync_state   → 双向同步状态管理
```

这样做的好处：

- ✅ 复用现有 MCP 工具体系，不引入新协议
- ✅ 知识库 UI 可统一管理"事实"和"远程记忆"
- ✅ 企业部署时，远程记忆与知识库走同一套审计/合规链路

### 4.2 边界澄清

| 场景 | 用知识库还是用远程记忆 |
|---|---|
| 用户主动维护的合规文档 | 知识库 |
| AI 沉淀的工作偏好 | 本地记忆（v0.6） |
| 跨工具共享的工作偏好 | 远程记忆（v0.7+，借助知识库基础设施） |
| 团队共享的项目决策 | 远程记忆（v0.7+） |
| 团队共享的产品规范 | 知识库（团队主动维护） |

### 4.3 建议演进路径

```
v0.6（当前）：
  本地记忆完整 + 知识库引用机制（reference 类型节点）

v0.7：
  借助知识库基础设施扩展"远程记忆 v1"
  - 形态 A 优先（跨工具共享，单用户）
  - 形态 B 试点（企业版团队协作）

v0.8+：
  形态 C（企业级中央存储与治理）
```

## 五、v0.6 设计中需要保留的扩展点

为了让 v0.7+ 远程记忆顺利演进，v0.6 设计中以下决策**必须**保持：

| v0.6 决策 | 为什么不能改 |
|---|---|
| **md+json 双存格式** | 标准格式可移植到任何远程存储 |
| **节点 ID 全局唯一** | 跨实例合并时不冲突 |
| **frontmatter 完整溯源** | 远程合并时能识别"哪个实例写的" |
| **L0 永不删除（折叠层）** | 远程同步时保证数据完整性 |
| **关系独立 edges/ 目录** | 远程同步时关系层可单独传输 |
| **Reference 节点 `knowledgeBaseRef` 字段** | 这是知识库与记忆的桥梁，未来扩展为远程记忆指针的基础 |
