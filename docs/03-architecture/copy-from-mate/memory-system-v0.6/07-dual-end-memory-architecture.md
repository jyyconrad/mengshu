# 双端记忆架构设计（修正版）

> **状态**：设计中
> **版本**：v0.6-rev7
> **日期**：2026-06-01
> **前置依赖**：[00-current-implementation-analysis.md](./00-current-implementation-analysis.md)、[06-p0-implementation-plan.md](./06-p0-implementation-plan.md)

---

## 一、核心原则

1. **agent 对双端无感知**：agent 只调用本地 `memory-recall`，本地负责合并双端结果
2. **远程是只读的"权威源"**：本地不能直接修改远程记忆，远程由后端独立治理
3. **同步是单向的（远程 → 本地）**：本地维护一份"远程记忆缓存"，定期更新
4. **本地→远程的贡献走非 agent 通道**：用户通过 SKILL 晋升机制贡献、管理员后台录入、后端从对话日志中聚合（不是 agent 实时推送）
5. **数据结构统一**：本地原生记忆 + 远程同步缓存 共用同一套 schema（5 type 体系）
6. **记忆→SKILL→团队**：本地 experience 的团队贡献路径是"模式识别 → 晋升为 SKILL → 分享到团队"，而非直接上传原始记忆
7. **5 type 在双端各司其职**：profile/task-context 主要在本地；rules/experience/resource 双端共存，远程优先级更高

## 一·二、5 type 在双端的 scope 分布

> **重要约束（v0.6-rev7）**：5 type 体系下，每个 type 在双端的归属和职责有明确分工。

### type × scope × 端 映射

| Type | personal（本地） | project（本地） | team（远程同步） | enterprise（远程同步） | 服务的本质问题 |
|------|---------------|---------------|----------------|---------------------|--------------|
| **profile** | ◉ 偏好/习惯 | — | — | ◉ 岗位/团队归属（identity） | Q1 我为谁工作 |
| **task-context** | — | ◉ 项目目标/阶段 | — | — | Q2 我在做什么 |
| **rules** | ◉ 个人约束 | ◉ 项目约束 | ◉ 团队规范 | ◉ 合规约束 | Q3 什么不能做 |
| **experience** | ◉ 个人方法论种子 | ◉ 项目决策 | ◉ 团队 SKILL 来源 | ◉ 企业最佳实践 | Q4 之前怎么做过 |
| **resource** | ◉ 个人工具 | ◉ 项目资源 | ◉ 团队资源 | ◉ 企业知识库 | Q5 有什么可用资源 |

### 关键设计

- `profile`：personal 来源本地（用户编辑），岗位/团队归属来源远程 identity，agent 注入时合并
- `task-context`：纯本地，无远程对应；自动化任务的 task-context 也归本地
- `rules`：双端共存，**远程优先级更高**（合规强制覆盖、规范优先于本地）
- `experience`：本地原始 experience 经 memory-review 识别后晋升为 SKILL → 分享到团队/企业；远程 experience 是已经过审核的"团队最佳实践种子"
- `resource`：双端按 scope 分层，远程提供"权威资源指针"（企业知识库根入口），本地提供"个人/项目专属资源"

## 三、远程记忆的来源（不由 agent 上传）

### 3.1 三种合法来源

| 来源 | 谁触发 | 通道 |
|------|--------|------|
| **管理员录入** | 企业管理员 | 后端管理界面（不经过 APP） |
| **用户主动贡献** | 个人用户 | APP UI 点"分享到团队"按钮 → HTTP 接口（非 agent 工具） |
| **系统自动聚合** | 后端定时任务 | 从全员对话日志/用户行为中聚合，需用户授权 + 脱敏 |

### 3.2 为什么 agent 不能主动上传

| 风险 | 说明 |
|------|------|
| **隐私泄露** | agent 无法判断哪些是个人隐私（健康、家庭、敏感财务） |
| **质量失控** | agent 提取的记忆 confidence 不稳定，污染团队共识 |
| **滥用风险** | agent 频繁推送可能被用作信息窃取通道 |
| **审计不可控** | agent 自主上传的记忆无法溯源到用户操作 |

### 3.3 用户贡献入口：SKILL 晋升机制

**核心思路**：用户不直接上传原始记忆，而是通过**将本地记忆晋升为 SKILL** 后分享到团队。SKILL 是结构化、可复用、经过提炼的知识单元，比原始记忆更适合团队共享。

**晋升流程**：

```
本地工作记忆（local/ 目录）
   ↓
内置 skill「工作记忆回顾」识别可晋升模式（见 3.4 节）
   ↓
输出"SKILL 候选"，存入本地 SKILL 草稿区
   ↓
用户在 SKILL 管理页查看候选，编辑/确认
   ↓
用户点"分享到团队"按钮 + 选择目标团队
   ↓
HTTP 请求到后端审核队列
   ↓
后端 LLM 自动审核（脱敏 + 质量评估 + 格式校验）
   ↓
团队管理员审批（可选，按团队配置）
   ↓
入库为团队 SKILL
   ↓
下次同步时下发到团队成员本地 SKILL 库
```

**入口位置**：设置 → 技能 → SKILL 管理页 → 候选 Tab → 单条 SKILL 右侧"分享到团队"

**与原始记忆直接上传的区别**：

| 维度 | 直接上传记忆（已废弃） | SKILL 晋升（当前方案） |
|------|---------------------|---------------------|
| 共享物 | 原始记忆节点 | 结构化 SKILL（含标题/触发条件/执行步骤） |
| 质量保障 | 依赖后端 LLM 审核 | 系统预筛 + 用户编辑 + 后端审核（三层） |
| 隐私风险 | 高（原始记忆含上下文） | 低（SKILL 是提炼后的规则，不含具体上下文） |
| 可操作性 | 记忆只能读 | SKILL 可被 agent 直接调用执行 |
| 团队价值 | 参考性信息 | 可执行的最佳实践 |

**注意**：这是 UI 流程，**不是 agent 工具**。agent 在 system prompt 中无法触发此操作。

### 3.4 内置 Skill：工作记忆回顾与 SKILL 生成

> 这是"本地记忆 → SKILL → 远程团队"链条的**起点**，由 banto 内置提供，不需要用户手动安装。

#### 3.4.1 Skill 定位

| 属性 | 值 |
|------|-----|
| **名称** | `memory-review`（工作记忆回顾） |
| **类型** | 内置 skill（banto 自带，不可删除） |
| **触发模式** | `auto`（系统定时触发）+ `slash`（用户手动 `/memory-review`） |
| **输入** | 本地工作记忆（`local/` 目录全量） |
| **输出** | SKILL 候选列表（存入本地 SKILL 草稿区） |
| **执行频率** | 自动模式：每周一次（周末空闲时段）；手动模式：用户随时触发 |

#### 3.4.2 核心能力

**能力 A：记忆模式识别**

扫描 `local/` 目录下所有记忆节点，识别可晋升为 SKILL 的重复模式：

```
识别规则：

1. 跨项目重复模式
   - 同一 feedback 类型记忆在 ≥3 个项目中出现
   - 同一工具调用序列在 ≥3 个 session 中重现
   - 同一决策模式在 ≥2 个项目中被引用

2. 高置信度规范
   - confidence ≥ 0.85 的 feedback/reference 类型
   - 含"规范/规则/必须/禁止/流程/步骤"等关键词
   - 被 agent 主动 recall 次数 ≥ 5

3. 方法论沉淀
   - 含"框架/方法/原则/模板"关键词
   - 内容结构化程度高（含列表/步骤/条件判断）
   - 跨时间跨度 ≥ 2 周仍被引用
```

**能力 B：SKILL 候选生成**

将识别到的模式转化为结构化 SKILL 草稿：

```yaml
# SKILL 候选输出格式
---
title: "服务命名规范"
trigger: auto
category: coding-standard
confidence: 0.92
source_memories:
  - id: "mem_abc123"
    project: "项目A"
  - id: "mem_def456"
    project: "项目B"
  - id: "mem_ghi789"
    project: "项目C"
status: draft          # draft → confirmed → shared
---

## 触发条件
当用户创建新服务或讨论命名时自动激活

## 执行规则
1. 服务名使用 kebab-case 格式
2. 前缀为业务域缩写（如 user-、order-）
3. 禁止使用下划线和驼峰

## 来源说明
基于 3 个项目中的 5 次重复使用记录自动生成
```

**能力 C：持续优化已有 SKILL**

在后续对话中，当 agent 使用了某个已确认的 SKILL 并产生新的反馈时：

```
优化触发条件：
1. 用户在对话中修正了 SKILL 的某条规则
2. 新的记忆与已有 SKILL 相关但补充了新场景
3. SKILL 被 agent 调用后用户给出负面反馈

优化动作：
- 生成 SKILL 更新建议（diff 形式）
- 通知用户确认更新
- 更新后版本号 +1，保留历史版本
```

#### 3.4.3 执行流程

```
┌─────────────────────────────────────────────────────────────┐
│  内置 Skill: memory-review                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────┐    ┌──────────────┐    ┌─────────────────┐    │
│  │ 定时触发 │───→│ 扫描 local/  │───→│ 模式识别引擎    │    │
```

#### 3.4.4 与其他模块的关系

| 模块 | 关系 |
|------|------|

## 四、本地→远程候选清单（基于 banto 定位）

### 4.1 banto 定位回顾

banto 是**职场效率工具**，三类工作容器：
- **短时任务对话**：临时性问答与小任务
- **Project**：长期任务容器（写作、研发、分析）
- **自动化**：定时任务

支撑体系：专家智能体 / 技能 / 连接器 / 知识库。

### 4.2 升级模式：四档分级

由于 **agent 不能上传**（见第三节），所谓"自动同步"只能是"系统自动识别 + 通知用户 + 一键确认"，与"用户主动发起"是不同的决策强度。本地→远程的升级路径分为四档：

| 档位 | 决策强度 | 触发方 | 用户介入 | 适用场景 |
|------|---------|-------|---------|---------|
| **L1 静默自动** | 系统判定 | 系统 | 无 | 仅限"管理员从后端下发的全员豁免内容"（极少使用） |
| **L2 系统提议 + 一键确认** | 系统判定 + 用户授权 | 系统识别候选，**通知中心提示** | 一键"是" / "否" / "永不再问" | 高置信度 + 隐私敏感度低 + 团队价值明确 |
| **L3 用户主动发起** | 用户判定 | 用户在 UI 点"分享到团队" | 填写分享理由 + 选择 scope | 中等置信度 + 用户认为有共享价值 |
| **L4 管理员录入** | 管理员判定 | 管理员后台 | 直接录入 | 合规约束、企业规范、客户主数据 |
| **禁止上传** | — | — | — | 个人偏好、敏感字段、临时上下文 |

**关键约束**：
- 所有档位最终都不由 agent 触发
- L2 的"一键确认"是用户操作，不是 agent 调用
- L1 在 v0.6 范围内**不启用**（保留架构口子，避免静默上传引发信任问题）

每类候选的升级路径、判定条件、目标 scope：

| # | 候选类型 | 示例 | 升级档位 | 自动判定条件 | 目标 scope |
|---|---------|------|---------|-------------|-----------|
| **A1** | 命名/编码规范 | "服务命名用 kebab-case" | **L2** | 跨 ≥3 项目引用 + confidence ≥ 0.85 + 含"规范/规则/必须/禁止"关键词 + 不含人名 | 团队 |
| **A2** | 文档/产出规范 | "API 文档必须含示例" | **L2** | 同 A1 | 团队 |
| **A3** | 流程约束 | "合并前必过 review" | **L2** | 同 A1 | 团队 |
| **A4** | 安全合规 | "客户数据不外发" | **L4** | — | 企业 |
| **A5** | 工具使用规范 | "Jira issue 必填 Resolution" | **L2** | 跨 ≥2 项目 + confidence ≥ 0.85 + 关联连接器 ID | 团队 |
| **B1** | 写作模板 | "周报先写结论后细节" | **L3** | — | 团队 |

| **C2** | 客户约束 | "客户 A 要求 ISO 27001" | **L3** | — | 团队 |

### 4.4 L2 自动判定的统一阈值

L2 档位需要系统自动判定"是否提议给用户"，判定逻辑：

```
候选条件（同时满足才进入提议队列）：

1. 类型匹配：记忆 type ∈ {feedback, project, reference}
   （user 类型直接排除，因其多为个人偏好）

2. 跨项目复用：
   - A1/A2/A3：referenceCount ≥ 3 且 distinctProjects ≥ 3
   - A5/B3：referenceCount ≥ 3 且 distinctProjects ≥ 2
   - D1/D2：referenceCount ≥ 2 且 distinctSessions ≥ 3

3. 置信度门槛：confidence ≥ 0.85

4. 隐私安全（必须全部满足）：
   - 不含人名（启发式：中文姓 + 1-2 字、英文 First Last 模式）
   - 不含联系方式（手机号/邮箱/IP/URL token）
   - 不含敏感字段关键词（密码、token、密钥、cookie、secret）
   - 不含主观评价（"难搞"、"水平差"、"不专业"等）

5. 关键词命中（按候选类型匹配）：
   - A1/A2/A3：含"规范/规则/必须/禁止/不得/约束/原则"
   - A5：tags 含连接器名 + 含上述规范关键词
   - B3：含"流程/方法/步骤/原则/框架"
   - D1/D2：tags 含专家 ID 或技能 ID

6. 用户偏好排除：
   - 用户曾对同 name 的提议选择"永不再问" → 跳过
   - 用户当前未授权"团队记忆贡献" → 跳过整个 L2 流程
```

**满足以上 6 项 → 进入提议队列；否则不出现在通知中心。**

### 4.5 通知中心设计

通知中心是记忆系统与用户之间的**唯一主动触达通道**，承载所有需要用户感知或决策的记忆相关事件。

#### 4.5.1 职责边界

| 职责 | 说明 | 不属于通知中心的 |
|------|------|----------------|
| L2 SKILL 晋升提议 | 系统识别的 SKILL 候选，请求用户确认分享 | agent 对话中的实时反馈 |
| 远程记忆变更通知 | 团队/企业记忆有重要更新（如合规约束变更） | 静默同步的增量更新 |
| SKILL 候选提醒 | memory-review 生成了新的 SKILL 候选 | SKILL 的日常执行日志 |
| 冲突提示 | 远程记忆与本地记忆产生冲突需人工裁决 | 自动覆盖的合规类冲突 |
| 贡献状态反馈 | 用户分享的 SKILL 审核通过/拒绝 | 后端内部审核流程 |

**不做的事**：
- 不推送 agent 对话过程中的记忆提取通知（那是后台静默操作）
- 不推送每次远程同步的详情（只推重要变更）
- 不推送其他用户的活动（不是社交动态流）

#### 4.5.2 通知类型分类

| 类型 | 优先级 | 触发源 | 示例 |
|------|--------|--------|------|
| **合规变更** | P0（高） | 远程同步检测到 enterprise/compliance 变更 | "企业新增合规约束：客户数据禁止存储到第三方云盘" |
| **冲突裁决** | P1（中高） | 远程同步与本地记忆冲突且无法自动解决 | "团队规范'API 文档格式'与你的本地版本冲突，请选择保留哪个" |
| **SKILL 候选** | P2（中） | memory-review 内置 skill 输出 | "发现 2 个可晋升为 SKILL 的工作模式，请查看" |
| **L2 晋升提议** | P2（中） | L2 自动判定通过 | "3 条 SKILL 可能对团队有价值，是否分享？" |
| **贡献反馈** | P3（低） | 后端审核完成 | "你分享的'服务命名规范' SKILL 已通过审核，已入库团队" |
| **远程重要更新** | P3（低） | 远程同步检测到团队记忆重大变更 | "团队新增 5 条最佳实践 SKILL，已同步到本地" |

#### 4.5.3 聚合策略

```
聚合规则（避免信息轰炸）：

实时推送（不聚合）：
  - P0 合规变更 → 立即推送，红色标记
  - P1 冲突裁决 → 立即推送，需用户操作才消失

每周聚合（周一上午）：
  - P2 SKILL 候选 + L2 晋升提议 → 合并为一条摘要通知
  - 单次最多展示 5 条候选，超出留到下周
  - 如果本周无新候选，不发送空通知

按事件聚合（发生时）：
  - P3 贡献反馈 → 审核完成时推送，同一天多条合并
  - P3 远程更新 → 同步完成后推送，按"新增 N 条"聚合

静默期：
  - 用户正在对话中（isRunning=true）→ 所有 P2/P3 延迟到对话结束
  - 22:00-08:00 → 非 P0 通知延迟到次日 08:00
```

#### 4.5.4 用户交互模式

**单条操作**：

| 操作 | 适用类型 | 效果 |
|------|---------|------|
| **确认** | L2 提议 / SKILL 候选 | 触发分享流程（走 POST /memory/contribute） |
| **拒绝** | L2 提议 / SKILL 候选 | 标记为已拒绝，30 天内不再提议同名内容 |
| **查看详情** | 所有类型 | 展开完整内容 + 来源记忆列表 |
| **永不再问** | L2 提议 | 永久屏蔽该条目（可在设置中恢复） |
| **选择保留** | 冲突裁决 | 选择本地版本或远程版本 |
| **已读** | 远程更新 / 贡献反馈 | 标记为已读，从未读列表移除 |

**批量操作**：

| 操作 | 说明 |
|------|------|
| **全部确认** | 一键确认当前所有 L2 提议（需二次确认弹窗） |
| **全部已读** | 清除所有 P3 通知的未读标记 |
| **稍后再说** | 关闭通知面板，下次打开时仍显示 |

**设置级控制**：

| 设置项 | 默认值 | 说明 |
|--------|--------|------|
| 启用 L2 提议 | 开 | 关闭后不再收到任何 SKILL 晋升提议 |
| 启用 SKILL 候选提醒 | 开 | 关闭后 memory-review 仍运行但不通知 |
| 启用远程变更通知 | 开 | 关闭后仅保留 P0 合规变更（不可关闭） |
| 聚合频率 | 每周 | 可选：每周 / 每两周 / 每月 |
| 静默时段 | 22:00-08:00 | 可自定义 |

#### 4.5.5 克制通知原则

通知中心的设计哲学是**宁可漏推，不可扰人**：

1. **频率克制**：P2/P3 通知每周最多聚合推送一次，不实时打扰
2. **数量克制**：单次最多展示 5 条候选，超出排队到下周
3. **重复克制**：用户拒绝过的同名内容 30 天内不再提议
4. **场景克制**：用户对话中不弹出任何非 P0 通知
5. **可关闭**：除 P0 合规变更外，所有通知类型均可在设置中关闭
6. **渐进信任**：新用户前 2 周不推送 L2 提议，先积累足够记忆数据
7. **无红点焦虑**：P3 通知不显示角标数字，仅在打开通知面板时可见

### 4.6 不上传清单（明确禁止）

以下内容**任何档位都不允许上传**，包括 L3 用户主动发起也会被前端拦截：

| 类型 | 示例 | 拦截方式 |
|------|------|---------|
| 个人偏好 | "我喜欢简洁回答" | 类型为 `user` 的记忆 → 前端按钮置灰 |
| 个人身份信息 | "我住北京" | 含 PII 关键词 → 前端拦截 |
| 临时会话上下文 | "刚才说的 bug 修了" | scope=session + 未晋升 → 前端拦截 |
| 敏感字段 | "API token 是 xxx" | 正则匹配密码/token/密钥 → 前端拦截 |
| AI 失败/学习 | "上次输出格式错了" | source=auto-correction → 前端隐藏 |
| 个人项目细节 | "我家装修预算" | scope=personal 且 tags 含生活类 → 前端拦截 |
| 未脱敏的人名 | "张三说要这样改" | 包含中文姓+名 / 英文 First Last → 前端高亮提示用户脱敏 |
| 情绪/评价 | "客户 A 难搞" | 主观情绪关键词 → 前端拦截 |

**双层保护**：
- 第一层：前端按上述规则拦截
- 第二层：后端 `/memory/contribute` 接口 LLM 审核再次拦截（防止前端绕过）

## 五、后端需具备的记忆能力

> 这里只列**后端内部能力**，不是对 agent 暴露的 API。agent 只调本地 `memory-recall`。

### 5.1 五项核心能力

#### 能力 1：记忆存储与组织

| 子能力 | 说明 |
|--------|------|
| 多 scope 存储 | 个人/项目/团队/企业四级 scope |
| 同构数据模型 | 与本地共用一套 schema（详见第七节） |
| 版本管理 | 每条记忆有 version + updatedAt + 历史快照 |
| 软删除 | 标记 deleted 而非物理删除，支持回溯 |

**实现方式**：关系数据库（Postgres）存元数据 + 对象存储（OSS）存内容 + 向量库（Milvus/PgVector）存向量。

#### 能力 2：记忆树折叠（L0 → L1 → L2）

| 子能力 | 说明 |
|--------|------|
| L0 原始记忆 | 永不删除，承载事实层 |
| L1 主题/时间摘要 | 后端 LLM 服务每周折叠 |
| L2 跨主题综合 | 每月折叠 |
| 增量更新 | 新增 L0 时只重新折叠相关 L1 |

**实现方式**：定时任务（每周一凌晨）触发 LLM 服务，按 scope + 主题分组生成摘要。摘要节点的 `sourceNodeIds` 反向链回 L0。

#### 能力 3：决策链追溯

| 子能力 | 说明 |
|--------|------|
| 关系图谱存储 | 节点 + 5 类关系（derives_from/depends_on/...） |
| 链路查询 | 给定决策节点，向上追溯依据 |
| 链路注入 | 同步到本地时随决策节点一起下发上下文链 |

**实现方式**：图数据库（Neo4j）或关系表（adjacency list）。关系建立由两种方式：
- 后端 LLM 服务在折叠 L1 时识别"决策→依据"模式，自动建立关系
- 管理员在后端 UI 手动建立关系

#### 能力 4：知识融合（与企业知识库联动）

| 子能力 | 说明 |
|--------|------|
| 记忆引用知识库 | 记忆节点 `knowledgeBaseRef` 字段指向知识库文档 |
| 知识库内容嵌入记忆 | 折叠 L1 时可引用知识库片段 |
| 知识库更新触发记忆刷新 | 知识库文档变更 → 引用它的记忆标记"待复核" |

**实现方式**：复用现有 `KnowledgeScope`/`kb_get_context` 基础设施，记忆与知识库通过 `chunkId` 关联。

#### 能力 5：同步分发（远程 → 本地）

| 子能力 | 说明 |
|--------|------|
| 全量快照 | 用户登录时下发当前 scope 下所有可见记忆 |
| 增量推送 | 定时（每小时）下发自上次同步后的变更 |
| 变更通知 | 重要记忆变更（如合规约束更新）主动推送 |
| 权限过滤 | 按用户所属团队/企业过滤可见范围 |

**实现方式**：HTTP 长轮询 / WebSocket / Server-Sent Events，本地维护 `lastSyncAt` 游标。

### 5.2 后端能力对外接口（统一）

后端只对 APP 暴露**两类内部接口**（不是 agent 工具）：

| 接口 | 调用方 | 用途 |
|------|--------|------|
| `GET /memory/sync/snapshot` | APP 主进程（启动时） | 拉取全量快照 |
| `GET /memory/sync/incremental?since=<ts>` | APP 主进程（定时） | 拉取增量变更 |

**注意**：
- 这两个接口由 APP 主进程调用，不是 agent 通过 MCP 调用
- 调用结果写入本地 `~/.iflymate/memory/remote/`，agent 通过 `memory-recall` 间接读到
- 用户贡献走另一个独立接口 `POST /memory/contribute`（UI 触发，非 agent 工具）

## 六、远程→本地同步机制

### 6.1 同步时机

| 时机 | 拉取方式 | 频率 |
|------|----------|------|
| **用户登录** | 全量快照 | 一次 |
| **APP 启动后台同步** | 增量 | 启动后 30s |
| **定时同步** | 增量 | 每 1 小时 |
| **后端主动推送** | 变更通知 | 重要记忆变更时 |
| **手动同步** | 增量 | 用户在 UI 点"立即同步" |

### 6.2 本地存储位置

> **重要变更（v0.6-rev7）**：本地目录按 **5 type 分目录**组织，每个容器内部都有 5 个 type 子目录。

```
~/.iflymate/memory/
├── local/                              # 本地原生记忆（用户 + AI 维护）
│   ├── global/
│   │   ├── profile/                    # Q1 用户画像
│   │   ├── rules/                   # Q3 个人约束
│   │   └── resource/                   # Q5 个人资源
│   ├── projects/<projectId>/
│   │   ├── task-context/               # Q2 项目上下文
│   │   ├── rules/                   # Q3 项目约束
│   │   ├── experience/                 # Q4 项目决策
│   │   └── resource/                   # Q5 项目资源
│   ├── automation/<taskId>/
│   │   ├── task-context/               # 自动化任务上下文
│   │   └── experience/                 # 自动化历史经验
│   └── _sessions/<sid>/                # 会话候选区（待晋升）
│       ├── candidates/                 # 候选 5 type 节点
│       └── _archived/                  # 7 天后归档
└── remote/                             # 远程记忆缓存（只读）
    ├── _meta.json                      # 同步元信息（lastSyncAt / cursor）
    ├── identity/                       # Q1 岗位/团队归属
    │   └── user.json
    ├── enterprise/                     # 企业级记忆
    │   ├── rules/                   # Q3 合规约束
    │   └── resource/                   # Q5 企业知识库入口
    └── teams/<teamId>/                 # 团队级记忆
        ├── rules/                   # Q3 团队规范
        ├── experience/                 # Q4 团队最佳实践（已晋升 SKILL 来源）
        └── resource/                   # Q5 团队公共资源
```

**关键约束**：
- 每个目录下记忆节点按 type 同质（不混类型）
- 容器（global/projects/.../teams/.../enterprise）与 type 是正交两维度
- agent 通过 memory-recall 不感知目录结构，只看到组装后的 5 槽位

### 6.3 本地覆盖策略

下拉远程记忆时，按 **type + scope** 决定冲突处理：

| Type | 远程 scope | 冲突时策略 | 理由 |
|------|-----------|-----------|------|
| `rules` | enterprise | **远程强制覆盖** | 合规优先，个人不能违反 |
| `rules` | team | **远程优先 + 本地保留副本** | 团队共识优先，但保留本地以备审计 |
| `experience` | team | **远程追加，本地不覆盖** | 团队最佳实践只读，本地私有决策保留 |
| `experience` | enterprise | **远程追加，本地不覆盖** | 同上 |
| `resource` | enterprise | **远程优先** | 企业知识库入口由后端权威 |
| `resource` | team | **远程优先 + 本地副本** | 团队公共资源 |
| `profile` | enterprise（identity） | **远程强制覆盖**（仅岗位/团队字段） | 后端是权威源 |
| `profile` | personal | **本地优先** | 远程不应干涉个人偏好 |
| `task-context` | — | 无远程对应 | 纯本地 |

### 6.4 冲突检测与处理

```
同步流程伪代码：

for 远程记忆 R in 增量推送结果:
  本地查找同 type + 同 name + 同 scope 的记忆 L
  
  case 1: L 不存在
    → 直接写入 remote/<type>/ 目录，不动 local/
  
  case 2: L 存在但在 local/ 目录（本地原生）
    → 按"本地覆盖策略表"判断：
      - 远程强制覆盖：在 local/ 中标记 L 为 superseded，写入 remote/
      - 远程优先 + 副本：写入 remote/，本地 L 标记 conflictWith: R.id
      - 本地优先：仅写入 remote/，本地 L 不动
  
  case 3: L 存在且在 remote/ 目录（上次同步的缓存）
    → 比较 R.version vs L.version：
      - R.version > L.version：覆盖
      - R.version == L.version：跳过
      - R.version < L.version：异常，记录日志
  
  case 4: L 在 remote/ 但 R 标记 deleted
    → 物理删除 L
```

### 6.5 注入时的合并策略：5 槽位组装

> **重要变更（v0.6-rev7）**：`memory-recall` 不再按"必读层 + 任务相关层 + 按需检索层"裸三层模型，而是**按 5 槽位组装**，每槽对应一个本质问题。

```
召回流程：

必读层（4000 字符预算，5 槽位结构）：

  [槽 1] 我为谁工作（profile L1，~600 字符）
    合并：本地 local/global/profile/ + 远程 identity/user.json
    
  [槽 2] 我在做什么（task-context L1，~1000 字符）
    来源：本地 local/projects/<id>/task-context/（仅 Project 内会话）
    
  [槽 3] 什么不能做（rules 全量，~1000 字符）
    优先级合并：
      1. 远程 enterprise/rules/  ← 合规最高
      2. 远程 teams/<myTeam>/rules/ ← 团队规范
      3. 本地 local/projects/<id>/rules/  ← 项目约束
      4. 本地 local/global/rules/  ← 个人偏好
    
  [槽 4] 有什么资源（resource L1，~600 字符）
    合并：
      - 远程 enterprise/resource/  ← 企业 KB 入口
      - 远程 teams/<myTeam>/resource/  ← 团队公共
      - 本地 local/projects/<id>/resource/  ← 项目相关
      - 本地 local/global/resource/  ← 个人常用
    
  [槽 5] 高频历史经验（experience top-3，~800 字符）
    合并：
      - 本地 local/projects/<id>/experience/  ← 项目优先
      - 本地 local/global/experience/  ← 全局补充
      （远程 team experience 不直接召回，已晋升 SKILL）
    复合分排序：score = α × hotness + β × confidence + γ × task_relevance + δ × user_signal

任务相关层（3000 字符预算）：
  - 任务相关 rules 细节（关键词 + relatedResourceId 匹配）
  - 任务相关 experience 详情 + 决策链下钻（derives_from）
  - 任务相关 resource 具体使用方式（按 subType 过滤）

按需检索层（agent 通过工具调用）：
  - agent 调用 memory_query(type, scope, keywords, traverse)
  - 工具内部合并双端搜索，对 agent 透明
```

详见 [03-runtime-mechanisms.md 第五节](./03-runtime-mechanisms.md)。

## 七、统一数据结构与接口

### 7.1 同构 Schema（5 type 体系）

本地原生记忆 + 远程同步缓存 + 后端存储 **共用一套 schema**：

```typescript
// 5 type 体系 + summary（L1/L2 摘要）
type MemoryType =
  | 'profile'        // Q1 我为谁工作
  | 'task-context'   // Q2 我在做什么
  | 'rules'       // Q3 什么不能做
  | 'experience'     // Q4 之前怎么做过
  | 'resource'       // Q5 有什么可用资源
  | 'summary';       // L1/L2 摘要节点

// resource 节点的子类型
type ResourceSubType =
  | 'kb_doc'         // 知识库文档
  | 'external_system'// 外部系统位置
  | 'expert'         // 专家智能体
  | 'skill'          // 技能
  | 'connector';     // 连接器

interface MemoryNode {
  // ── 标识 ──────────────────────────
  id: string;                  // 全局唯一
  level: 'L0' | 'L1' | 'L2';   // 折叠层
  type: MemoryType;            // 5 type + summary
  subType?: ResourceSubType;   // 仅 type=resource 时使用
  
  // ── 归属 ──────────────────────────
  scope: 'personal' | 'project' | 'team' | 'enterprise';
  scopeId?: string;            // 项目/团队/企业 ID
  origin: 'local' | 'remote';  // 来源（本地原生 vs 远程同步）
  ownerId: string;             // 创建者用户 ID
  
  // ── 内容 ──────────────────────────
  content: {
    title: string;
    body: string;
    tags: string[];
  };
  
  // ── 关联 ──────────────────────────
  relatedResourceId?: string;  // rules/experience 关联到 resource 节点
  knowledgeBaseRef?: {         // experience 引用知识库
    repoId: string;
    chunkIds: string[];
    lastValidatedAt: string;
  };
  
  // ── 元数据 ────────────────────────
  metadata: {
    confidence: number;
    source: string;            // chat-end-extraction/manual/admin/aggregation
    sourceSessionId?: string;  // 来源会话 ID（可下钻到会话日志）
    createdAt: string;
    updatedAt: string;
    version: number;
    promotedToSkill?: {        // 仅 experience 升格后
      skillId: string;
      promotedAt: string;
    };
    syncedAt?: string;         // 远程同步缓存独有
  };
  
  // ── 统计 ──────────────────────────
  stats: {
    referenceCount: number;
    lastAccessedAt?: string;
  };
  
  // ── 关系（仅 L0/L1） ───────────────
  relations?: {
    outgoing: Array<{ type: RelationType; targetId: string; weight: number }>;
    incoming: Array<{ type: RelationType; sourceId: string; weight: number }>;
  };
  
  // ── L1/L2 摘要专属 ───────────────────
  subjectType?: MemoryType;    // L1 摘要的目标 type（如 task-context L1 摘要的 subjectType=task-context）
  sourceNodeIds?: string[];    // L1/L2 → 反向链回 L0
  trigger?: { type: string; value: string };
}

type RelationType =
  | 'derives_from'    // experience 依据关系（决策链）
  | 'supersedes'      // 推翻旧决策
  | 'conflicts_with'  // 冲突
  | 'depends_on'      // 依赖
  | 'references';     // 引用
```

**关键设计**：
- `type` 字段强制 5 type + summary，不允许其他值
- `subType` 仅 type=resource 时使用
- `relatedResourceId` 替代旧的 `relatedExpertIds` / `relatedSkillIds` / `sourceConnector` 等多字段
- `subjectType` 让 L1 摘要明确"自己摘要的是哪个 type"
- `origin` 字段区分本地原生 vs 远程同步缓存
- `scope` 字段统一表达归属
- 远程记忆下拉到本地后，schema 完全一致，agent 无感知

### 7.2 后端对外接口（仅 3 个，非 agent 工具）

```http
# 1. 全量快照（登录时调用）
GET /api/v1/memory/sync/snapshot
Headers: Authorization, X-Device-Id
Response: {
  memories: MemoryNode[],
  edges: MemoryEdge[],
  syncCursor: string,
  expiresAt: string
}

# 2. 增量同步（定时调用）
GET /api/v1/memory/sync/incremental?since=<cursor>
Response: {
  added: MemoryNode[],
  updated: MemoryNode[],
  deleted: string[],     // ID 列表
  edges: { added: [], updated: [], deleted: [] },
  newCursor: string
}

# 3. 用户贡献（UI 触发，非 agent）
POST /api/v1/memory/contribute
Body: {
  memory: MemoryNode,    // 本地选中的记忆
  shareReason: string,   // 用户填写的分享理由
  targetScope: 'team' | 'enterprise',
  targetScopeId: string
}
Response: {
  contributionId: string,
  status: 'pending-review' | 'auto-approved'
}
```

**约束**：
- 这 3 个接口由 APP **主进程**调用（不是 agent / MCP）
- agent 调用 `memory-recall` 时看到的是同构数据，不知道哪些来自远程
- 用户贡献走 UI → 主进程 → HTTP，**不暴露给 agent**

### 7.3 本地 memory-recall 模块的职责

```typescript
class MemoryRecall {
  // 唯一对 agent 暴露的查询入口
  recall(params: {
    query: string;
    types?: MemoryType[];
    limit?: number;
  }): Promise<MemoryNode[]> {
    // 1. 加载本地原生记忆
    const localResults = await this.loadLocal();
    
    // 2. 加载远程同步缓存
    const remoteResults = await this.loadRemote();
    
    // 3. 合并 + 复合分排序
    const merged = this.merge(localResults, remoteResults);
    
    // 4. 按 limit 截断
    return merged.slice(0, params.limit ?? 10);
  }
  
  // 仅 APP 主进程使用，不暴露给 agent
  syncFromRemote(): Promise<void> { ... }
  contributeToRemote(memory: MemoryNode, reason: string): Promise<void> { ... }
}
```

## 八、实施路径（修正版）

| 阶段 | 范围 | 依赖后端 |
|------|------|---------|
| **P0** | 本地原生记忆完整能力（recall + 分层召回 + Session 晋升 + 元数据摘要） | ❌ 不依赖 |
| **P1** | 本地 `remote/` 缓存目录 + memory-recall 合并双端 | ❌ 不依赖（用空缓存兜底） |
| **P2** | 后端实现：能力 1（存储）+ 能力 5（同步分发） | ✅ 依赖后端 |
| **P3** | APP 实现：登录拉全量 + 定时拉增量 + 本地覆盖策略 | ✅ 依赖 P2 |
| **P4** | 用户贡献 UI + 后端审核流程 | ✅ 依赖后端 |
| **P5** | 后端实现：能力 2（记忆树）+ 能力 3（决策链）+ 能力 4（知识融合） | ✅ 依赖后端 |

**关键节点**：
- P0+P1 完全在 APP 端，可独立完成（远程缓存为空也不影响本地能力）
- P2 之后才需要后端配合
- P5 是后端独立工作，APP 端无需改动（schema 已统一）

## 九、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| agent 是否能调用上传/搜索远程 | **不能** | 隐私、质量、滥用风险 |
| 远程数据来源 | 管理员录入 + 用户主动贡献 + 后端聚合 | 三个独立通道，质量可控 |
| 同步方向 | 远程 → 本地（单向） | 本地不能修改远程，远程是权威源 |
| 本地覆盖策略 | 按记忆类型差异化 | 合规强制 / 规范优先 / 个人偏好不动 |
| Schema | 本地远程同构 | agent 无感知，便于演进 |
| 用户贡献通道 | 独立 HTTP 接口（UI 触发） | 与 agent 隔离，可审核 |
| 候选信息范围 | 通用规范 / 方法论 / 决策依据 / 工具模式 | 团队/企业有共享价值 |
