# banto 记忆系统架构设计

> 日期：2026-06-02
> 状态：设计中（v0.6-rev8）
> 范围：定义 banto/iFlyMate 记忆系统的完整架构，指引后续实施
> 子文档：[v0.6 细化方案](./memory-system-v0.6/README.md)

---

## 1. 一句话结论

banto 的记忆系统是服务"短时任务对话 / Project"主线的**工作进化系统**。它的核心使命是在 agent 启动任务时，回答 5 个本质问题，让 agent 知道"该怎么做"。

### 1.1 5 个本质问题（设计主线）

| # | 问题 | 召回什么 | 决定 agent 的什么 |
|---|------|---------|-----------------|
| Q1 | **我为谁工作？** | 用户画像与偏好 | 输出风格、详细度、表达方式 |
| Q2 | **我在做什么？** | 任务上下文 | 目标、优先级、阶段感 |
| Q3 | **什么不能做？** | 行为约束边界 | 行动边界、合规底线 |
| Q4 | **之前怎么做过？** | 历史经验 | 执行模式、决策依据 |
| Q5 | **有什么可用资源？** | 资源指针 | 调用哪些工具/文档 |

整套记忆系统的所有设计——存储分层、节点类型、召回策略、注入方式、晋升机制——都围绕回答这 5 个问题展开。**任何不能回答这 5 个问题之一的设计都是噪音。**

### 1.2 工作进化三重职能

5 个本质问题服务三重职能：

| 职能 | 服务的问题 | 对应能力 |
|------|----------|---------|
| **做好当下** | Q1-Q5 全部 | 记忆写入、召回、上下文注入 |
| **沉淀能力** | Q4 历史经验 → SKILL 库 | 模式识别、SKILL 候选生成、升格审核 |
| **优化未来** | 基于 Q4 主动诊断 | 工作模式分析、流程优化建议 |

## 2. 产品定位

### 2.1 服务主线

banto/iFlyMate 的一级概念：

- **短时任务对话**：用户快速开始一件事。
- **Project**：持续推进和沉淀一件事。

支撑体系：专家智能体 / 技能 / 连接器 / 知识库。

补充边界：
- **自动化/定时任务**不作为独立记忆主线。它本质上是一次由系统触发的对话执行，复用全局记忆 + Project 记忆；执行过程归会话日志/执行日志，只有被提炼出的长期约束、经验、资源或任务上下文才按 5 type 写入。
- **Identity** 在产品设计上已经移除，不作为一级产品对象或独立记忆 scope；但岗位、团队归属、默认工作角色等身份信息可以作为 `profile` / `task-context` 召回时的重要输入维度，用来重现用户画像的信息模型。

### 2.2 核心价值

记忆系统的价值不是"存更多"，而是消除以下重复成本：

- 用户反复解释自己是谁、偏好什么（Q1）
- 用户反复说明当前 Project 的背景（Q2）
- 用户反复纠正 banto 的行为方式（Q3）
- banto 在长对话中丢失本次任务已形成的判断（Q2 + Q4）
- 团队/企业重复沉淀相同的规范和最佳实践（Q3 + Q4）
- agent 不知道有哪些工具、文档、专家可用（Q5）

## 3. 双端架构总览

记忆体系采用 **APP 本地 + 后端远程** 双端架构，遵循 6 条核心原则：

| # | 原则 | 含义 |
|---|------|------|
| 1 | agent 对双端无感知 | agent 只调用本地 `memory-recall`，本地负责合并双端结果 |
| 2 | 远程是只读的"权威源" | 本地不能直接修改远程记忆，远程由后端独立治理 |
| 3 | 同步是单向的（远程 -> 本地） | 本地维护一份"远程记忆缓存"，定期更新 |
| 4 | 本地->远程的贡献走非 agent 通道 | 用户通过 SKILL 晋升机制贡献，管理员后台录入，后端从日志聚合 |
| 5 | 数据结构统一 | 本地原生记忆 + 远程同步缓存共用同一套 MemoryNode schema |
| 6 | 记忆->SKILL->团队 | 贡献路径是"晋升"而非直接上传原始记忆 |

### 3.1 架构示意

```
┌─────────────────────────────────────────────────────────────────┐
│                      Banto 记忆体系                              │
├─────────────────────────────────────────┬───────────────────────┤
│  APP 本地（用户 + AI 实时维护）           │  后端（管理员 + 系统维护） │
├─────────────────────────────────────────┼───────────────────────┤
│ ┌─────────────────────────────────────┐ │ ┌───────────────────┐ │
│ │  原生本地记忆                        │ │ │  企业/团队记忆     │ │
│ │  ~/.iflymate/memory/local/          │ │ │  数据库 + 向量库   │ │
│ │  - profile（Q1 用户画像）            │ │ │  - rules       │ │
│ │  - task-context（Q2 任务上下文）     │ │ │    （Q3 合规规范）│ │
│ │  - rules（Q3 个人约束）           │ │ │  - experience     │ │
│ │  - experience（Q4 历史经验）         │ │ │    （Q4 跨用户决策）│ │
│ │  - resource（Q5 本地资源指针）       │ │ │  - resource       │ │
│ │  （Extractor 自动提取 + 用户手动添加）│ │ │    （Q5 团队 SKILL）│ │
│ └─────────────────────────────────────┘ │ └─────────┬─────────┘ │
│ ┌─────────────────────────────────────┐ │           │           │
│ │  远程记忆缓存（只读）                │◄┼───────────┘           │
│ │  ~/.iflymate/memory/remote/         │ │  固定同步              │
│ │  - rules/RULE.md（合规约束）  │ │  - 登录拉全量          │
│ │  - experience/<teamId>/<RULE.md/SPEC.md>（团队规范+决策）  │ │  - 定时拉增量          │
│ │  - profile-inputs/（岗位/团队归属） │ │  - 后端推送变更        │
│ └─────────────────────────────────────┘ │                       │
│ ┌─────────────────────────────────────┐ │                       │
│ │  memory-recall（统一入口）            │ │                       │
│ │  按 Q1-Q5 5 槽位组装注入              │ │                       │
│ └─────────────────────────────────────┘ │                       │
└─────────────────────────────────────────┴───────────────────────┘

非 agent 通道（本地 -> 远程贡献）：
  用户在 SKILL 管理页点"分享到团队" -> HTTP 接口 -> 后端审核 -> 入库
  管理员在后端管理界面录入 -> 入库
  后端从全员对话日志中聚合 -> 入库
```

### 3.2 后端五项核心能力

1. 记忆存储与组织（多 scope、版本管理、软删除）
2. 记忆树折叠（L0 -> L1 -> L2，后端 LLM 定时折叠）
3. 决策链追溯（节点 + 关系图谱查询）
4. 知识融合（与企业知识库联动）
5. 同步分发（全量快照 + 增量推送 + 变更通知）

详见 [07-dual-end-memory-architecture.md 第五节](./memory-system-v0.6/07-dual-end-memory-architecture.md)。

## 4. 记忆类型：5 type 体系

记忆类型直接对应 5 个本质问题。每个 type 回答一个问题，不交叉、不冗余。

### 4.1 5 个核心 type

| Type | 回答问题 | 内容 | 典型示例 |
|------|---------|------|---------|
| **`profile`** | Q1 我为谁工作 | 用户身份、岗位、偏好、表达习惯、沟通方式 | "用户是合规项目经理，输出偏好先结论后细节，500 字以内" |
| **`task-context`** | Q2 我在做什么 | 项目目标、当前阶段、客户/合作方约束、关键里程碑 | "金融客户合规整改 Project，审计准备期，剩余 3 周" |
| **`rules`** | Q3 什么不能做 | 个人偏好约束、团队规范、企业合规、专家/技能/连接器边界 | "客户数据不能发外部"、"周报必须含三段式"、"飞书连接器只读" |
| **`experience`** | Q4 之前怎么做过 | 关键决策依据、领域知识沉淀、跨项目方法论的种子 | "选 JWT 而非 Session 因支持分布式部署"、"客户 A 偏好邮件简短附件详细" |
| **`resource`** | Q5 有什么可用资源 | 知识库文档指针、外部系统位置、专家/技能/连接器适用边界 | "周报模板：飞书文档 X"、"代码审查找 code-reviewer 专家"、"合规清单在企业 KB" |

### 4.2 5 type 的存储与召回策略

每个 type 的存储方式、召回方式、总结策略各不相同：

| Type | 存储方式 | 召回方式 | 总结/控制策略 |
|------|---------|---------|-------------|
| **`profile`** | 完整内容存储 | **全文导入** prompt | 总结时控制完整字符数上限（~600 字符） |
| **`task-context`** | 完整内容存储 | **全文导入** prompt | 递进式编写（一般不建议，防止任务跑偏）；遇到必须写很长时才递进 |
| **`rules`** | 类型 + 条目编写 | **全量条目导入** prompt | 总结时自动去重；最多保留 15-20 条 |
| **`experience`** | Markdown 主文件（frontmatter 元数据 + 正文） | **只注入概要索引**；复杂详情通过附录索引下钻 | 复杂详情才递进式拆附录 |
| **`resource`** | Markdown 主文件（frontmatter 元数据 + 正文） | **只注入概要索引**；复杂详情通过附录索引下钻；tool 类型关联 SKILL | 复杂详情才递进式拆附录 |

#### profile 策略详解

- **存储**：完整内容，一份文件承载用户画像全貌
- **召回**：全文导入 system prompt（槽 1），不做摘要裁剪
- **总结**：chat-end 阶段 1.5 生成 L1 时，控制总字符数不超过 600 字符；新信息合并进已有 profile 而非追加新条目
- **写入规则**：新提取的 profile 信息与已有内容合并（更新），不产生多条 profile 节点

#### task-context 策略详解

- **存储**：完整内容，一份文件承载当前 Project 的完整上下文
- **召回**：全文导入 system prompt（槽 2），不做摘要裁剪
- **总结**：默认完整写入；当内容确实很长时采用递进式编写（先写核心目标+当前阶段，后续按需追加里程碑/客户约束等细节）
- **注意**：一般不建议递进式，因为 task-context 不完整会导致 agent 任务跑偏
- **写入规则**：每个 Project 只有一份 task-context，新信息覆盖更新（保留版本历史）

#### rules 策略详解

- **存储**：类型标签 + 条目列表，每条 rules 是一个独立条目
- **召回**：全量条目导入 system prompt（槽 3），按优先级排序（合规 > 团队 > 项目 > 个人）
- **总结**：
  - 自动去重：相同语义的 rules 合并（bigram Jaccard > 0.8）
  - 条目上限：最多保留 15-20 条有效 rules
  - 超出时按优先级淘汰（个人偏好类先淘汰，合规类永不淘汰）
- **写入规则**：每条 rules 独立存储，带 scope 标签和优先级

#### experience 策略详解

- **存储**：每条 experience 是一个 Markdown 主文件，frontmatter 存元数据，正文紧跟在元数据后面。默认正文直接承载决策依据、Why、适用边界等内容。
  ```yaml
  # experience 主文件 frontmatter（L0）
  id: exp_jwt_decision_001
  type: experience
  content:
    title: "选 JWT 而非 Session"
    summary: "支持分布式部署 + 移动端友好"  # 概要（注入用）
    tags: [auth, architecture, jwt]
    appendixRefs: []  # 仅复杂详情存在时填写
  metadata:
    confidence: 0.92
    referenceCount: 3
  ```
- **召回**：必读层槽 5 只注入概要索引（title + summary + tags），约 800 字符放 3-5 条
- **完整内容**：P0 只把概要索引自动注入 prompt；主文件正文就是完整内容。只有当内容非常复杂、详细、篇幅过长时，才把细节拆成附录文件，主文件保留概述和附录索引。
- **写入规则**：每条 experience 独立存储为 .md 主文件；概要写入 _index.json。附录文件不是默认结构，只用于承载复杂长内容。

#### resource 策略详解

- **存储**：每条 resource 是一个 Markdown 主文件，frontmatter 存元数据，正文紧跟在元数据后面。默认正文直接承载资源说明、适用场景、使用边界等内容。
  ```yaml
  # resource 主文件 frontmatter（L0）
  id: res_feishu_connector
  type: resource
  subType: connector
  content:
    title: "飞书 MCP 连接器"
    summary: "只读：文档/日历；不可发消息"  # 概要（注入用）
    tags: [feishu, connector, readonly]
    relatedSkillIds: []  # tool 类型关联的 SKILL
    appendixRefs: []  # 仅复杂详情存在时填写
  metadata:
    confidence: 1.0
  ```
- **召回**：必读层槽 4 只注入概要索引（title + summary + subType），约 600 字符放 5-8 条
- **完整内容**：P0 只把概要索引自动注入 prompt；主文件正文就是完整内容。只有当资源说明非常复杂、详细、篇幅过长时，才把细节拆成附录文件，主文件保留概述和附录索引。
- **tool 类型关联 SKILL**：subType=skill 或 subType=connector 的 resource 节点通过 `relatedSkillIds` 字段关联到 SKILL 系统，agent 看到资源索引后可直接调用对应 SKILL
- **写入规则**：每条 resource 独立存储为 .md 主文件；概要写入 _index.json。附录文件不是默认结构，只用于承载复杂长内容。

#### 递进式编写的核心思想

experience 和 resource 的递进式编写不是指默认只存索引，而是指**主文件先承载概述和必要正文；只有遇到非常复杂、详细的内容时，才把详细内容拆成附录文件**。核心思想是"主文件可读，复杂细节可追"：

```
必读层注入（~1400 字符）：
  experience: 3-5 条概要（title + summary + tags）
  resource: 5-8 条概要（title + summary + subType）

后续能力需要详情时：
  先读取该记忆的 Markdown 主文件正文
  若主文件存在 appendixRefs，再读取附录文件
  主文件内容包含：概述、必要正文、适用边界、附录索引
  附录内容包含：详细决策依据 / 参数配置 / 历史版本 / 长篇说明
```

这样设计的好处：
- 必读层不会被大量 experience/resource 撑爆
- agent 有"目录感"，知道有哪些经验和资源可用
- 主文件自身就是可读记忆，不需要再通过 fileRef 找正文
- 只有复杂长内容才拆附录，不浪费 token，也避免主文件失去可读性

### 4.3 type 与 scope 的二维矩阵

每个 type 在不同 scope 下都有内容。下表说明双端如何协作填充：

| Type | personal（个人） | project（项目） | team（团队） | rules/enterprise（企业） |
|------|----------------|---------------|------------|------------------|
| **profile** | 个人偏好/习惯 | — | — | 岗位/团队归属（作为画像输入维度）|
| **task-context** | — | 项目目标/阶段/客户约束 | — | — |
| **rules** | 个人纠正约束 | 项目特定约束 | 团队规范 | 合规约束 |
| **experience** | 个人方法论种子 | 项目决策依据 | 团队最佳实践（已晋升 SKILL）| 企业级最佳实践 |
| **resource** | 个人常用文档/工具 | 项目相关资源 | 团队公共资源 | 企业知识库 |

**关键约束**：
- profile / task-context 主要在本地（个人和项目维度）
- rules / resource 双端都有，远程优先级更高
- experience 跨双端，本地原始 + 远程沉淀

### 4.4 记忆系统的边界

记忆系统负责管理**可持久复用的工作事实**，包括提取、存储、召回、注入和治理。它的输入来自各种外部路径（对话、专家会话、技能执行、连接器调用、自动化触发等），Extractor 从中提取有长期价值的信息，按 5 type 分类写入。

**记忆系统不管的事**：
- 不管对话消息本身怎么存（那是 gateway 的事）
- 不管专家/技能/连接器怎么运行（那是各自模块的事）
- 不管知识库文档怎么维护（那是知识库系统的事）
- 不管 TODO 列表、执行日志怎么管理（那是各自模块的事）

**记忆系统负责的事**：
- 从上述路径中提取出的偏好 → `profile`
- 从上述路径中提取出的任务上下文 → `task-context`
- 从上述路径中提取出的约束 → `rules`
- 从上述路径中提取出的决策/经验 → `experience`
- 从上述路径中提取出的资源指针 → `resource`

这些外部路径是记忆系统的**素材来源**，不是记忆系统要去管理或替代的对象。

### 4.5 旧记忆 type 的迁移（仅涉及记忆系统内部）

当前记忆系统内部存储使用的 4 个 type（user / feedback / project / reference）合并为新 5 type：

| 旧 type（记忆系统内部） | 新 type | 迁移规则 |
|----------------------|---------|---------|
| `user` | `profile` | 全部迁移 |
| `feedback`（偏好部分） | `profile` | 含"喜欢/偏好/习惯/希望"关键词 |
| `feedback`（约束部分） | `rules` | 含"不要/禁止/必须/约束/不能"关键词 |
| `project`（目标/阶段） | `task-context` | 含"目标/阶段/范围/客户约束/里程碑"关键词 |
| `project`（决策/方法论） | `experience` | 含"决定/选择/原因/因为/考虑到"关键词 |
| `reference` | `resource` | 全部迁移，扩展 subType 字段 |

这只是记忆系统内部存储格式的调整，不影响外部模块。

详见 [01-memory-layers-and-containers.md](./memory-system-v0.6/01-memory-layers-and-containers.md) 和 [02-work-graph-data-model.md](./memory-system-v0.6/02-work-graph-data-model.md)。

## 5. 容器分层与折叠层

记忆按容器划分作用域，按折叠层划分粒度。这是与"5 type 体系"正交的两个维度。

### 5.1 容器分层（与产品形态对应）

| 容器 | 主要承载的 type | 生命周期 | 双端归属 |
|------|---------------|---------|---------|
| **全局** | profile + 个人 rules + 个人 resource | 跨会话、跨项目持久化 | APP 本地（local/global/） |
| **项目** | task-context + 项目 rules + 项目 experience + 项目 resource | Project 生命周期 | APP 本地（local/projects/） |
| **会话** | profile/rules/experience 的候选区（待晋升） | 会话期 + 7 天归档 | APP 本地（local/_sessions/） |
| **团队** | 团队 rules + 团队 experience（已晋升 SKILL）+ 团队 resource | 团队存续期 | 后端 -> 同步到 remote/teams/ |
| **企业** | 企业 rules（合规）+ 企业 resource（知识库指针） | 企业存续期 | 后端 -> 同步到 remote/rules/enterprise/ |

自动化/定时任务不再建立独立记忆容器。它触发的内容按普通对话进入绑定的 Project 和 Session 链路；执行日志仍归自动化模块，长期可复用信息再按 5 type 写入全局或项目容器。

核心流动规则：

```
本地内向上晋升：
  会话候选 -> 项目 -> 全局（用户审核）

双端贡献（非 agent 通道）：
  本地 experience -> memory-review 识别 -> SKILL 候选 -> 用户分享 -> 团队/企业

远程下发（单向同步）：
  团队/企业 rules/resource -> 拉到 remote/ 缓存 -> memory-recall 注入
```

### 5.2 折叠层（控制召回粒度）

| 层 | 含义 | 谁生成 |
|---|---|---|
| **L0 原始** | 单条事实/偏好/决策 | 写入触发时立即生成（APP 本地） |
| **L1 摘要** | 同 type 同 scope 内的聚合摘要 | chat-end 阶段 1.5 + TTL Flush（APP 本地元数据聚合 / 后端 LLM 语义摘要） |
| **L2 综合** | 跨 type 的月度综合 | 后端定时任务（每月） |

L0 的"永不删除"指系统不会因为摘要折叠、TTL 或后台治理而主动删除原始节点；用户在 Banto 中显式删除、撤回或清空某条记忆时，属于用户主动操作，不受该约束限制。L1/L2 的 `sourceNodeIds` 反向指向仍存在的 L0。

详见 [01-memory-layers-and-containers.md](./memory-system-v0.6/01-memory-layers-and-containers.md)。

### 6.2 写入触发

四种写入时机（按优先级排序）：

1. **用户显式保存** — 用户点击"记住这个"，置信度最高
2. **AI 实时观察** — 对话中 AI 判断"这值得记住"，主动标记
3. **关键节点触发** — 项目创建、专家调用、技能执行等系统事件
4. **chat-end 整理** — 会话结束后批量分析提取

写入时 Extractor 必须为每条记忆指定 type，归属 5 type 之一，否则不入库。

### 6.3 chat-end 四阶段流程

```
阶段 1（做好当下）：会话结束 -> 10s debounce -> LLM 提取 -> 按 5 type 分类写入 L0
阶段 1.5（折叠层维护）：L0 变更 -> bucket 满/TTL 到期 -> 按 type 分组生成 L1 摘要
阶段 2（沉淀能力）：基于 experience L1 -> 模式识别 -> 生成 SKILL 候选
阶段 3（优化未来）：基于全 type L1 -> 工作模式分析 -> 生成预测/建议/缺口
```

四阶段全部异步执行，不阻塞用户。

### 6.4 必读层 5 槽位（核心设计）

agent 看到的 system prompt 不是"一堆记忆条目"，而是"5 段答案"：

```
┌────────────────────────────────────────────────────────┐
│ 必读层 4000 字符（按 5 问题固定槽位）                     │
├────────────────────────────────────────────────────────┤
│ [槽 1] 我为谁工作         ~600 字符                      │
│   来源：profile L1 摘要（合并本地软画像 + 远程硬画像）    │
│                                                        │
│ [槽 2] 我在做什么         ~1000 字符                     │
│   来源：当前 Project task-context L1（目标+阶段+客户约束）│
│                                                        │
│ [槽 3] 什么不能做         ~1000 字符                     │
│   来源：rules 全量（合规 > 团队规范 > 个人约束）       │
│                                                        │
│ [槽 4] 有什么可用资源     ~600 字符                      │
│   来源：resource L1（当前项目+全局可用资源清单）         │
│                                                        │
│ [槽 5] 高频历史经验       ~800 字符                      │
│   来源：experience top-3（按复合分排序）                │
└────────────────────────────────────────────────────────┘
```

**关键设计**：
- 槽位固定，每槽预算固定，宁可填不满，也不串槽
- 槽 3 rules 优先级最高，4000 字符紧张时优先保槽 3
- 槽 5 是必读层中唯一按"复合分"动态排序的槽位
- 槽 1-4 都是 L1 摘要（一段话），槽 5 是 L0 节点 top-K

### 6.5 任务相关层（按需召回）

预算 3000 字符，按当前任务关键词召回：

| 召回内容 | 召回方式 |
|---------|---------|
| 与任务相关的具体 rules（如某专家的使用边界） | 关键词 + tags + relatedTo 字段匹配 |
| 与任务相关的 experience 详情（决策链下钻） | 关键词 + 图谱遍历（derives_from 关系） |
| 与任务相关的 resource 具体使用方式 | 关键词 + 适用场景匹配 |

任务相关层不召回 profile / task-context（这两已在必读层全量注入）。

### 6.6 后续按需检索层（非 P0）

P0 只做系统自动注入，不开放 agent 主动调用记忆工具。后续阶段可提供 `memory_query` 工具，让 agent 按 type 和关键词主动查询。典型场景：
- "之前我们对客户 A 做过哪些类似决策？" -> query type=experience + 客户 A
- "团队对周报格式有什么具体要求？" -> query type=rules + scope=team + 周报

### 6.7 复合召回排序算法（用于槽 5 + 任务相关层）

```
score = α × hotness + β × confidence + γ × task_relevance + δ × user_signal
默认权重：α=0.35, β=0.30, γ=0.30, δ=0.05
```

| 维度 | 公式 | 数据来源 |
|------|------|---------|
| **hotness** | `recency × 0.5 + frequency × 0.5` | `updatedAt` + `referenceCount` |
| **confidence** | 直接读取 | frontmatter `confidence` 字段（0-1） |
| **task_relevance** | `max(keyword_score, tag_score) × 0.7 + container_score × 0.3` | 对话关键词 vs 记忆 title/tags/body |
| **user_signal** | `manual: true` -> 1.0, 否则 0.0 | 用户手动标记 |

### 6.8 升格机制（5 type 视角）

| 升格路径 | 触发条件 | 目标 |
|---------|---------|------|
| **会话 -> 项目** | confidence ≥ 0.9 自动；或同记忆被 ≥ 2 个 session 重复提取 | 项目 task-context / experience / rules |
| **项目 -> 全局** | 项目 rules 在 ≥ 2 个项目重复 | 全局 rules |
| **experience -> SKILL** | 同主题 experience ≥ 5 次引用 + 跨 ≥ 3 项目 | 本地 SKILL 草稿 -> 用户审核 -> 团队 SKILL |

升格不变更原始记忆，只复制晋升副本到目标容器。

详见 [03-runtime-mechanisms.md](./memory-system-v0.6/03-runtime-mechanisms.md) 和 [06-p0-implementation-plan.md](./memory-system-v0.6/06-p0-implementation-plan.md)。

## 7. 与支撑体系的关系

四大支撑体系全部归到 `resource` type 统一管理：

| 支撑体系 | 在 resource 中如何表达 | 召回时如何使用 |
|----------|---------------------|--------------|
| **专家智能体** | `resource` 节点 + subType=expert + `relatedExpertIds` 反向索引 | 专家调用时召回该专家相关 rules + experience |
| **技能** | `resource` 节点 + subType=skill；高频 experience 晋升为 SKILL | SKILL 在 SKILL 系统独立运行，记忆只存指针和使用偏好 |
| **连接器（MCP）** | `resource` 节点 + subType=connector + 使用边界存为 rules | 连接器返回的外部数据不进记忆，只提炼决策为 experience |
| **知识库** | `resource` 节点 + subType=kb + `knowledgeBaseRef` | 注入时给指针，不复制内容；权威事实归知识库 |

设计边界：
- 连接器返回的外部系统数据属于**临时上下文**，不进记忆系统
- 知识库承接"用户可编辑权威事实"职能，banto 不引入第三方文件镜像
- 技能执行结果回流到 experience，驱动置信度调整

详见 [04-coupling-with-capability-system.md](./memory-system-v0.6/04-coupling-with-capability-system.md)。

## 8. SKILL 晋升与团队贡献

experience type 是 SKILL 库的来源。本地记忆的团队贡献路径：

```
experience 记忆 -> 内置 skill「memory-review」识别可晋升模式
  -> 输出 SKILL 候选（存入本地 SKILL 草稿区）
  -> 用户在 SKILL 管理页查看/编辑/确认
  -> 用户点"分享到团队" -> HTTP 接口 -> 后端审核 -> 入库团队 SKILL
  -> 下次同步时下发到团队成员本地 SKILL 库
```

### 8.1 四档升级模式

| 档位 | 决策强度 | 触发方 | 适用场景 |
|------|---------|--------|----------|
| **L1 静默自动** | 系统判定 | 系统 | 管理员下发的全员豁免内容（v0.6 不启用） |
| **L2 系统提议** | 系统判定 + 用户授权 | 系统识别，通知中心提示 | 高置信度 + 隐私敏感度低 + 团队价值明确 |
| **L3 用户主动** | 用户判定 | 用户在 UI 点"分享到团队" | 用户认为有共享价值 |
| **L4 管理员录入** | 管理员判定 | 管理员后台 | 合规约束、企业规范 |

关键约束：所有档位最终都不由 agent 触发。P0-P3 阶段不开放任何上传通道。

### 8.2 通知中心

通知中心是记忆系统与用户之间的唯一主动触达通道，设计哲学是"宁可漏推，不可扰人"：

- P0 合规变更：实时推送
- P1 冲突裁决：实时推送，需用户操作
- P2 SKILL 候选 + L2 晋升提议：每周聚合推送一次
- P3 贡献反馈 / 远程更新：按事件聚合
- 用户对话中不弹出任何非 P0 通知

### 8.3 禁止上传清单

以下内容任何档位都不允许上传（前端 + 后端双层拦截）：profile（个人画像）、个人身份信息、临时会话上下文、敏感字段（密码/token/密钥）、AI 失败/学习记录、未脱敏人名、情绪/评价。

详见 [07-dual-end-memory-architecture.md 第三、四节](./memory-system-v0.6/07-dual-end-memory-architecture.md)。

## 9. 实施路径

分 8 个优先级阶段推进，P0-P3 完全在 APP 端独立完成，P4 之后需要后端配合：

| 阶段 | 范围 | 端 | 依赖后端 |
|------|------|----|----|
| **P0** | 5 type 体系落地 + 旧 type 兼容迁移 + Extractor Prompt 改造 + `getMemoryPrompt` 5 槽位系统自动注入 | APP | 否 |
| **P1** | `recallMemory` 评分改造 + 任务相关层自动补充 + Session 候选区/晋升 + 轻量 L1 元数据摘要 | APP | 否 |
| **P2** | JSON 双存 + edges/关系目录 + 本地决策链追溯 + 后续按需检索工具准备 | APP | 否 |
| **P3** | 本地 `remote/` 缓存目录骨架 + memory-recall 合并双端（远程缓存为空时降级为纯本地） | APP | 否 |
| **P4** | 后端能力 1（存储）+ 能力 5（同步分发：snapshot / incremental） | 后端 | 是 |
| **P5** | 用户主动贡献 UI（"分享到团队"）+ 后端审核流程 + memory-review 内置 skill | 双端 | 是 |
| **P6** | 后端能力 2（记忆树折叠）+ 能力 3（决策链）+ 能力 4（知识融合） | 后端 | 是 |
| **P7** | chat-end 阶段 2/3（模式识别 + 主动建议）+ insights/ 主动观察输出 | APP | 否 |

向后兼容：现有 user/feedback/project/reference 记忆按规则迁移到新 5 type（具体迁移规则在 P0 阶段确定）。

详见 [06-p0-implementation-plan.md](./memory-system-v0.6/06-p0-implementation-plan.md) 和 [07-dual-end-memory-architecture.md 第八节](./memory-system-v0.6/07-dual-end-memory-architecture.md)。

## 10. 关键设计决策

以下决策已在 v0.6 设计迭代中确认：

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | 记忆体系定位 | 工作进化系统（围绕 5 个本质问题） | 记忆是为了让 agent 做好工作，不是为了存得多 |
| 2 | 记忆类型 | 5 type（profile/task-context/rules/experience/resource） | 直接对应 5 个问题，每条记忆都有明确归属 |
| 3 | 会话过程数据归属 | 移出记忆系统（归会话日志 / 执行日志） | reasoning/tool-chain 不是长期记忆 |
| 4 | 必读层结构 | 固定 5 槽位 | agent 看到的是"5 段答案"而非"一堆条目" |
| 5 | Identity 处理 | 不作为产品对象或记忆 scope | 产品设计已移除 Identity，但岗位/团队/角色信息仍可作为 profile/task-context 的输入维度 |
| 6 | 自动化记忆处理 | 不建独立记忆容器 | 自动化本质是系统触发的对话，复用全局记忆 + Project 记忆 |
| 7 | 会话记忆晋升 | 选择性晋升（需用户审核） | 系统提议 + 用户确认 |
| 8 | 关系存储 | 独立 edges 目录 | 便于图谱遍历，主要承载 derives_from（决策链） |
| 9 | 置信度阈值 | 0.7 | chat-end 提取的记忆必须达标才落盘 |
| 10 | chat-end 整理 | 异步执行 | 不阻塞用户 |
| 11 | 连接器上下文 | 不直接进入记忆 | 临时数据归会话，决策提炼为 experience |
| 12 | agent 是否能调用远程 | 不能 | 隐私、质量、滥用风险 |
| 13 | 同步方向 | 远程->本地（单向） | 本地不能修改远程 |
| 14 | 本地覆盖策略 | 按 type 差异化 | rules 远程强制 / resource 远程优先 / profile 本地优先 |
| 15 | Schema 设计 | 本地远程同构 | agent 无感知，便于演进 |
| 16 | 主动建议输出时机 | 混合策略 | 低风险建议主动推送，高风险建议被动出 |
