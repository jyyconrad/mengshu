# 开源项目文档编写规范

## 核心原则

1. **面向用户，而非开发者**
   - 用户关心"如何使用"，而非"如何实现"
   - 避免暴露内部实现细节
   - 使用简洁、清晰的语言

2. **文档即产品**
   - 文档质量直接影响用户体验
   - 保持文档与代码同步
   - 定期审查和更新

3. **渐进式引导**
   - 从快速开始到高级用法
   - 从简单示例到完整参考
   - 提供足够的链接和导航

## 文档目录结构

```
docs/
├── README.md                    # 文档导航和概览
├── guides/
│   ├── getting-started.md       # 快速开始（必备）
│   ├── configuration.md         # 配置说明
│   ├── integration.md           # 集成指南
│   └── best-practices.md        # 最佳实践
├── api/
│   ├── cli-commands.md          # CLI 命令参考
│   └── memory-api.md            # API 接口文档
├── architecture/
│   ├── system-architecture.md   # 系统架构（高层次）
│   └── technology-stack.md      # 技术栈说明
└── design/
    ├── memory-system-unified-design.md  # 核心设计（单一事实来源）
    └── schema.md                # 数据模型
```

### 文档分类

| 类型 | 位置 | 用途 | 受众 |
|------|------|------|------|
| **用户指南** | `docs/guides/` | 使用教程、最佳实践 | 所有用户 |
| **API 参考** | `docs/api/` | 接口规范、命令参考 | 开发者、集成者 |
| **架构文档** | `docs/architecture/` | 系统架构、技术选型 | 架构师、贡献者 |
| **设计文档** | `docs/design/` | 核心设计、算法说明 | 深度贡献者 |
| **内部文档** | `.memory-docs/` | 过程记录、测试用例、缺陷 | 团队内部 |

## 编写规范

### 1. 快速开始（Getting Started）

**必须包含**：
- 安装步骤（清晰、可复制）
- 最简配置（5 分钟内完成）
- Hello World 示例（可运行）
- 下一步指引

**示例结构**：
```markdown
# 快速开始

## 安装
\`\`\`bash
npm install -g mengshu
\`\`\`

## 初始化
\`\`\`bash
ms init
\`\`\`

## 基本使用
\`\`\`typescript
import { MemoryService } from 'mengshu';
// ...
\`\`\`

## 下一步
- [配置详解](configuration.md)
- [集成指南](integration.md)
```

### 2. 配置说明（Configuration）

**必须包含**：
- 配置文件位置
- 所有配置项说明（表格形式）
- 默认值
- 示例配置
- 常见问题

**格式**：
```markdown
## 配置项说明

| 字段 | 说明 | 默认值 | 必填 |
|------|------|--------|------|
| `apiKey` | LLM API 密钥 | - | ✓ |
| `model` | 模型名称 | `gpt-4o-mini` | ✗ |
```

### 3. API 参考（API Reference）

**必须包含**：
- 接口签名
- 参数说明
- 返回值说明
- 示例代码
- 错误处理

**格式**：
```markdown
## memory.recall(options)

召回记忆。

### 参数

- `query` (string) - 查询文本
- `limit` (number) - 召回数量，默认 5
- `minImportance` (number) - 最低 importance 阈值

### 返回值

\`\`\`typescript
{
  memories: Memory[],
  context: string
}
\`\`\`

### 示例

\`\`\`typescript
const result = await memory.recall({
  query: '用户偏好',
  limit: 5
});
\`\`\`
```

### 4. 架构文档（Architecture）

**原则**：
- 高层次概览，避免实现细节
- 使用图表说明（Mermaid / ASCII）
- 说明"为什么"而非"怎么做"

**不要写**：
- 详细的类图、方法签名
- 内部实现细节
- 过程性讨论

**应该写**：
- 系统整体架构
- 模块划分和职责
- 技术选型理由
- 扩展性考虑

### 5. 设计文档（Design）

**适用场景**：
- 核心算法说明
- 关键设计决策
- 数据模型

**原则**：
- 精简版，聚焦"单一事实来源"
- 移除过程性内容（"下一步"、"待实施"）
- 保留已实施的决策和算法

## 禁止事项

### ❌ 不要暴露的内容

1. **内部过程**
   - 需求讨论过程
   - 设计迭代过程
   - 测试用例细节
   - 缺陷记录

2. **实现细节**
   - 内部函数名
   - 文件组织结构（除非必要）
   - 代码实现细节

3. **临时信息**
   - "下一步"、"待实施"
   - "TODO"、"FIXME"
   - 未完成的功能

### ✓ 应该暴露的内容

1. **用户价值**
   - 功能特性
   - 使用方法
   - 最佳实践

2. **API 契约**
   - 公共接口
   - 数据格式
   - 错误代码

3. **架构决策**
   - 设计理念
   - 技术选型
   - 扩展性考虑

## 文档更新流程

### 新增功能

1. **用户指南** - 添加使用说明
2. **API 参考** - 更新接口文档
3. **配置说明** - 更新配置项（如有）
4. **Changelog** - 记录变更

### 修改功能

1. **识别影响范围** - 哪些文档需要更新
2. **同步更新** - 代码和文档一起提交
3. **版本说明** - 标注破坏性变更

### 删除功能

1. **标记废弃** - 先标记 deprecated
2. **迁移指南** - 提供替代方案
3. **移除文档** - 下个大版本移除

## 工具和命令

### 文档检查

```bash
# 检查链接有效性
npx markdown-link-check docs/**/*.md

# 拼写检查
npx cspell "docs/**/*.md"
```

### 文档生成

```bash
# API 文档自动生成（如果支持）
npm run docs:api

# 预览文档
npm run docs:serve
```

## 检查清单

提交文档前检查：

- [ ] 用户视角（是否易懂）
- [ ] 代码示例可运行
- [ ] 链接有效
- [ ] 无拼写错误
- [ ] 无内部实现细节
- [ ] 无"待实施"、"下一步"
- [ ] 格式一致（Markdown 规范）

---

**记住**：文档是给用户看的，不是给自己看的。简洁、清晰、实用优先。
